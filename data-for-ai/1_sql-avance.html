L'ordre d'exécution SQL démystifié

⏱ 20 min Fondamental Indispensable

La plus grande source de confusion en SQL est l'écart entre l'ordre dans lequel on écrit une requête et l'ordre dans lequel le moteur l'exécute. On écrit SELECT en premier, mais c'est presque la dernière clause évaluée. Comprendre cet ordre est la clé pour déboguer des erreurs mystérieuses et écrire des requêtes correctes du premier coup.

L'ordre logique d'exécution

Le moteur SQL exécute les clauses dans cet ordre précis, quoi qu'il arrive :

Ordre d'exécution — du début à la fin
-- 1. FROM          → quelles tables ?
-- 2. JOIN / ON     → fusion des tables selon les conditions
-- 3. WHERE         → filtrage ligne par ligne (avant agrégation)
-- 4. GROUP BY      → regroupement en buckets
-- 5. HAVING        → filtrage sur les groupes (après agrégation)
-- 6. SELECT        → calcul des colonnes à retourner
-- 7. DISTINCT      → élimination des doublons (si présent)
-- 8. ORDER BY      → tri du résultat
-- 9. LIMIT/OFFSET  → pagination

-- La syntaxe qu'on ÉCRIT (ordre différent) :
SELECT   col1, SUM(col2) AS total   -- étape 6
FROM     ma_table                    -- étape 1
JOIN     autre_table ON ...          -- étape 2
WHERE    col3 > 100                  -- étape 3
GROUP BY col1                        -- étape 4
HAVING   SUM(col2) > 500            -- étape 5
ORDER BY total DESC                  -- étape 8
LIMIT    10;                         -- étape 9

Conséquences concrètes

1. Les alias de SELECT ne sont pas disponibles dans WHERE

Puisque WHERE est évalué avant SELECT, les alias définis dans le SELECT n'existent pas encore au moment du filtrage.

Erreur classique — alias dans WHERE
-- ✗ ERREUR : 'total' n'existe pas encore lors du WHERE
SELECT customer_id, SUM(amount) AS total
FROM orders
WHERE total > 1000;          -- ERROR: column "total" does not exist

-- ✓ CORRECT : répéter l'expression, ou utiliser une sous-requête
SELECT customer_id, SUM(amount) AS total
FROM orders
GROUP BY customer_id
HAVING SUM(amount) > 1000;   -- HAVING est évalué après SELECT

-- ✓ AUSSI CORRECT via sous-requête
SELECT * FROM (
  SELECT customer_id, SUM(amount) AS total
  FROM orders
  GROUP BY customer_id
) t
WHERE total > 1000;

2. ORDER BY peut utiliser les alias (exception)

La plupart des moteurs SQL (PostgreSQL, MySQL, SQLite, Snowflake) autorisent les alias définis dans SELECT dans le ORDER BY. C'est une exception à la règle générale, introduite pour des raisons de lisibilité. En revanche, le standard SQL strict ne le garantit pas — SQL Server l'accepte aussi mais certains outils analytiques pas.

Alias dans ORDER BY — fonctionne sur la plupart des moteurs
SELECT
  EXTRACT(YEAR FROM order_date) AS annee,
  SUM(amount)                   AS chiffre_affaires
FROM orders
GROUP BY annee           -- PostgreSQL/MySQL acceptent GROUP BY avec alias
ORDER BY chiffre_affaires DESC;  -- ORDER BY accepte l'alias partout
Piège — GROUP BY alias MySQL et PostgreSQL acceptent les alias dans GROUP BY, mais c'est une extension non standard. SQL Server et le standard SQL l'interdisent. Pour du code portable, répétez l'expression dans GROUP BY : GROUP BY EXTRACT(YEAR FROM order_date).

3. WHERE filtre AVANT l'agrégation, HAVING APRÈS

Cette distinction a un impact énorme sur les performances. Si vous pouvez exprimer votre filtre avec WHERE, faites-le — le moteur élimine les lignes avant de calculer les agrégats, ce qui réduit considérablement le travail. HAVING est réservé aux conditions portant sur des fonctions d'agrégation (SUM, COUNT, etc.).

WHERE vs HAVING — bien choisir
-- ✓ OPTIMAL : WHERE filtre 90% des lignes avant GROUP BY
SELECT category, SUM(amount) AS total
FROM orders
WHERE order_date >= '2024-01-01'   -- filtre tôt = moins de calculs
GROUP BY category
HAVING SUM(amount) > 10000;        -- filtre sur l'agrégat = HAVING obligatoire

-- ✗ SOUS-OPTIMAL (même résultat, mais tout est agrégé avant le filtre de date)
SELECT category, SUM(amount) AS total
FROM orders
GROUP BY category
HAVING order_date >= '2024-01-01'  -- ERREUR : order_date n'est plus disponible ici
  AND SUM(amount) > 10000;

4. Les window functions sont évaluées après WHERE et GROUP BY

Les fonctions de fenêtre (que nous étudierons dans le Module 6) opèrent sur le résultat déjà filtré par WHERE et agrégé par GROUP BY, mais avant ORDER BY. Cette position intermédiaire signifie que vous ne pouvez pas utiliser une window function dans un WHERE — il faut passer par une sous-requête ou une CTE.

Window function dans WHERE — solution correcte
-- ✗ ERREUR : window function interdite dans WHERE
SELECT *, ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) AS rn
FROM employees
WHERE rn = 1;   -- ERROR: window functions not allowed in WHERE

-- ✓ CORRECT : passer par un CTE ou sous-requête
WITH ranked AS (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) AS rn
  FROM employees
)
SELECT * FROM ranked WHERE rn = 1;

Mémo visuel

Ordre mental à mémoriser
-- "From Join Where Group Having Select Order Limit"
-- F → J → W → G → H → S → O → L
--
-- Mnémotechnique FR : "Fais Joindre,Wise Groupes Ont Sélectionné, Ordres Limités"
--
-- Ce qui existe à chaque étape :
-- WHERE  : colonnes des tables de base (pas les alias SELECT)
-- HAVING : colonnes de base + fonctions d'agrégation
-- ORDER  : tout ce qui précède + alias SELECT
-- LIMIT  : appliqué sur le résultat final trié
Bonne pratique Quand une requête complexe se comporte de façon inattendue, reposez-vous la question : "À quelle étape de l'exécution suis-je ?" C'est la première chose à vérifier avant de chercher une erreur de syntaxe.

NULL — le troisième état et ses pièges

⏱ 25 min Piège majeur

NULL est l'une des caractéristiques les plus mal comprises de SQL. Ce n'est pas zéro. Ce n'est pas une chaîne vide. C'est l'absence de valeur — une valeur inconnue. Cette distinction a des implications profondes sur la logique de filtrage, les jointures et les agrégations.

La logique à trois valeurs (Three-Valued Logic)

SQL ne fonctionne pas avec une logique booléenne classique (vrai/faux) mais avec une logique ternaire : TRUE, FALSE, et UNKNOWN. Toute comparaison impliquant un NULL produit UNKNOWN, pas FALSE.

NULL dans les comparaisons — tout retourne UNKNOWN
-- Toutes ces expressions retournent NULL (UNKNOWN), pas FALSE
SELECT
  NULL = NULL,      -- NULL (pas TRUE !)
  NULL != NULL,     -- NULL
  NULL > 5,         -- NULL
  NULL < 5,         -- NULL
  NULL = 0,         -- NULL
  NULL = '',        -- NULL
  NULL IS NULL,     -- TRUE  ← la bonne façon de tester
  NULL IS NOT NULL; -- FALSE
Conséquence critique — WHERE filtre les UNKNOWN La clause WHERE ne conserve que les lignes où la condition est TRUE. Les lignes où la condition est UNKNOWN (à cause d'un NULL) sont silencieusement éliminées. Ce comportement est la source de nombreux bugs difficiles à repérer.
NULL dans WHERE — lignes silencieusement perdues
-- Table employees
-- id | name    | manager_id
-- 1  | Alice   | NULL        (CEO, pas de manager)
-- 2  | Bob     | 1
-- 3  | Carol   | 1

-- ✗ Ne retourne PAS Alice (NULL != 1 → UNKNOWN, pas TRUE)
SELECT * FROM employees WHERE manager_id != 1;
-- Résultat : Bob, Carol... mais Alice est absente !

-- ✓ Pour inclure les NULL, il faut être explicite
SELECT * FROM employees
WHERE manager_id != 1 OR manager_id IS NULL;

NULL dans les agrégations

Les fonctions d'agrégation ignorent les valeurs NULL, à une exception notable près : COUNT(*) compte toutes les lignes quelle que soit leur contenu, tandis que COUNT(colonne) ne compte que les valeurs non-NULL de cette colonne.

COUNT(*) vs COUNT(col) — différence fondamentale
-- Table orders : 5 lignes, discount est NULL pour 2 d'entre elles
-- id | amount | discount
-- 1  | 100    | 10
-- 2  | 200    | NULL
-- 3  | 150    | 5
-- 4  | 300    | NULL
-- 5  | 250    | 20

SELECT
  COUNT(*)        AS total_commandes,  -- 5 (toutes les lignes)
  COUNT(discount) AS avec_remise,      -- 3 (ignore les NULL)
  SUM(discount)   AS total_remises,    -- 35 (ignore les NULL, pas 0+0=0)
  AVG(discount)   AS remise_moyenne,   -- 35/3 = 11.67 (pas 35/5 = 7 !)
  MIN(discount)   AS remise_min,       -- 5 (ignore les NULL)
  MAX(discount)   AS remise_max        -- 20 (ignore les NULL)
FROM orders;
Piège AVG et NULL AVG(discount) calcule la moyenne des valeurs NON-NULL. Si 40% de vos lignes ont discount = NULL, votre moyenne est biaisée. Pour avoir la moyenne réelle incluant 0 pour les absences : AVG(COALESCE(discount, 0)).

NULL dans les jointures

NULL ne s'égale pas à lui-même. Deux lignes ayant une colonne NULL ne vont pas se joindre via un JOIN ... ON a.id = b.id si l'une des valeurs est NULL. Ce comportement est souvent source de données manquantes dans les résultats.

NULL et JOIN — les lignes NULL ne se matchent pas
-- customers : id | name
--              1 | Alice
--           NULL | Bob (clé étrangère mal renseignée)

-- orders : customer_id | amount
--          1           | 500
--          NULL        | 300  (commande sans client connu)

-- INNER JOIN : ni Bob ni la commande orpheline n'apparaissent
SELECT c.name, o.amount
FROM customers c
JOIN orders o ON c.id = o.customer_id;
-- Résultat : Alice | 500 seulement

-- NULL = NULL → UNKNOWN → condition ON non satisfaite → ligne exclue

Les outils pour gérer NULL

COALESCE — premier non-NULL

COALESCE
-- Retourne la première valeur non-NULL
SELECT COALESCE(discount, 0)              AS remise_effective,
       COALESCE(nickname, first_name, 'N/A') AS nom_affiche
FROM users;

NULLIF — transformer une valeur en NULL

NULLIF — éviter la division par zéro
-- NULLIF(a, b) retourne NULL si a = b, sinon retourne a
-- Usage classique : éviter division par zéro
SELECT total / NULLIF(quantity, 0) AS prix_unitaire
FROM orders;
-- Si quantity = 0 → division par NULL → résultat NULL (pas une erreur)

IS DISTINCT FROM — comparaison NULL-safe

IS DISTINCT FROM (PostgreSQL, standard SQL)
-- Comparaison qui traite NULL comme une valeur normale
-- NULL IS DISTINCT FROM NULL → FALSE (ils ne sont PAS distincts)
-- NULL IS DISTINCT FROM 5   → TRUE  (ils sont distincts)

-- Utilisation : comparer deux colonnes qui peuvent être NULL
SELECT *
FROM current_data c
JOIN previous_data p ON c.id = p.id
WHERE c.value IS DISTINCT FROM p.value;
-- Trouve les lignes où la valeur a changé, même si l'une est NULL

-- MySQL/SQL Server : pas de IS DISTINCT FROM natif → utiliser :
WHERE NOT (c.value <=> p.value)   -- MySQL (spaceship operator)
WHERE COALESCE(c.value,'') != COALESCE(p.value,'')  -- workaround

NULL et NOT IN — le piège le plus dangereux

Si un sous-ensemble de NOT IN contient un NULL, la requête entière retourne zéro ligne. C'est l'un des bugs les plus difficiles à repérer en production.

NOT IN avec NULL — résultat vide garanti
-- La sous-requête retourne (1, 2, NULL)
SELECT * FROM orders
WHERE customer_id NOT IN (
  SELECT id FROM blacklisted_customers  -- contient une ligne avec id=NULL
);
-- Résultat : 0 lignes ! Même si customer_id = 99 n'est pas blacklisté.
-- Pourquoi : 99 NOT IN (1, 2, NULL) → 99!=1 AND 99!=2 AND 99!=NULL
--           → TRUE AND TRUE AND UNKNOWN → UNKNOWN → ligne exclue

-- ✓ Solution : utiliser NOT EXISTS (immune aux NULL)
SELECT * FROM orders o
WHERE NOT EXISTS (
  SELECT 1 FROM blacklisted_customers bc WHERE bc.id = o.customer_id
);
Règle d'or Préférez toujours NOT EXISTS à NOT IN quand la sous-requête porte sur des colonnes qui peuvent contenir des NULL. Nous reviendrons en détail sur ce sujet dans le Module 3.

Types de données et casting

⏱ 20 min Fondamental

Le choix du bon type de données influence la précision des calculs, la taille des tables et les performances des requêtes. Les erreurs de type — et surtout les conversions implicites non souhaitées — sont à l'origine de nombreux bugs silencieux en production.

Types numériques — précision vs performance

Panorama des types numériques
-- Entiers (stockage exact)
SMALLINT    -- 2 octets, -32 768 à 32 767
INTEGER     -- 4 octets, ~-2 milliards à 2 milliards
BIGINT      -- 8 octets, ~-9.2×10^18 à 9.2×10^18

-- Décimaux exacts (calculs financiers)
DECIMAL(10,2)   -- 10 chiffres au total, 2 après la virgule
NUMERIC(15,4)   -- synonyme de DECIMAL sur la plupart des moteurs

-- Flottants (approximatifs - jamais pour de l'argent)
REAL            -- 4 octets, ~6 chiffres significatifs
DOUBLE PRECISION -- 8 octets, ~15 chiffres significatifs
FLOAT           -- alias DOUBLE selon le moteur

-- Exemple du problème flottant :
SELECT 0.1 + 0.2;   -- 0.30000000000000004 (float)
SELECT 0.1::NUMERIC + 0.2::NUMERIC;  -- 0.3 (exact)
Ne jamais utiliser FLOAT pour l'argent Les montants financiers doivent être stockés en DECIMAL ou NUMERIC. Un FLOAT ne peut représenter exactement que les fractions dont le dénominateur est une puissance de 2. 0.1 n'en fait pas partie. Cumulées sur des millions de transactions, les erreurs d'arrondi peuvent représenter des écarts comptables significatifs.

Division entière — piège classique

Division entière vs réelle
SELECT 5 / 2;         -- 2 (division entière, résultat tronqué !)
SELECT 5.0 / 2;       -- 2.5 (un opérande décimal force la division réelle)
SELECT 5 / 2.0;       -- 2.5
SELECT CAST(5 AS DECIMAL) / 2; -- 2.5

-- Calcul de taux de conversion — erreur fréquente
SELECT COUNT(CASE WHEN paid THEN 1 END) / COUNT(*) AS taux;  -- 0 ou 1 !
-- ✓ Correct :
SELECT COUNT(CASE WHEN paid THEN 1 END)::DECIMAL / COUNT(*) AS taux;

Types texte — CHAR, VARCHAR, TEXT

Choisir le bon type texte
CHAR(n)       -- longueur fixe, complété par des espaces. Rarement utile.
VARCHAR(n)    -- longueur variable, max n caractères. Standard.
TEXT          -- longueur illimitée (PostgreSQL, MySQL). Pas en SQL Server.
NVARCHAR(n)   -- Unicode, SQL Server. Pour les textes multi-langues.

-- En pratique (PostgreSQL/MySQL) : TEXT ou VARCHAR(n)
-- La limite n dans VARCHAR est utile comme contrainte métier,
-- pas pour les performances (PostgreSQL stocke TEXT et VARCHAR pareil)

-- Comparaisons texte : attention à la casse selon la collation
SELECT * FROM users WHERE email = 'Alice@Example.com';
-- Selon la collation : peut trouver 'alice@example.com' ou pas

-- Solution portable : normaliser au moment de la saisie ou indexer lower()
SELECT * FROM users WHERE lower(email) = lower('Alice@Example.com');

Types date et heure

Date, Timestamp et Timezone
DATE          -- '2024-03-15' (pas d'heure)
TIME          -- '14:30:00'
TIMESTAMP     -- '2024-03-15 14:30:00' (sans timezone — stocke tel quel)
TIMESTAMPTZ   -- '2024-03-15 14:30:00+02' (avec timezone — converti en UTC)

-- TIMESTAMP vs TIMESTAMPTZ : une différence critique
-- TIMESTAMP stocke ce qu'on lui donne, sans conversion
-- TIMESTAMPTZ stocke en UTC et affiche dans le TZ de la session

-- Arithmétique sur les dates
SELECT
  NOW() - INTERVAL '30 days' AS il_y_a_30_jours,
  DATE_TRUNC('month', NOW())  AS debut_du_mois,
  EXTRACT(DOW FROM NOW())      AS jour_semaine, -- 0=dimanche, 6=samedi
  NOW()::DATE                  AS aujourd_hui;

-- Différence entre deux dates
SELECT
  AGE(end_date, start_date)   AS duree_exacte,    -- '2 years 3 months 5 days'
  end_date - start_date        AS jours_entiers;   -- integer (nb de jours)

CAST et conversions

Syntaxes de conversion
-- Syntaxes équivalentes (standard SQL vs PostgreSQL shorthand)
CAST(col AS INTEGER)
col::INTEGER        -- PostgreSQL uniquement, plus concis

CAST('2024-01-15' AS DATE)
'2024-01-15'::DATE  -- PostgreSQL

-- Conversions utiles
CAST(amount AS TEXT)           -- nombre vers texte
CAST('123.45' AS DECIMAL(8,2)) -- texte vers nombre
CAST(ts AS DATE)               -- timestamp vers date (tronque l'heure)

-- TRY_CAST (SQL Server) / safe_cast (BigQuery) : ne plante pas sur erreur
SELECT TRY_CAST(user_input AS INTEGER)  -- NULL si conversion impossible

Conversions implicites — sources de bugs

Conversions implicites à surveiller
-- Comparaison texte vs entier : MySQL convertit silencieusement
SELECT * FROM users WHERE id = '5abc';
-- MySQL : id = 5 (convertit '5abc' en 5, prend les chiffres du début)
-- PostgreSQL : erreur de type (plus sûr)

-- Concaténation avec NULL
SELECT 'Bonjour ' || NULL;   -- NULL (pas 'Bonjour ') en SQL standard
SELECT CONCAT('Bonjour ', NULL);  -- 'Bonjour ' (MySQL/PostgreSQL ignorent NULL)

-- Type date depuis une chaîne
SELECT * FROM orders WHERE order_date = '2024-01-15';
-- Fonctionne si order_date est DATE et que le format est reconnu
-- Risque : formats ambigus '01/02/2024' (1er février ou 2 janvier ?)
Bonne pratique — être explicite Préférez les conversions explicites aux conversions implicites. Elles rendent l'intention claire et évitent les surprises lors d'un changement de moteur ou de version.

Sous-requêtes dans le FROM — les derived tables

⏱ 20 min Fondamental

Une derived table est une sous-requête placée dans la clause FROM. Le moteur exécute d'abord cette sous-requête et traite son résultat comme une table temporaire. C'est le mécanisme de base pour enchaîner des transformations SQL et le précurseur direct des CTE (que nous verrons en Module 4).

Syntaxe et règles fondamentales

Structure d'une derived table
-- La sous-requête DOIT avoir un alias
SELECT dt.category, dt.total
FROM (
  SELECT category, SUM(amount) AS total
  FROM orders
  WHERE status = 'completed'
  GROUP BY category
) AS dt                       -- alias obligatoire
WHERE dt.total > 10000
ORDER BY dt.total DESC;
Alias obligatoire Toute sous-requête dans le FROM doit être aliasée. Oublier l'alias est une erreur de syntaxe sur tous les moteurs SQL. En PostgreSQL, le mot-clé AS est optionnel mais fortement recommandé pour la lisibilité.

Cas d'usage principaux

1. Pré-agréger avant de joindre

C'est l'un des patterns les plus performants : au lieu de joindre toutes les lignes détail puis d'agréger, on agrège d'abord (réduction drastique des lignes) puis on joint.

Agrégation avant jointure — gain de performance
-- ✗ MOINS EFFICACE : joint 1M lignes d'orders puis agrège
SELECT c.name, SUM(o.amount) AS total
FROM customers c
JOIN orders o ON c.id = o.customer_id
GROUP BY c.id, c.name;

-- ✓ PLUS EFFICACE : agrège d'abord (1M → N clients), puis joint N lignes
SELECT c.name, agg.total
FROM customers c
JOIN (
  SELECT customer_id, SUM(amount) AS total
  FROM orders
  GROUP BY customer_id
) AS agg ON c.id = agg.customer_id;

2. Filtrer sur un résultat agrégé (sans HAVING)

Filtrer sur un alias d'agrégat via derived table
-- Trouver les catégories où le produit le moins cher > 100€
SELECT category, min_price
FROM (
  SELECT category, MIN(price) AS min_price
  FROM products
  GROUP BY category
) AS cat_stats
WHERE min_price > 100;
-- Note : équivalent avec HAVING, mais la derived table permet
-- de réutiliser l'alias dans d'autres clauses

3. Appliquer une transformation puis filtrer

Calcul intermédiaire dans une derived table
-- Trouver les commandes dont le montant est dans le top 10%
SELECT *
FROM (
  SELECT *,
    NTILE(10) OVER (ORDER BY amount DESC) AS decile
  FROM orders
) AS ranked
WHERE decile = 1;   -- top 10%
-- Impossible sans la derived table car WHERE s'exécute avant OVER()

4. Derived tables multiples

Joindre plusieurs derived tables
-- Comparer les ventes de deux années
SELECT
  curr.category,
  curr.total AS ventes_2024,
  prev.total AS ventes_2023,
  ROUND((curr.total - prev.total) / prev.total * 100, 1) AS evolution_pct
FROM
  (SELECT category, SUM(amount) AS total
   FROM orders WHERE EXTRACT(YEAR FROM order_date) = 2024
   GROUP BY category) AS curr
FULL OUTER JOIN
  (SELECT category, SUM(amount) AS total
   FROM orders WHERE EXTRACT(YEAR FROM order_date) = 2023
   GROUP BY category) AS prev
  ON curr.category = prev.category;

Derived table vs CTE — quand choisir ?

Derived table vs CTE — comparaison
-- Derived table : inline, un seul usage, requêtes courtes
SELECT * FROM (SELECT ...) AS dt WHERE ...;

-- CTE : nommée, réutilisable, plus lisible pour les transformations longues
WITH monthly_sales AS (
  SELECT DATE_TRUNC('month', order_date) AS month,
         SUM(amount) AS total
  FROM orders GROUP BY 1
)
SELECT month, total,
       LAG(total) OVER (ORDER BY month) AS prev_month
FROM monthly_sales;

-- Règle pratique :
-- ≤ 1 usage et ≤ 5 lignes → derived table
-- > 1 usage ou logique complexe → CTE (Module 4)
Note de performance Sur la plupart des moteurs modernes (PostgreSQL 12+, MySQL 8+, SQL Server), les derived tables simples sont "inlinées" par l'optimiseur — il n'y a pas de matérialisation intermédiaire. L'optimiseur les fusionne avec la requête externe. Les CTE ont un comportement différent selon le moteur (Module 4).

⏱ 30 min Indispensable

INNER JOIN orders o ON c.id = o.customer_id;
-- Résultat :
-- Alice | 500
-- Alice | 300
-- Bob   | 750
-- Carol est absente (aucune commande)

LEFT JOIN — toutes les lignes de gauche, NULLs pour les non-matchs

LEFT JOIN
SELECT c.name, o.amount
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id;
-- Résultat :
-- Alice | 500
-- Alice | 300
-- Bob   | 750
-- Carol | NULL  ← Carol est présente avec NULL pour amount

-- Cas d'usage : tous les clients, avec ou sans commande
-- Carol apparaît car elle est dans la table de GAUCHE (customers)

FULL OUTER JOIN — toutes les lignes des deux côtés

FULL OUTER JOIN
SELECT c.name, o.amount
FROM customers c
FULL OUTER JOIN orders o ON c.id = o.customer_id;
-- Résultat :
-- Alice | 500
-- Alice | 300
-- Bob   | 750
-- Carol | NULL  (client sans commande)
-- NULL  | 999   (si une commande orpheline existait sans customer)

-- Cas d'usage : réconciliation de deux datasets, détection des orphelins
-- Note : MySQL ne supporte pas FULL OUTER JOIN nativement
-- Workaround MySQL : LEFT JOIN UNION ALL RIGHT JOIN WHERE l.id IS NULL

Le piège du filtre sur un LEFT JOIN

C'est l'erreur la plus fréquente avec les jointures : appliquer un filtre WHERE sur une colonne de la table droite d'un LEFT JOIN annule silencieusement l'effet du LEFT JOIN et le transforme en INNER JOIN.

LEFT JOIN → INNER JOIN accidentel
-- ✗ Le WHERE filtre les NULLs, Carol disparaît → comportement INNER JOIN
SELECT c.name, o.amount
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id
WHERE o.amount > 100;   -- o.amount IS NULL pour Carol → UNKNOWN → exclue

-- ✓ Déplacer le filtre dans le ON pour conserver tous les clients
SELECT c.name, o.amount
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id AND o.amount > 100;
-- Carol apparaît avec NULL (aucune commande > 100 trouvée)
-- Alice et Bob n'apparaissent que pour les commandes > 100

-- Différence sémantique :
-- ON  : filtre les lignes candidates AU JOIN
-- WHERE : filtre le résultat APRÈS le join (détruit l'effet LEFT)
Règle d'or LEFT JOIN Si vous filtrez sur une colonne de la table droite dans le WHERE après un LEFT JOIN, vous obtenez de facto un INNER JOIN. Le filtre doit aller dans le ON si vous souhaitez conserver les lignes sans correspondance.

Multiplication des lignes — quand le join explose

Un point souvent oublié : si la table droite contient plusieurs lignes pour une même valeur de clé, chaque ligne de gauche sera dupliquée autant de fois. Un client avec 3 commandes apparaîtra 3 fois. Ce n'est pas un bug SQL — c'est le comportement attendu — mais cela peut poser problème si on s'attend à obtenir une ligne par client.

Multiplication des lignes — à anticiper
-- customers : 1000 lignes
-- orders : 50 000 lignes (50 par client en moyenne)
SELECT c.*, o.*
FROM customers c
JOIN orders o ON c.id = o.customer_id;
-- Résultat : 50 000 lignes (pas 1000 !)

-- Si on veut juste vérifier SI un client a des commandes → SEMI JOIN
SELECT DISTINCT c.*
FROM customers c
JOIN orders o ON c.id = o.customer_id;
-- Ou mieux (plus performant) :
SELECT * FROM customers c WHERE EXISTS (
  SELECT 1 FROM orders WHERE customer_id = c.id
);

Chaîner plusieurs jointures

Multi-table JOIN
-- Les JOIN sont évalués de gauche à droite (associativité gauche)
SELECT
  c.name        AS client,
  p.name        AS produit,
  oi.quantity,
  o.order_date
FROM customers c
JOIN orders     o  ON c.id      = o.customer_id
JOIN order_items oi ON o.id    = oi.order_id
JOIN products   p  ON oi.product_id = p.id
WHERE o.order_date >= '2024-01-01';

-- Ordre recommandé : table principale → dépendances → enrichissements
-- Chaque JOIN filtre potentiellement le résultat si INNER
-- Un LEFT JOIN intermédiaire peut propager des NULLs vers les JOIN suivants

CROSS JOIN et SELF JOIN

⏱ 20 min Technique

Deux types de jointures moins fréquents mais très puissants dans des contextes analytiques : le CROSS JOIN qui génère toutes les combinaisons possibles, et le SELF JOIN qui permet à une table de se joindre à elle-même.

CROSS JOIN — le produit cartésien explicite

Un CROSS JOIN génère toutes les paires possibles entre les lignes de deux tables. Si la table A a M lignes et B a N lignes, le résultat a M × N lignes. Il n'y a pas de condition ON.

CROSS JOIN — syntaxes équivalentes
-- Syntaxe explicite (recommandée pour la lisibilité)
SELECT a.color, b.size
FROM colors a
CROSS JOIN sizes b;

-- Syntaxe implicite (ANSI old-style — éviter en production)
SELECT a.color, b.size
FROM colors a, sizes b;

-- Avec 3 couleurs × 4 tailles = 12 combinaisons de produits

Cas d'usage 1 — Générer un calendrier

Générer toutes les dates d'un mois
-- Générer toutes les dates de janvier 2024
WITH digits AS (
  SELECT generate_series(0, 30) AS n  -- PostgreSQL
  -- MySQL : remplacer par une table de nombres ou une CTE récursive
)
SELECT ('2024-01-01'::DATE + n) AS calendar_date
FROM digits
WHERE ('2024-01-01'::DATE + n) < '2024-02-01';

-- Cas d'usage analytics : s'assurer qu'il y a une ligne par jour
-- même pour les jours sans données (pour les graphes sans trous)
SELECT
  cal.calendar_date,
  COALESCE(daily.revenue, 0) AS revenue
FROM (
  SELECT generate_series('2024-01-01'::date, '2024-12-31'::date, '1 day') AS calendar_date
) cal
LEFT JOIN (
  SELECT order_date::date, SUM(amount) AS revenue
  FROM orders GROUP BY 1
) daily ON cal.calendar_date = daily.order_date
ORDER BY cal.calendar_date;

Cas d'usage 2 — Générer toutes les combinaisons pour un rapport

Skeleton complet région × catégorie × mois
-- Créer un squelette complet région × catégorie pour éviter les trous
WITH regions AS (SELECT DISTINCT region FROM stores),
     categories AS (SELECT DISTINCT category FROM products),
     months AS (
       SELECT DATE_TRUNC('month', d) AS month
       FROM generate_series('2024-01-01'::date, '2024-12-01'::date, '1 month') d
     )
SELECT r.region, c.category, m.month,
       COALESCE(s.total, 0) AS total_sales
FROM regions r
CROSS JOIN categories c
CROSS JOIN months m
LEFT JOIN (
  SELECT store_region, product_category,
         DATE_TRUNC('month', sale_date) AS month,
         SUM(amount) AS total
  FROM sales
  GROUP BY 1,2,3
) s ON r.region = s.store_region
   AND c.category = s.product_category
   AND m.month = s.month;
CROSS JOIN et la taille des données Un CROSS JOIN sur deux tables de 10 000 lignes chacune produit 100 millions de lignes. Toujours vérifier les volumes avant d'utiliser un CROSS JOIN en production. Si vous avez un CROSS JOIN accidentel (oubli de la condition ON dans un vieux style de jointure), le résultat peut saturer la mémoire.

SELF JOIN — une table jointe à elle-même

Un SELF JOIN est une jointure d'une table avec elle-même, en utilisant deux alias différents. C'est le pattern naturel pour les structures hiérarchiques (manager/employé, catégorie/sous-catégorie) et les comparaisons intra-table.

Cas d'usage 1 — Hiérarchie manager/employé

SELF JOIN — organigramme
-- Table employees(id, name, manager_id)
-- manager_id fait référence à employees.id

SELECT
  emp.name   AS employe,
  mgr.name   AS manager
FROM employees emp
LEFT JOIN employees mgr ON emp.manager_id = mgr.id;
-- LEFT JOIN pour inclure le CEO (manager_id = NULL)
-- Résultat :
-- Alice   | NULL     (CEO)
-- Bob     | Alice
-- Carol   | Alice
-- David   | Bob

Cas d'usage 2 — Comparer des lignes consécutives

SELF JOIN — détecter les doublons et variations
-- Trouver les produits dont le prix a augmenté entre deux enregistrements
-- price_history(product_id, price, recorded_at)
SELECT
  a.product_id,
  a.price   AS ancien_prix,
  b.price   AS nouveau_prix,
  b.price - a.price AS hausse
FROM price_history a
JOIN price_history b
  ON a.product_id = b.product_id
  AND b.recorded_at > a.recorded_at
WHERE b.price > a.price
  -- S'assurer qu'il n'y a pas d'enregistrement intermédiaire
  AND NOT EXISTS (
    SELECT 1 FROM price_history c
    WHERE c.product_id = a.product_id
      AND c.recorded_at > a.recorded_at
      AND c.recorded_at < b.recorded_at
  );

Cas d'usage 3 — Trouver toutes les paires

SELF JOIN — toutes les paires uniques d'utilisateurs
-- Générer toutes les paires uniques (sans doublons ni auto-pairing)
SELECT a.id AS user1_id, b.id AS user2_id
FROM users a
JOIN users b ON a.id < b.id;   -- a.id < b.id évite (1,2) ET (2,1) et (1,1)

-- Avec 100 utilisateurs → 100*99/2 = 4950 paires uniques
-- Cas d'usage : matrice de similarité, graphes de réseau social
Alternative moderne aux SELF JOINs Beaucoup de SELF JOINs pour comparer des lignes consécutives peuvent être remplacés par les fonctions LAG/LEAD (Module 6), qui sont plus lisibles et souvent plus performantes. Gardez le SELF JOIN pour les hiérarchies et les cas où LAG/LEAD n'offre pas assez de flexibilité.

ANTI JOIN et SEMI JOIN

⏱ 25 min Piège NULL

Le SEMI JOIN et l'ANTI JOIN ne sont pas des mots-clés SQL — ce sont des patterns qui répondent à deux questions fondamentales de l'analyse de données : "quelles lignes A ont un correspondant dans B ?" et "quelles lignes A n'en ont PAS ?". Plusieurs implémentations existent, avec des comportements différents face aux NULLs.

SEMI JOIN — "existe dans l'autre table"

Un semi-join retourne des lignes de la table A seulement si un match existe dans B, sans inclure les colonnes de B et sans dupliquer les lignes de A en cas de multiples matchs.

SEMI JOIN — trois façons équivalentes
-- Trouver les clients qui ont passé au moins une commande

-- ① EXISTS (recommandé) : court-circuite au premier match, pas de duplicats
SELECT * FROM customers c
WHERE EXISTS (
  SELECT 1 FROM orders WHERE customer_id = c.id
);

-- ② IN : correct si customer_id n'est jamais NULL dans orders
SELECT * FROM customers
WHERE id IN (SELECT customer_id FROM orders);

-- ③ INNER JOIN + DISTINCT : fonctionne mais peut être lent sur gros volumes
SELECT DISTINCT c.*
FROM customers c
JOIN orders o ON c.id = o.customer_id;

-- Performances : ① ≥ ② > ③ (règle générale, vérifiez avec EXPLAIN)

ANTI JOIN — "n'existe PAS dans l'autre table"

L'anti-join retourne les lignes de A pour lesquelles aucun match n'existe dans B. C'est ici que les pièges avec NULL sont les plus dangereux.

ANTI JOIN — trois approches
-- Trouver les clients qui n'ont JAMAIS commandé

-- ① NOT EXISTS (RECOMMANDÉ) : sûr face aux NULLs, court-circuite
SELECT * FROM customers c
WHERE NOT EXISTS (
  SELECT 1 FROM orders WHERE customer_id = c.id
);

-- ② LEFT JOIN + IS NULL : très courant, lisible, aussi sûr
SELECT c.*
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id
WHERE o.customer_id IS NULL;
-- Note : tester IS NULL sur une colonne NOT NULL de la table droite
-- garantit qu'on ne capture que les lignes sans match

-- ③ NOT IN : DANGEREUX si la sous-requête peut contenir des NULLs
SELECT * FROM customers
WHERE id NOT IN (SELECT customer_id FROM orders);
-- ⚠ Si orders contient une ligne avec customer_id = NULL
--   → résultat = 0 ligne (piège de la logique à trois valeurs)

Le piège NOT IN / NULL en détail

Démonstration du piège NOT IN
-- Supposons : orders.customer_id peut être NULL (commande en cours sans client)
-- Sous-requête retourne : (1, 3, 5, NULL)

-- Pour le client avec id=2 :
-- 2 NOT IN (1, 3, 5, NULL)
-- = 2 != 1 AND 2 != 3 AND 2 != 5 AND 2 != NULL
-- = TRUE  AND TRUE  AND TRUE  AND UNKNOWN
-- = UNKNOWN → ligne EXCLUE (alors qu'on attendrait TRUE !)

-- Résultat : AUCUN client retourné, même ceux qui n'ont vraiment pas commandé
-- Bug très difficile à diagnostiquer en production

-- ✓ Solution systématique : NOT EXISTS ou filtrer les NULLs dans IN
SELECT * FROM customers
WHERE id NOT IN (
  SELECT customer_id FROM orders WHERE customer_id IS NOT NULL
);
Règle absolue — NOT IN avec sous-requête N'utilisez jamais NOT IN (SELECT ...) sans avoir vérifié ou forcé que la sous-requête ne peut pas contenir de NULL. Préférez systématiquement NOT EXISTS qui est immunisé contre ce problème et souvent plus performant.

EXCEPT — l'anti-join ensembliste

EXCEPT comme anti-join
-- EXCEPT retourne les lignes du premier SELECT absentes du second
-- Traite NULL = NULL (contrairement aux JOIN)
SELECT id FROM all_users
EXCEPT
SELECT user_id FROM active_users;
-- Retourne les IDs présents dans all_users mais pas dans active_users

-- Différences vs NOT EXISTS :
-- ① EXCEPT déduplique (comme UNION/INTERSECT)
-- ② EXCEPT compare toutes les colonnes SELECT
-- ③ EXCEPT est plus lent sur gros volumes (tri/hash complet)
-- ④ EXCEPT traite NULL = NULL (comportement différent des JOIN)

-- EXCEPT ALL (garde les doublons) :
SELECT id FROM table_a EXCEPT ALL SELECT id FROM table_b;

Comparaison de performance

Choisir la bonne implémentation
-- Petit ensemble B (quelques centaines de valeurs connues et sans NULL)
WHERE id NOT IN (1, 2, 3, 5, 8)  -- OK, direct et lisible

-- Sous-requête de taille inconnue ou potentiellement avec NULLs
WHERE NOT EXISTS (SELECT 1 FROM B WHERE B.key = A.key)  -- toujours sûr

-- Déduplication nécessaire + comparaison ensembliste
SELECT ... EXCEPT SELECT ...  -- lisible mais moins performant

-- Vérification de performance : utiliser EXPLAIN ANALYZE
-- PostgreSQL utilise Hash Anti Join pour NOT EXISTS (très efficace)
-- Nested Loop Anti Join pour les petites tables

Jointures complexes et inéquijoins

⏱ 25 min Avancé

La plupart des jointures en production utilisent l'égalité (a.id = b.id), mais SQL permet des conditions ON bien plus riches : inégalités, intervalles, expressions. Ces jointures "non-équi" ouvrent des possibilités analytiques puissantes, notamment pour les dimensions à variation lente et les analyses temporelles.

USING — raccourci pour les égalités sur colonnes identiques

USING vs ON
-- ON classique (verbose)
SELECT c.name, o.amount
FROM customers c
JOIN orders o ON c.id = o.id;   -- si les colonnes ont le même nom "id"

-- USING (plus concis quand les noms de colonnes sont identiques)
SELECT name, amount
FROM customers
JOIN orders USING (id);
-- USING élimine la colonne dupliquée du résultat
-- ON garde les deux colonnes (c.id et o.id)

-- Plusieurs colonnes avec USING
SELECT * FROM fact_sales
JOIN dim_date USING (date_key, region_key);

-- Limitation : USING ne fonctionne que pour l'égalité
-- Pour tout autre type de condition → ON obligatoire

Inéquijoins — jointures sur des inégalités

Un inéquijoin utilise des opérateurs autres que l'égalité dans la condition ON. Ils permettent de joindre des lignes basées sur des intervalles ou des relations d'ordre, ce qu'on ne peut pas faire avec une simple clé étrangère.

Cas d'usage 1 — Dimension à variation lente (SCD Type 2)

Inéquijoin — retrouver le tarif en vigueur à la date d'achat
-- prices(product_id, price, valid_from, valid_to)
-- Chaque changement de prix crée une nouvelle ligne
-- valid_to = NULL pour le tarif actuel

SELECT
  o.id,
  o.product_id,
  o.quantity,
  p.price,
  o.quantity * p.price AS total
FROM orders o
JOIN prices p
  ON o.product_id = p.product_id
  AND o.order_date >= p.valid_from
  AND (o.order_date < p.valid_to OR p.valid_to IS NULL);
-- Pour chaque commande, trouve le prix qui était valide à la date d'achat

Cas d'usage 2 — Joindre sur des tranches (brackets)

Inéquijoin — tranches de commission
-- commission_brackets(min_sales, max_sales, rate)
-- 0–1000    → 5%
-- 1000–5000 → 8%
-- 5000+     → 12%

SELECT
  s.salesperson,
  s.total_sales,
  cb.rate,
  s.total_sales * cb.rate / 100 AS commission
FROM monthly_sales s
JOIN commission_brackets cb
  ON s.total_sales >= cb.min_sales
  AND (s.total_sales < cb.max_sales OR cb.max_sales IS NULL);

-- Alternative sans jointure : CASE WHEN (plus simple pour peu de tranches)
SELECT salesperson, total_sales,
  CASE
    WHEN total_sales < 1000 THEN total_sales * 0.05
    WHEN total_sales < 5000 THEN total_sales * 0.08
    ELSE total_sales * 0.12
  END AS commission
FROM monthly_sales;

Cas d'usage 3 — Fenêtre temporelle (event matching)

Inéquijoin — événements dans une fenêtre de temps
-- Trouver les clics qui surviennent dans les 30 minutes après une impression
SELECT
  i.user_id,
  i.ad_id,
  i.impression_time,
  c.click_time,
  c.click_time - i.impression_time AS time_to_click
FROM impressions i
JOIN clicks c
  ON i.user_id = c.user_id
  AND i.ad_id = c.ad_id
  AND c.click_time BETWEEN i.impression_time
                       AND i.impression_time + INTERVAL '30 minutes'
  AND c.click_time > i.impression_time;  -- le clic APRÈS l'impression

Jointures sur plusieurs colonnes

Multi-column JOIN — clé composite
-- Clé composite : (product_id, warehouse_id) ensemble
SELECT
  s.product_id,
  s.warehouse_id,
  s.quantity_sold,
  i.quantity_in_stock,
  i.quantity_in_stock - s.quantity_sold AS remaining
FROM sales_summary s
JOIN inventory i
  ON s.product_id   = i.product_id
  AND s.warehouse_id = i.warehouse_id
  AND s.sale_month  = i.inventory_month;

-- Piège : si une colonne de la clé est NULL → le match échoue
-- Utiliser IS NOT DISTINCT FROM si les NULLs doivent matcher
ON (s.product_id = i.product_id OR (s.product_id IS NULL AND i.product_id IS NULL))
Performance des inéquijoins Les inéquijoins ne peuvent pas utiliser les index de type Hash Join (qui nécessitent l'égalité). L'optimiseur utilisera en général un Nested Loop (O(N×M)) ou un Merge Join si les données sont triées. Sur de gros volumes, envisagez une pré-filtration dans une CTE avant l'inéquijoin pour réduire le nombre de lignes.

Jointures avec agrégats

⏱ 20 min Performance

Combiner jointures et agrégations est l'une des opérations les plus courantes en analytics. L'ordre dans lequel vous choisissez d'agréger — avant ou après le join — a un impact majeur sur les performances et parfois sur la correction des résultats.

Agréger avant vs après le JOIN

Deux approches — résultat identique, coût différent
-- Contexte : orders (10M lignes), customers (100K lignes)

-- ① APRÈS le JOIN (classique mais coûteux)
SELECT c.country, SUM(o.amount) AS total
FROM customers c
JOIN orders o ON c.id = o.customer_id
GROUP BY c.country;
-- L'optimiseur doit joindre 10M lignes PUIS agréger

-- ② AVANT le JOIN (meilleur dans certains cas)
SELECT c.country, agg.total
FROM customers c
JOIN (
  SELECT customer_id, SUM(amount) AS total
  FROM orders
  GROUP BY customer_id
) agg ON c.id = agg.customer_id
GROUP BY c.country;
-- Réduit d'abord 10M → 100K lignes, puis joint 100K × 100K

-- En pratique : les optimiseurs modernes font souvent cette transformation
-- automatiquement. Vérifiez avec EXPLAIN avant d'optimiser manuellement.

Fan-out — doublement accidentel des agrégats

Le "fan-out" est un bug classique : quand une table est jointe à une autre qui a plusieurs lignes par clé, les agrégats sont artificiellement multipliés.

Fan-out — double comptage silencieux
-- orders(id, customer_id, amount)
-- shipments(order_id, tracking_number) -- un order peut avoir plusieurs shipments

-- ✗ BUG : le amount est compté autant de fois qu'il y a d'expéditions
SELECT o.customer_id, SUM(o.amount) AS total_spend
FROM orders o
JOIN shipments s ON o.id = s.order_id
GROUP BY o.customer_id;
-- Un order avec 3 expéditions compte son amount 3 fois !

-- ✓ SOLUTION 1 : JOIN sur les shipments après agrégation
SELECT o.customer_id, SUM(o.amount) AS total_spend
FROM orders o
WHERE EXISTS (SELECT 1 FROM shipments WHERE order_id = o.id)
GROUP BY o.customer_id;

-- ✓ SOLUTION 2 : déduplication avec SUM DISTINCT (non standard) ou pré-agréger
SELECT customer_id, SUM(amount) AS total_spend
FROM (
  SELECT DISTINCT ON (o.id) o.customer_id, o.amount
  FROM orders o
  JOIN shipments s ON o.id = s.order_id
) deduped
GROUP BY customer_id;
Fan-out — diagnostic Si vos totaux semblent trop élevés après une jointure, vérifiez si la table jointe peut avoir plusieurs lignes par clé. Un COUNT(*) avant et après le join vous dira si des lignes sont dupliquées.

Jointures sur des résultats de window functions

Joindre sur le résultat d'une window function
-- Enrichir chaque commande avec le rang du client (par total de dépenses)
WITH customer_ranks AS (
  SELECT
    customer_id,
    SUM(amount)    AS total_spend,
    DENSE_RANK() OVER (ORDER BY SUM(amount) DESC) AS spend_rank
  FROM orders
  GROUP BY customer_id
)
SELECT
  o.id,
  o.amount,
  cr.total_spend,
  cr.spend_rank
FROM orders o
JOIN customer_ranks cr ON o.customer_id = cr.customer_id;

Agréger différentes granularités avec des ratios

Ratio ligne / total — pattern classique
-- Part de chaque catégorie dans les ventes totales
SELECT
  p.category,
  SUM(oi.amount) AS ventes_categorie,
  totaux.grand_total,
  ROUND(SUM(oi.amount) * 100.0 / totaux.grand_total, 1) AS part_pct
FROM order_items oi
JOIN products p ON oi.product_id = p.id
CROSS JOIN (
  SELECT SUM(amount) AS grand_total FROM order_items
) totaux
GROUP BY p.category, totaux.grand_total
ORDER BY ventes_categorie DESC;
-- Le CROSS JOIN avec un scalaire évite une sous-requête corrélée

Sous-requêtes corrélées

⏱ 25 min Performance critique

Une sous-requête corrélée est une sous-requête qui référence une colonne de la requête externe. Contrairement à une sous-requête classique (évaluée une seule fois), une sous-requête corrélée est réévaluée pour chaque ligne de la requête externe. C'est puissant, mais potentiellement très lent.

Anatomie d'une sous-requête corrélée

Sous-requête corrélée vs sous-requête indépendante
-- ① Sous-requête INDÉPENDANTE : exécutée UNE seule fois
SELECT * FROM products
WHERE price > (SELECT AVG(price) FROM products);
--            └── pas de référence à la table externe

-- ② Sous-requête CORRÉLÉE : exécutée pour CHAQUE ligne de products
SELECT * FROM products p
WHERE price > (
  SELECT AVG(price) FROM products
  WHERE category = p.category  -- ← référence p.category de la requête externe
);
-- Pour chaque produit, calcule la moyenne de SA catégorie
-- Si 10 000 produits dans 50 catégories → sous-requête exécutée 10 000 fois !

Cas d'usage classique — dernière valeur par groupe

Le pattern "dernière ligne par groupe" est l'un des plus demandés en analytics. La sous-requête corrélée en est la formulation la plus directe, avant l'avènement des window functions.

Dernière commande par client
-- ① Via sous-requête corrélée (lisible mais lent sur gros volumes)
SELECT * FROM orders o
WHERE order_date = (
  SELECT MAX(order_date)
  FROM orders
  WHERE customer_id = o.customer_id  -- corrélation
);

-- ② Via window function (recommandé, une seule passe)
SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date DESC) AS rn
  FROM orders
) t WHERE rn = 1;

-- ③ Via DISTINCT ON (PostgreSQL uniquement, très concis)
SELECT DISTINCT ON (customer_id) *
FROM orders
ORDER BY customer_id, order_date DESC;

Sous-requête corrélée dans SELECT

Une sous-requête corrélée peut apparaître dans le SELECT pour calculer une valeur dérivée pour chaque ligne, sans avoir à écrire une jointure explicite.

Sous-requête scalaire corrélée dans SELECT
-- Pour chaque produit, afficher la moyenne de sa catégorie
SELECT
  p.name,
  p.price,
  p.category,
  (SELECT AVG(price) FROM products WHERE category = p.category) AS cat_avg,
  p.price - (SELECT AVG(price) FROM products WHERE category = p.category) AS vs_avg
FROM products p;

-- ✗ Problème : la sous-requête est exécutée DEUX fois par ligne
-- ✓ Solution : utiliser une CTE ou window function
SELECT
  p.name, p.price, p.category,
  AVG(p.price) OVER (PARTITION BY p.category) AS cat_avg,
  p.price - AVG(p.price) OVER (PARTITION BY p.category) AS vs_avg
FROM products p;
Performance — O(N) requêtes imbriquées Une sous-requête corrélée dans SELECT ou WHERE est réexécutée pour chaque ligne de la requête externe. Sur 1 million de lignes, cela représente 1 million d'exécutions de la sous-requête. Toujours chercher une alternative (window function, CTE, JOIN) pour les gros volumes.

Quand utiliser malgré tout les sous-requêtes corrélées

Cas où la sous-requête corrélée reste pertinente
-- ① Petits volumes : lisibilité > performance
SELECT * FROM config c
WHERE value != (SELECT default_value FROM config_defaults WHERE key = c.key);

-- ② Logique conditionnelle complexe difficile à exprimer autrement
SELECT *
FROM accounts a
WHERE
  (a.type = 'premium' AND (SELECT COUNT(*) FROM logins WHERE user_id = a.id AND login_date > NOW() - INTERVAL '30 days') > 5)
  OR
  (a.type = 'standard' AND (SELECT COUNT(*) FROM orders WHERE customer_id = a.id) > 10);

-- ③ EXISTS (toujours un anti-join ou semi-join corrélé, mais court-circuite)
SELECT * FROM categories c
WHERE EXISTS (SELECT 1 FROM products WHERE category_id = c.id AND price < 10);

EXISTS vs IN vs JOIN — différences cruciales

⏱ 25 min Piège NULL

EXISTS, IN et JOIN peuvent souvent retourner le même résultat, mais leur comportement face aux NULLs, leur sémantique, et leurs performances divergent de façon significative. Choisir la bonne forme n'est pas qu'une question de style.

Sémantique fondamentale

Trois façons de tester l'appartenance
-- Question : quels clients ont passé au moins une commande > 100€ ?

-- ① IN : liste de valeurs, teste l'égalité
SELECT * FROM customers
WHERE id IN (SELECT customer_id FROM orders WHERE amount > 100);
-- Sémantique : "l'id est dans cette liste"
-- Évalué : la sous-requête retourne un ensemble, puis chaque id est cherché

-- ② EXISTS : test d'existence, corrélé
SELECT * FROM customers c
WHERE EXISTS (
  SELECT 1 FROM orders WHERE customer_id = c.id AND amount > 100
);
-- Sémantique : "il existe au moins une ligne satisfaisant cette condition"
-- Court-circuite : s'arrête dès le premier match trouvé

-- ③ JOIN : fusion de tables
SELECT DISTINCT c.*
FROM customers c
JOIN orders o ON c.id = o.customer_id AND o.amount > 100;
-- Sémantique : "fusion des deux tables"
-- Danger : DISTINCT obligatoire si o peut avoir plusieurs lignes par client

Différences face aux NULLs

Comportement des trois formes avec NULL
-- Test : clients dont l'id n'est PAS dans une liste suspecte

-- ① NOT IN avec NULL dans la sous-requête → résultat vide !
SELECT * FROM customers
WHERE id NOT IN (SELECT customer_id FROM blacklist);
-- Si blacklist contient customer_id = NULL : 0 lignes retournées
-- (voir L1.2 — logique à trois valeurs)

-- ② NOT EXISTS → immunisé contre les NULLs
SELECT * FROM customers c
WHERE NOT EXISTS (
  SELECT 1 FROM blacklist WHERE customer_id = c.id
);
-- NULL = NULL ne match jamais dans ON → pas de problème

-- ③ LEFT JOIN + IS NULL → aussi sûr
SELECT c.* FROM customers c
LEFT JOIN blacklist b ON c.id = b.customer_id
WHERE b.customer_id IS NULL;
-- Si b.customer_id est NULL dans la source → match accidentel ?
-- Tester sur une colonne NOT NULL de blacklist pour la sûreté

Quand IN est préférable

IN — cas d'usage appropriés
-- ① Liste statique et connue, sans NULL possible
SELECT * FROM products WHERE status IN ('active', 'pending', 'review');

-- ② Sous-requête de petite taille, non corrélée, valeurs sans NULL
SELECT * FROM users WHERE plan_id IN (
  SELECT id FROM plans WHERE tier = 'premium'
);
-- Performant si plans est petit et plan_id est indexé

-- ③ IN avec ANY (équivalent sémantique)
SELECT * FROM products
WHERE price > ANY (SELECT price FROM competitor_prices WHERE category = products.category);
-- ANY retourne TRUE si la condition est vraie pour au moins une valeur
-- ALL retourne TRUE si vraie pour toutes les valeurs

Quand JOIN est la bonne réponse

JOIN — quand vous avez besoin des colonnes des deux tables
-- JOIN est obligatoire si vous voulez des colonnes de B dans le SELECT
SELECT c.name, c.email, o.amount, o.order_date
FROM customers c
JOIN orders o ON c.id = o.customer_id;
-- EXISTS et IN ne peuvent pas "ramener" des colonnes de la sous-requête

-- JOIN pour agréger des informations de B sur A
SELECT c.name, COUNT(o.id) AS nb_orders, SUM(o.amount) AS total
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id
GROUP BY c.id, c.name;

Guide de décision

Arbre de décision EXISTS / IN / JOIN
-- Ai-je besoin de colonnes de la table B dans SELECT ?
--   OUI → JOIN (+ DISTINCT si besoin)
--   NON → continuer...

-- Est-ce un test de présence/absence ?
--   OUI → EXISTS / NOT EXISTS (sûr, performant, clair)
--   NON → continuer...

-- La sous-requête peut-elle contenir des NULLs et j'utilise NOT ?
--   OUI → NOT EXISTS (jamais NOT IN dans ce cas)
--   NON → IN peut convenir si la liste est petite et statique

-- La sous-requête est-elle grande (> 10 000 lignes) ?
--   OUI → EXISTS (court-circuit) ou JOIN avec index
--   NON → les trois formes ont des performances similaires
Bonne pratique Adoptez EXISTS comme réflexe par défaut pour les tests de présence/absence. Il est sémantiquement le plus précis ("est-ce qu'il existe ?"), immunisé contre les NULLs, et l'optimiseur peut le transformer en Hash Semi Join — l'une des jointures les plus efficaces.

LATERAL JOIN — sous-requêtes qui voient l'extérieur

⏱ 25 min Avancé

Une sous-requête dans le FROM ne peut normalement pas référencer les tables qui la précèdent. Le mot-clé LATERAL lève cette restriction : il permet à une sous-requête de voir et d'utiliser les colonnes des tables déjà définies dans le FROM. C'est l'équivalent SQL de l'application d'une fonction à chaque ligne d'une table.

LATERAL — pourquoi c'est différent

Sans vs avec LATERAL
-- ✗ ERREUR : la sous-requête ne peut pas voir 'c.id' de la requête externe
SELECT c.name, recent.amount
FROM customers c,
     (SELECT amount FROM orders WHERE customer_id = c.id LIMIT 1) recent;
-- ERROR: invalid reference to FROM-clause entry for table "c"

-- ✓ LATERAL : la sous-requête "voit" c.id
SELECT c.name, recent.amount
FROM customers c,
     LATERAL (SELECT amount FROM orders WHERE customer_id = c.id
              ORDER BY order_date DESC LIMIT 1) recent;
-- LATERAL est implicite quand on utilise une fonction en FROM

Cas d'usage 1 — Top-N par groupe sans window function

C'est le cas d'usage le plus courant de LATERAL : récupérer les N premières lignes pour chaque ligne de la table principale.

Top-3 produits par catégorie
-- Récupérer les 3 produits les plus chers de chaque catégorie
SELECT cat.name AS category, top_products.*
FROM categories cat
CROSS JOIN LATERAL (
  SELECT p.name, p.price
  FROM products p
  WHERE p.category_id = cat.id
  ORDER BY p.price DESC
  LIMIT 3
) top_products;
-- CROSS JOIN LATERAL : chaque catégorie est "envoyée" à la sous-requête
-- Equivalent à : pour chaque catégorie, donne-moi les 3 meilleurs produits

-- OUTER variant : inclure les catégories sans produits (OUTER résulte en NULLs)
SELECT cat.name, top.*
FROM categories cat
LEFT JOIN LATERAL (
  SELECT p.name, p.price FROM products p
  WHERE p.category_id = cat.id
  ORDER BY p.price DESC LIMIT 3
) top ON TRUE;
-- ON TRUE est requis avec LEFT JOIN LATERAL

Cas d'usage 2 — Enrichir chaque ligne avec un calcul contextualisé

LATERAL pour des calculs par ligne complexes
-- Pour chaque commande, trouver le dernier prix connu du produit
-- au moment de la commande (SCD Type 2)
SELECT
  o.id,
  o.product_id,
  o.quantity,
  hist_price.price,
  o.quantity * hist_price.price AS estimated_total
FROM orders o
CROSS JOIN LATERAL (
  SELECT price
  FROM price_history
  WHERE product_id = o.product_id
    AND effective_date <= o.order_date
  ORDER BY effective_date DESC
  LIMIT 1
) hist_price;

-- Alternative : inéquijoin (L2.4), mais LATERAL est plus lisible
-- et mieux optimisé quand price_history a un index sur (product_id, effective_date)

Cas d'usage 3 — Appliquer une fonction retournant plusieurs colonnes

LATERAL avec une fonction table (UNNEST, JSON)
-- Éclater un tableau JSON en lignes, tout en gardant le contexte parent
SELECT
  u.id AS user_id,
  u.name,
  tag.value AS interest_tag
FROM users u,
     LATERAL jsonb_array_elements_text(u.interests) AS tag(value);
-- u.interests = '["sql","python","data"]'
-- Résultat :
-- user_id | name  | interest_tag
--       1 | Alice | sql
--       1 | Alice | python
--       1 | Alice | data
--       2 | Bob   | sql

-- Sans LATERAL : UNNEST(array_col) fonctionne en SELECT direct
-- Mais LATERAL permet d'utiliser les colonnes de la ligne courante dans la fonction

LATERAL dans d'autres moteurs

Équivalents dans SQL Server, MySQL, BigQuery
-- SQL Server : CROSS APPLY / OUTER APPLY (équivalent à CROSS/LEFT JOIN LATERAL)
SELECT c.name, recent.amount
FROM customers c
CROSS APPLY (
  SELECT TOP 1 amount FROM orders WHERE customer_id = c.id ORDER BY order_date DESC
) recent;

-- MySQL 8.0+ : LATERAL est supporté
SELECT c.name, top3.*
FROM categories c
JOIN LATERAL (
  SELECT name, price FROM products WHERE category_id = c.id ORDER BY price DESC LIMIT 3
) top3 ON TRUE;

-- BigQuery : pas de LATERAL, mais les sous-requêtes scalaires corrélées
-- dans SELECT peuvent faire une partie du travail.
-- Pour le top-N : ARRAY_AGG avec OFFSET/LIMIT dans une sous-requête STRUCT

Sous-requêtes dans tous les contextes

⏱ 20 min Synthèse

Les sous-requêtes ne sont pas limitées au WHERE. Elles peuvent apparaître dans presque toutes les clauses SQL : SELECT, FROM, WHERE, HAVING, ORDER BY, et même dans les instructions DML (UPDATE, DELETE, INSERT). Chaque position a ses règles et ses cas d'usage spécifiques.

Sous-requête dans SELECT — valeur scalaire

Une sous-requête dans le SELECT doit retourner exactement une ligne et une colonne (valeur scalaire). Si elle retourne plusieurs lignes, le moteur lève une erreur.

Sous-requêtes scalaires dans SELECT
SELECT
  p.name,
  p.price,
  (SELECT AVG(price) FROM products) AS prix_moyen_global,
  p.price - (SELECT AVG(price) FROM products) AS ecart_moyen,
  (SELECT COUNT(*) FROM order_items WHERE product_id = p.id) AS nb_commandes
FROM products p;

-- ⚠ Chaque sous-requête dans SELECT est exécutée pour CHAQUE ligne
-- Si elles sont non-corrélées (pas de référence à p), elles peuvent être
-- "hissées" (hoisted) par l'optimiseur et exécutées une seule fois
-- Si corrélées (référence à p.id) → N exécutions → lent

-- ✓ Alternative performante via window function ou CTE
WITH stats AS (SELECT AVG(price) AS avg_price FROM products)
SELECT p.name, p.price, s.avg_price, p.price - s.avg_price AS ecart
FROM products p CROSS JOIN stats s;

Sous-requête dans HAVING

HAVING avec sous-requête
-- Catégories dont le CA total dépasse la moyenne de toutes les catégories
SELECT category, SUM(amount) AS ca
FROM orders o
JOIN products p ON o.product_id = p.id
GROUP BY category
HAVING SUM(amount) > (
  SELECT AVG(cat_total)
  FROM (
    SELECT SUM(amount) AS cat_total
    FROM orders oo JOIN products pp ON oo.product_id = pp.id
    GROUP BY pp.category
  ) sub
);
-- La sous-requête dans HAVING peut être complexe
-- mais attention à ne pas recalculer plusieurs fois la même agrégation

Sous-requête dans ORDER BY

Tri basé sur une sous-requête
-- Trier les clients par leur CA total (calculé dynamiquement)
SELECT id, name, email
FROM customers c
ORDER BY (
  SELECT COALESCE(SUM(amount), 0)
  FROM orders WHERE customer_id = c.id
) DESC;
-- Utile quand on ne veut pas agréger dans la requête principale
-- Peu performant sur gros volumes → préférer une CTE

Sous-requêtes dans les instructions DML

UPDATE, DELETE, INSERT avec sous-requêtes
-- UPDATE : mettre à jour selon une valeur calculée dans une autre table
UPDATE products p
SET price = price * 1.10
WHERE category_id IN (
  SELECT id FROM categories WHERE name = 'Premium'
);

-- UPDATE avec JOIN (PostgreSQL)
UPDATE orders o
SET status = 'flagged'
FROM customers c
WHERE o.customer_id = c.id AND c.risk_score > 8;

-- DELETE avec EXISTS
DELETE FROM sessions
WHERE NOT EXISTS (
  SELECT 1 FROM users WHERE id = sessions.user_id
);
-- Supprime les sessions orphelines (sans utilisateur existant)

-- INSERT ... SELECT (copie filtrée)
INSERT INTO archive_orders
SELECT * FROM orders
WHERE created_at < NOW() - INTERVAL '2 years';

-- UPSERT avec CTE (PostgreSQL)
WITH new_data AS (
  SELECT 1 AS id, 'Alice' AS name, 'alice@ex.com' AS email
)
INSERT INTO users (id, name, email)
SELECT * FROM new_data
ON CONFLICT (id) DO UPDATE SET name = EXCLUDED.name, email = EXCLUDED.email;

Opérateurs de comparaison avec ensembles

ANY, ALL, = (sous-requête scalaire)
-- = avec sous-requête scalaire : doit retourner exactement 1 ligne
SELECT * FROM products WHERE price = (SELECT MAX(price) FROM products);

-- ANY : vrai si la condition est vraie pour au moins une valeur
SELECT * FROM products
WHERE price > ANY (SELECT price FROM products WHERE category = 'budget');
-- Equivaut à : price > MIN(SELECT price FROM products WHERE category = 'budget')

-- ALL : vrai si la condition est vraie pour toutes les valeurs
SELECT * FROM products
WHERE price > ALL (SELECT price FROM products WHERE category = 'budget');
-- Equivaut à : price > MAX(SELECT price FROM products WHERE category = 'budget')
-- ⚠ ALL avec une sous-requête vide → TRUE (vacuously true)
-- ⚠ ALL avec NULL dans la sous-requête → peut retourner UNKNOWN
Note de lisibilité Pour les requêtes complexes avec plusieurs niveaux de sous-requêtes, les CTE (Module 4) offrent une meilleure lisibilité sans sacrifier les performances sur les moteurs modernes. La règle pratique : si votre sous-requête a plus d'un niveau d'imbrication ou est référencée plusieurs fois, transformez-la en CTE.

Module 4 — CTE › Leçon 1

WITH clause — syntaxe et structure

⏱ 20 min Fondamental

Une CTE (Common Table Expression) est une sous-requête nommée définie avec le mot-clé WITH avant la requête principale. Elle agit comme une table temporaire dont la portée est limitée à la requête courante. Le gain principal n'est pas toujours la performance, mais la lisibilité : elle permet de décomposer des requêtes complexes en étapes nommées et compréhensibles.

Syntaxe fondamentale

CTE simple
-- Structure générale
WITH nom_cte AS (
  -- Corps de la CTE : n'importe quelle requête SELECT
  SELECT ...
)
-- Requête principale qui utilise la CTE comme une table
SELECT * FROM nom_cte WHERE ...;

-- Exemple concret : commandes récentes par client actif
WITH recent_orders AS (
  SELECT
    customer_id,
    COUNT(*)       AS nb_orders,
    SUM(amount)    AS total_spent,
    MAX(order_date) AS last_order
  FROM orders
  WHERE order_date >= NOW() - INTERVAL '90 days'
  GROUP BY customer_id
)
SELECT
  c.name,
  c.email,
  ro.nb_orders,
  ro.total_spent,
  ro.last_order
FROM customers c
JOIN recent_orders ro ON c.id = ro.customer_id
WHERE ro.total_spent > 1000
ORDER BY ro.total_spent DESC;

CTE dans INSERT, UPDATE, DELETE

CTE dans des instructions DML
-- INSERT avec CTE
WITH new_customers AS (
  SELECT DISTINCT customer_id
  FROM external_orders
  WHERE customer_id NOT IN (SELECT id FROM customers)
)
INSERT INTO customers (id, source)
SELECT customer_id, 'import' FROM new_customers;

-- UPDATE avec CTE (PostgreSQL)
WITH order_totals AS (
  SELECT customer_id, SUM(amount) AS lifetime_value
  FROM orders GROUP BY customer_id
)
UPDATE customers c
SET tier = CASE
  WHEN ot.lifetime_value >= 10000 THEN 'platinum'
  WHEN ot.lifetime_value >= 1000  THEN 'gold'
  ELSE 'standard'
END
FROM order_totals ot
WHERE c.id = ot.customer_id;

-- DELETE avec CTE (PostgreSQL — CTE avec RETURNING)
WITH deleted AS (
  DELETE FROM sessions WHERE expires_at < NOW()
  RETURNING user_id, session_id
)
INSERT INTO audit_log (action, user_id, ref_id, occurred_at)
SELECT 'session_expired', user_id, session_id, NOW()
FROM deleted;
CTE avec RETURNING (PostgreSQL) PostgreSQL permet d'utiliser RETURNING dans une CTE DML. Cela vous permet de chaîner des opérations : DELETE puis log des lignes supprimées, UPDATE puis notifier les lignes modifiées. C'est une fonctionnalité très puissante pour les opérations atomiques complexes.

CTE vs sous-requête — quand choisir ?

CTE vs derived table — comparaison
-- Sous-requête imbriquée (difficile à lire et déboguer)
SELECT c.name, t.total
FROM customers c
JOIN (
  SELECT customer_id, SUM(
    amount * (1 - COALESCE(
      (SELECT discount FROM discounts WHERE order_id = o.id LIMIT 1), 0
    ) / 100.0)
  ) AS total
  FROM orders o
  WHERE status = 'completed'
  GROUP BY customer_id
) t ON c.id = t.customer_id;

-- Même logique avec CTE (lisible, débogable étape par étape)
WITH completed_orders AS (
  SELECT * FROM orders WHERE status = 'completed'
),
orders_with_discount AS (
  SELECT
    o.customer_id,
    o.amount,
    COALESCE(d.discount, 0) AS discount_pct
  FROM completed_orders o
  LEFT JOIN discounts d ON d.order_id = o.id
),
customer_totals AS (
  SELECT
    customer_id,
    SUM(amount * (1 - discount_pct / 100.0)) AS total
  FROM orders_with_discount
  GROUP BY customer_id
)
SELECT c.name, ct.total
FROM customers c
JOIN customer_totals ct ON c.id = ct.customer_id;
Règle pratique — CTE ou sous-requête ? Sous-requête si : usage unique, logique simple (≤ 5 lignes), pas de réutilisation prévue.
CTE si : logique complexe (> 5 lignes), réutilisation dans la même requête, vous voulez pouvoir tester chaque étape indépendamment, ou vous avez besoin de récursivité.

  2. Identifier les départements où tous les employés gagnent moins que le manager — deux CTE : une pour les salaires des managers, une pour les max salaires par département.

CTE multiples et chaînées

⏱ 20 min Pipeline SQL

L'une des forces des CTE est de pouvoir en définir plusieurs dans une même requête, chacune pouvant référencer les précédentes. Cela permet de construire un véritable pipeline de transformation en SQL, lisible de haut en bas comme des étapes d'un algorithme.

Plusieurs CTE dans un WITH

Syntaxe multi-CTE
-- Les CTE sont séparées par des virgules, sous un seul WITH
WITH
  cte1 AS (SELECT ...),
  cte2 AS (SELECT ... FROM cte1 ...),  -- peut référencer cte1
  cte3 AS (SELECT ... FROM cte1 JOIN cte2 ...)  -- peut référencer cte1 et cte2
SELECT ... FROM cte3;  -- la requête finale peut utiliser n'importe quelle CTE

Exemple réel — Pipeline d'analyse de ventes

Pipeline complet : staging → calcul → rapport
WITH
-- Étape 1 : filtrer et nettoyer les données sources
raw_sales AS (
  SELECT
    s.id,
    s.seller_id,
    s.product_id,
    s.amount,
    s.sale_date,
    s.region
  FROM sales s
  WHERE s.sale_date BETWEEN '2024-01-01' AND '2024-12-31'
    AND s.status = 'confirmed'
    AND s.amount > 0   -- exclure les annulations
),

-- Étape 2 : enrichir avec les informations produit
enriched AS (
  SELECT
    rs.*,
    p.category,
    p.unit_cost,
    rs.amount - p.unit_cost AS gross_margin
  FROM raw_sales rs
  JOIN products p ON rs.product_id = p.id
),

-- Étape 3 : agréger par vendeur et catégorie
seller_stats AS (
  SELECT
    seller_id,
    category,
    COUNT(*)           AS nb_ventes,
    SUM(amount)        AS ca,
    SUM(gross_margin)  AS marge_brute,
    AVG(amount)        AS panier_moyen
  FROM enriched
  GROUP BY seller_id, category
),

-- Étape 4 : calculer le rang par catégorie
ranked_sellers AS (
  SELECT
    *,
    RANK() OVER (PARTITION BY category ORDER BY ca DESC) AS rang_cat
  FROM seller_stats
)

-- Requête finale : top 5 par catégorie avec infos vendeur
SELECT
  v.name     AS vendeur,
  rs.category,
  rs.nb_ventes,
  rs.ca,
  ROUND(rs.marge_brute / rs.ca * 100, 1) AS marge_pct,
  rs.rang_cat
FROM ranked_sellers rs
JOIN vendors v ON rs.seller_id = v.id
WHERE rs.rang_cat <= 5
ORDER BY rs.category, rs.rang_cat;

Réutiliser une CTE dans plusieurs endroits

L'un des avantages majeurs des CTE sur les sous-requêtes : une CTE peut être référencée plusieurs fois dans la requête finale sans être réécrite.

CTE réutilisée plusieurs fois
WITH monthly_revenue AS (
  SELECT
    DATE_TRUNC('month', order_date) AS month,
    SUM(amount) AS revenue
  FROM orders
  GROUP BY 1
)
-- Référencée deux fois dans la requête principale
SELECT
  curr.month,
  curr.revenue,
  prev.revenue AS prev_revenue,
  ROUND((curr.revenue - prev.revenue) / prev.revenue * 100, 1) AS growth_pct
FROM monthly_revenue curr
LEFT JOIN monthly_revenue prev   -- même CTE jointe à elle-même !
  ON curr.month = prev.month + INTERVAL '1 month'
ORDER BY curr.month;

-- Note : selon le moteur (voir L4.4), monthly_revenue peut être
-- calculée une ou deux fois selon la stratégie de matérialisation
Pattern staging CTE Adoptez le pattern de nommer vos CTE selon leur rôle : raw_* pour les données brutes filtrées, enriched_* pour les jointures d'enrichissement, aggregated_* pour les agrégations, ranked_* pour les fenêtrages. Une requête nommée ainsi se lit comme une documentation.

CTE récursives — hiérarchies et traversée

⏱ 35 min Avancé

Les CTE récursives permettent d'écrire des requêtes qui s'appellent elles-mêmes, comme une boucle contrôlée. Elles sont indispensables pour parcourir des structures hiérarchiques (organigrammes, catégories imbriquées) et pour générer des séquences de données.

Structure d'une CTE récursive

Anatomie d'une CTE récursive
-- PostgreSQL, SQLite, SQL Server 2005+, MySQL 8.0+
WITH RECURSIVE nom_cte AS (
  -- ① ANCRE (anchor member) : point de départ, exécuté UNE fois
  SELECT colonne1, colonne2
  FROM table_source
  WHERE condition_de_depart

  UNION ALL   -- UNION ALL conserve les doublons (UNION les élimine, plus lent)

  -- ② MEMBRE RÉCURSIF : référence la CTE elle-même
  SELECT t.colonne1, t.colonne2
  FROM table_source t
  JOIN nom_cte c ON t.parent_id = c.id   -- ← join sur la CTE en cours
  -- ⚠ OBLIGATOIRE : une condition de terminaison dans le WHERE
  WHERE profondeur < 10  -- ou une condition naturelle d'arrêt
)
SELECT * FROM nom_cte;

Cas d'usage 1 — Hiérarchie organisationnelle

Organigramme — descendance complète
-- employees(id, name, manager_id, department)
-- Trouver toute la hiérarchie sous Alice (id=1)

WITH RECURSIVE org_chart AS (
  -- Ancre : point de départ (Alice)
  SELECT
    id,
    name,
    manager_id,
    0 AS depth,             -- niveau de profondeur
    ARRAY[id] AS path,      -- chemin parcouru (PostgreSQL)
    name AS hierarchy_path  -- chemin lisible
  FROM employees
  WHERE id = 1

  UNION ALL

  -- Récursion : trouver les subordonnés directs de chaque nœud courant
  SELECT
    e.id,
    e.name,
    e.manager_id,
    oc.depth + 1,
    oc.path || e.id,
    oc.hierarchy_path || ' > ' || e.name
  FROM employees e
  JOIN org_chart oc ON e.manager_id = oc.id
  WHERE NOT (e.id = ANY(oc.path))  -- détection de cycles
)
SELECT
  REPEAT('  ', depth) || name AS org_view,  -- indentation visuelle
  depth,
  hierarchy_path
FROM org_chart
ORDER BY path;

Cas d'usage 2 — Générer une série de dates

Générateur de dates récursif
-- Générer toutes les dates d'un mois (utile quand generate_series n'est pas dispo)
WITH RECURSIVE date_series AS (
  SELECT '2024-01-01'::DATE AS dt  -- ancre

  UNION ALL

  SELECT dt + INTERVAL '1 day'     -- récursion : jour suivant
  FROM date_series
  WHERE dt < '2024-01-31'          -- condition d'arrêt
)
SELECT dt FROM date_series;

-- Version pour toutes les semaines de 2024
WITH RECURSIVE weeks AS (
  SELECT DATE_TRUNC('week', '2024-01-01'::DATE)::DATE AS week_start

  UNION ALL

  SELECT week_start + 7
  FROM weeks
  WHERE week_start < '2024-12-31'
)
SELECT week_start, week_start + 6 AS week_end FROM weeks;

Cas d'usage 3 — Catégories imbriquées (arbre)

Arbre de catégories — récupérer tous les descendants
-- categories(id, name, parent_id)
-- Trouver toutes les sous-catégories d'une catégorie donnée

WITH RECURSIVE subcategories AS (
  SELECT id, name, parent_id, 1 AS level
  FROM categories
  WHERE id = 42   -- catégorie racine

  UNION ALL

  SELECT c.id, c.name, c.parent_id, sc.level + 1
  FROM categories c
  JOIN subcategories sc ON c.parent_id = sc.id
)
-- Compter les produits dans chaque sous-catégorie
SELECT
  sc.id,
  sc.name,
  sc.level,
  COUNT(p.id) AS product_count
FROM subcategories sc
LEFT JOIN products p ON p.category_id = sc.id
GROUP BY sc.id, sc.name, sc.level
ORDER BY sc.level, sc.name;
Risque d'infinité — condition de terminaison obligatoire Si votre condition d'arrêt est absente ou mal formulée, la CTE récursive tournera indéfiniment. PostgreSQL a un paramètre max_recursion_depth (défaut : 1000) pour éviter les boucles infinies. SQL Server a MAXRECURSION (défaut : 100, max : 32767). Ajoutez toujours une condition explicite dans le WHERE du membre récursif.

Détection de cycles

Éviter les boucles infinies sur des graphes cycliques
-- Si les données forment un graphe (pas un arbre pur), il peut y avoir des cycles
-- Technique : conserver un tableau des IDs visités

WITH RECURSIVE graph_traversal AS (
  SELECT id, ARRAY[id] AS visited
  FROM nodes WHERE id = 1

  UNION ALL

  SELECT n.id, gt.visited || n.id
  FROM nodes n
  JOIN edges e ON e.from_id = gt.id AND e.to_id = n.id
  JOIN graph_traversal gt ON TRUE
  WHERE NOT (n.id = ANY(gt.visited))  -- ← évite les cycles
)
SELECT * FROM graph_traversal;

-- PostgreSQL 14+ : clause CYCLE (standard SQL:2011)
WITH RECURSIVE cte AS (
  SELECT id, parent_id FROM tree WHERE id = 1
  UNION ALL
  SELECT t.id, t.parent_id FROM tree t JOIN cte ON t.parent_id = cte.id
) CYCLE id SET is_cycle USING path
SELECT * FROM cte WHERE NOT is_cycle;

Performance CTE — matérialisation et alternatives

⏱ 25 min Performance

Les CTE ne sont pas toujours plus performantes que les sous-requêtes — parfois elles le sont moins. La différence clé est la matérialisation : est-ce que le moteur exécute la CTE une seule fois et mémorise le résultat, ou la fusionne-t-il avec la requête externe comme une sous-requête inline ?

Matérialisation — le comportement historique

PostgreSQL < 12 — CTE = optimization fence
-- Avant PostgreSQL 12 : les CTE étaient TOUJOURS matérialisées
-- L'optimiseur ne pouvait pas "voir à travers" une CTE pour pousser des filtres

-- Exemple : filtrer dans la requête externe n'aide pas (pré-v12)
WITH all_orders AS (
  SELECT * FROM orders  -- scan complet de toute la table
)
SELECT * FROM all_orders WHERE customer_id = 42;
-- PostgreSQL < 12 : scannait tous les orders, PUIS filtrait
-- PostgreSQL 12+ : peut "inline" la CTE et pousser le filtre WHERE dans le scan

-- Vérification du plan d'exécution :
EXPLAIN (ANALYZE, COSTS)
WITH cte AS (SELECT * FROM orders)
SELECT * FROM cte WHERE customer_id = 42;

Contrôle explicite — MATERIALIZED / NOT MATERIALIZED

Forcer ou interdire la matérialisation (PostgreSQL 12+)
-- Forcer la matérialisation : utile quand la CTE est réutilisée plusieurs fois
-- et son calcul est coûteux (évite de la recalculer)
WITH MATERIALIZED expensive_calc AS (
  SELECT product_id, COUNT(*) AS cnt, SUM(amount) AS total
  FROM orders
  GROUP BY product_id
  HAVING COUNT(*) > 100
)
SELECT a.product_id, a.total, b.cnt
FROM expensive_calc a
JOIN expensive_calc b ON a.product_id = b.product_id  -- réutilisée 2 fois
WHERE a.total > 10000;
-- Sans MATERIALIZED : risque de calculer expensive_calc DEUX fois

-- Interdire la matérialisation : l'optimiseur doit pouvoir pousser les filtres
WITH NOT MATERIALIZED raw_data AS (
  SELECT * FROM events WHERE event_type = 'click'  -- filtre interne
)
SELECT * FROM raw_data WHERE user_id = 5;
-- Avec NOT MATERIALIZED, le WHERE user_id=5 peut être combiné avec le filtre interne

Comportements selon les moteurs

Comparaison des moteurs
-- PostgreSQL 12+ : inline par défaut (sauf CTE avec effets de bord)
-- PostgreSQL < 12 : matérialisé toujours

-- SQL Server : inline par défaut depuis longtemps
-- Mais : les CTE récursives sont toujours matérialisées étape par étape

-- MySQL 8.0+ : inline par défaut

-- Snowflake / BigQuery :
-- Chaque référence à une CTE peut déclencher un scan complet
-- Dans BigQuery, utilisez CREATE TEMP TABLE pour les gros calculs intermédiaires

-- BigQuery — CTE vs TEMP TABLE pour les gros volumes
-- Avec CTE (peut être recalculée à chaque référence) :
WITH heavy_cte AS (SELECT ... FROM big_table GROUP BY ...)
SELECT * FROM heavy_cte a JOIN heavy_cte b ON ...;  -- calcul x2 possible

-- Avec TEMP TABLE (calcul garanti une seule fois) :
CREATE TEMP TABLE heavy_results AS
SELECT ... FROM big_table GROUP BY ...;
SELECT * FROM heavy_results a JOIN heavy_results b ON ...;

CTE vs sous-requête vs table temporaire — quand choisir

Guide de choix
-- ① Sous-requête inline (derived table) → usage unique, simple
SELECT * FROM (SELECT ...) sub WHERE ...;
-- Quand : logique de 1 à 3 lignes, pas réutilisée, l'optimiseur peut l'inline

-- ② CTE → lisibilité, réutilisation modérée
WITH cte AS (...) SELECT ... FROM cte;
-- Quand : logique complexe, réutilisation dans la même requête,
--         ou pour déboguer (commenter/décommenter des étapes)

-- ③ Table temporaire → gros volumes, réutilisation garantie
CREATE TEMP TABLE temp_result AS SELECT ... FROM big_table;
CREATE INDEX ON temp_result(key_col);  -- indexer le résultat intermédiaire !
SELECT * FROM temp_result WHERE ...;   -- accès rapide
DROP TABLE temp_result;
-- Quand : résultat intermédiaire > 1M lignes, accès répété dans la même session

-- ④ Vue matérialisée → résultat précalculé, rafraîchi périodiquement
CREATE MATERIALIZED VIEW daily_stats AS
SELECT DATE_TRUNC('day', created_at), COUNT(*), SUM(amount) FROM orders GROUP BY 1;
REFRESH MATERIALIZED VIEW daily_stats;
-- Quand : requête lente et fréquemment exécutée, données fraiches acceptables
Technique de débogage — isoler une CTE Pour tester une CTE individuellement, terminez simplement la requête après la CTE :
WITH cte AS (...) SELECT * FROM cte LIMIT 100;
Cela vous permet de vérifier le résultat de chaque étape sans exécuter la requête complète.

GROUP BY — règles, pièges et colonnes multiples

⏱ 20 min Fondamental avancé

GROUP BY est la clause d'agrégation centrale de SQL. Derrière son apparente simplicité se cachent des règles strictes, des variations entre moteurs, et des patterns d'optimisation que tout data analyst doit maîtriser.

La règle fondamentale du GROUP BY

Toute colonne dans le SELECT doit soit être dans le GROUP BY, soit être enveloppée dans une fonction d'agrégation. C'est la règle de base — mais les moteurs la gèrent différemment.

Règles GROUP BY selon les moteurs
-- Standard SQL (PostgreSQL, SQL Server) : règle stricte
SELECT dept, name, AVG(salary)   -- ✗ ERREUR si name n'est pas dans GROUP BY
FROM employees GROUP BY dept;    -- ERROR: column "employees.name" must appear in...

-- MySQL en mode permissif (default avant 5.7) : pas d'erreur, mais valeur arbitraire
-- MySQL STRICT ou ONLY_FULL_GROUP_BY : même règle que PostgreSQL

-- Exception : dépendance fonctionnelle
-- Si name est fonctionnellement dépendant de la clé primaire dans GROUP BY,
-- PostgreSQL accepte (rare en pratique)
SELECT dept_id, d.name, COUNT(*)    -- d.name dépend de d.id (FK)
FROM employees e JOIN departments d ON e.dept_id = d.id
GROUP BY dept_id, d.id;             -- d.name implicitement autorisé (PostgreSQL)

GROUP BY sur plusieurs colonnes

Groupement multi-niveaux
-- Agréger par région ET catégorie
SELECT
  region,
  category,
  COUNT(*)       AS nb_orders,
  SUM(amount)    AS ca,
  AVG(amount)    AS panier_moyen
FROM orders o
JOIN products p ON o.product_id = p.id
GROUP BY region, category   -- groupe unique = (région, catégorie)
ORDER BY region, ca DESC;

-- GROUP BY 1, 2 : numéros de position (moins lisible mais compact)
SELECT region, category, SUM(amount)
FROM orders o JOIN products p ON o.product_id = p.id
GROUP BY 1, 2;  -- 1=region, 2=category

-- GROUP BY sur une expression
SELECT
  EXTRACT(YEAR FROM order_date)  AS year,
  EXTRACT(MONTH FROM order_date) AS month,
  SUM(amount) AS monthly_revenue
FROM orders
GROUP BY EXTRACT(YEAR FROM order_date), EXTRACT(MONTH FROM order_date)
ORDER BY year, month;

HAVING vs WHERE — timing et performances

Quand filtrer — WHERE ou HAVING ?
-- WHERE : filtre AVANT agrégation (plus efficace)
-- HAVING : filtre APRÈS agrégation (moins efficace, mais parfois obligatoire)

-- ✓ OPTIMAL : double filtre (WHERE réduit les lignes, HAVING sur l'agrégat)
SELECT dept, AVG(salary) AS avg_sal
FROM employees
WHERE hire_date >= '2020-01-01'   -- ← filtre pré-agrégation (réduire les lignes)
GROUP BY dept
HAVING AVG(salary) > 50000        -- ← filtre post-agrégation (condition sur agrégat)
  AND COUNT(*) >= 3;              -- ← condition sur le comptage (impossible en WHERE)

-- ✗ ERREUR CLASSIQUE : mettre dans HAVING ce qui devrait être dans WHERE
SELECT dept, AVG(salary)
FROM employees
GROUP BY dept
HAVING hire_date >= '2020-01-01';  -- ERREUR : hire_date n'est plus accessible ici

-- ⚠ PIÈGE : HAVING sans GROUP BY → filtre sur l'ensemble entier comme un groupe unique
SELECT COUNT(*), AVG(salary)
FROM employees
HAVING AVG(salary) > 50000;  -- retourne une ligne ou rien du tout

COUNT(*) vs COUNT(col) vs COUNT(DISTINCT col)

Variantes de COUNT — différences importantes
SELECT
  COUNT(*)                    AS total_lignes,        -- toutes les lignes, NULLs inclus
  COUNT(email)                AS avec_email,          -- non-NULL seulement
  COUNT(DISTINCT email)       AS emails_uniques,      -- valeurs distinctes non-NULL
  COUNT(DISTINCT customer_id, product_id) AS combos  -- MySQL: DISTINCT multi-col
FROM orders;

-- COUNT(DISTINCT) est coûteux : nécessite un tri ou un hash complet
-- Pour de grands volumes : utiliser HyperLogLog ou fonctions approximatives
SELECT
  approx_count_distinct(user_id) AS approx_users  -- BigQuery
  -- HLL_COUNT.MERGE_PARTIAL(HLL_COUNT.INIT(user_id)) AS approx  -- BigQuery HLL
  -- COUNT_DISTINCT_APPROX(user_id) AS approx  -- Snowflake
FROM events;
GROUP BY vs DISTINCT SELECT DISTINCT a, b FROM t et SELECT a, b FROM t GROUP BY a, b retournent le même résultat en termes de déduplication. Mais GROUP BY permet d'ajouter des agrégats. Utilisez DISTINCT uniquement quand vous ne faites pas d'agrégation — et interrogez-vous sur pourquoi vous avez des doublons en premier lieu.

ROLLUP, CUBE et GROUPING SETS

⏱ 30 min Analytique

SQL propose trois extensions de GROUP BY pour calculer simultanément des sous-totaux à plusieurs niveaux de granularité. Elles évitent d'écrire plusieurs requêtes distinctes et de les assembler avec UNION ALL.

ROLLUP — sous-totaux hiérarchiques

ROLLUP(a, b, c) calcule les agrégats pour toutes les combinaisons de "préfixes" : (a,b,c), (a,b), (a), et (). L'ordre des colonnes définit la hiérarchie.

ROLLUP — rapport avec sous-totaux
SELECT
  region,
  category,
  SUM(amount) AS total
FROM sales
GROUP BY ROLLUP(region, category)
ORDER BY region NULLS LAST, category NULLS LAST;

-- Résultat :
-- region   | category    | total
-- Europe   | Electronics | 50000
-- Europe   | Clothing    | 30000
-- Europe   | NULL        | 80000   ← sous-total Europe (toutes catégories)
-- Americas | Electronics | 40000
-- Americas | NULL        | 40000   ← sous-total Americas
-- NULL     | NULL        | 120000  ← grand total (toutes régions)

-- Note : les lignes de sous-total ont NULL dans les colonnes agrégées
-- (car elles représentent "toutes les valeurs" de cette dimension)

Identifier les lignes de sous-total avec GROUPING()

GROUPING() — distinguer sous-totaux des vrais NULLs
-- GROUPING(col) retourne 1 si la colonne est agrégée dans cette ligne, 0 sinon
-- Utile pour distinguer un NULL "agrégé" d'un vrai NULL dans les données

SELECT
  CASE WHEN GROUPING(region) = 1   THEN 'Toutes régions'   ELSE region   END AS region,
  CASE WHEN GROUPING(category) = 1 THEN 'Toutes catégories' ELSE category END AS category,
  SUM(amount) AS total,
  GROUPING(region) AS is_region_total,     -- 0 ou 1
  GROUPING(category) AS is_cat_total       -- 0 ou 1
FROM sales
GROUP BY ROLLUP(region, category)
ORDER BY GROUPING(region), region, GROUPING(category), category;

CUBE — toutes les combinaisons

CUBE(a, b, c) génère les agrégats pour toutes les combinaisons possibles : 2³ = 8 niveaux pour 3 colonnes. C'est l'outil idéal pour les rapports cross-dimensionnels (type tableur croisé).

CUBE — rapport multidimensionnel complet
SELECT
  year, region, category,
  SUM(amount) AS total
FROM sales
GROUP BY CUBE(year, region, category)
ORDER BY year NULLS LAST, region NULLS LAST, category NULLS LAST;

-- Génère les agrégats pour :
-- (year, region, category) → détail complet
-- (year, region)           → par année et région
-- (year, category)         → par année et catégorie
-- (region, category)       → par région et catégorie
-- (year)                   → par année seulement
-- (region)                 → par région seulement
-- (category)               → par catégorie seulement
-- ()                       → grand total
-- Avec 3 dimensions → 2^3 = 8 combinaisons d'agrégation

GROUPING SETS — contrôle précis

GROUPING SETS vous permet de spécifier exactement les combinaisons souhaitées, sans calculer toutes les combinaisons comme CUBE ni uniquement les préfixes comme ROLLUP.

GROUPING SETS — flexibilité maximale
-- Calculer uniquement les agrégats dont on a besoin
SELECT
  region, category, month, SUM(amount)
FROM sales
GROUP BY GROUPING SETS (
  (region, category, month),   -- détail complet
  (region, month),             -- par région × mois (pas catégorie)
  (category),                  -- par catégorie seulement
  ()                           -- grand total
);
-- ROLLUP = GROUPING SETS sur les préfixes successifs
-- CUBE   = GROUPING SETS sur toutes les combinaisons
-- GROUPING SETS = contrôle total : exactement ce dont vous avez besoin

-- Équivalence ROLLUP → GROUPING SETS :
GROUP BY ROLLUP(a, b)
-- est équivalent à :
GROUP BY GROUPING SETS((a,b), (a), ())

Rapport de dashboard avec ROLLUP

  SELECT
    c.country, c.city,
    e.name AS seller,

FILTER clause et agrégations conditionnelles

⏱ 20 min Pivot SQL

L'agrégation conditionnelle permet de calculer plusieurs métriques différentes sur les mêmes données en une seule passe, en filtrant les lignes à l'intérieur de chaque fonction d'agrégation. C'est le mécanisme sous-jacent du "pivot" en SQL.

La clause FILTER — standard SQL moderne

FILTER WHERE — syntaxe propre et lisible
-- Compter et sommer selon des conditions différentes en une passe
SELECT
  COUNT(*)                                    AS total_orders,
  COUNT(*) FILTER (WHERE status = 'completed')  AS completed,
  COUNT(*) FILTER (WHERE status = 'cancelled')  AS cancelled,
  COUNT(*) FILTER (WHERE status = 'pending')    AS pending,
  SUM(amount) FILTER (WHERE status = 'completed') AS revenue_completed,
  AVG(amount) FILTER (WHERE amount > 1000)       AS avg_high_value
FROM orders;

-- FILTER est équivalent au pattern CASE WHEN mais plus lisible
-- Supporté : PostgreSQL 9.4+, SQLite 3.30+, DuckDB
-- Non supporté : MySQL, SQL Server (utiliser CASE WHEN)

CASE WHEN dans une agrégation — portable sur tous les moteurs

Pattern CASE WHEN — universel
-- Équivalent de FILTER mais fonctionne sur MySQL, SQL Server, etc.
SELECT
  COUNT(CASE WHEN status = 'completed' THEN 1 END) AS completed,
  -- Note : COUNT(CASE WHEN ... THEN 1 ELSE NULL END) = même chose
  -- NULL est ignoré par COUNT, donc la clause ELSE NULL est optionnelle

  SUM(CASE WHEN status = 'completed' THEN amount ELSE 0 END) AS revenue_completed,
  -- Ici ELSE 0 est important : sans lui, les non-completed sont NULL, pas 0
  -- Pour COUNT → ELSE NULL, pour SUM → ELSE 0

  AVG(CASE WHEN region = 'EU' THEN amount END) AS avg_eu_order
  -- Moyenne uniquement des commandes EU (ignore les autres → NULL ignoré)
FROM orders;

Pattern PIVOT — tableau croisé avec agrégations conditionnelles

Pivot manuel — une colonne de valeurs → plusieurs colonnes
-- Source : une ligne par (mois, catégorie, montant)
-- Cible : une ligne par mois, une colonne par catégorie

SELECT
  DATE_TRUNC('month', sale_date) AS month,
  SUM(amount) FILTER (WHERE category = 'Electronics') AS electronics,
  SUM(amount) FILTER (WHERE category = 'Clothing')    AS clothing,
  SUM(amount) FILTER (WHERE category = 'Books')       AS books,
  SUM(amount) FILTER (WHERE category = 'Food')        AS food,
  SUM(amount) AS total  -- toutes catégories confondues
FROM sales
GROUP BY DATE_TRUNC('month', sale_date)
ORDER BY month;

-- Variation : taux de complétion par catégorie
SELECT
  category,
  COUNT(*) AS total,
  ROUND(
    100.0 * COUNT(*) FILTER (WHERE status = 'paid') / COUNT(*),
    1
  ) AS payment_rate_pct
FROM orders
GROUP BY category;

Patterns avancés avec CASE/FILTER
-- Calculer des métriques glissantes dans la même requête
SELECT
  customer_id,
  COUNT(*)                                          AS total_orders,
  COUNT(*) FILTER (WHERE order_date >= NOW()-'30d') AS orders_last_30d,

  product_id,

FROM product_reviews

⏱ 25 min Statistiques

Au-delà de COUNT, SUM, AVG, SQL propose un riche écosystème de fonctions d'agrégation pour les analyses statistiques, la manipulation de texte, et la génération de structures de données complexes.

Agrégation de texte — STRING_AGG et GROUP_CONCAT

Concaténer des valeurs d'un groupe
-- PostgreSQL : STRING_AGG(expression, délimiteur ORDER BY ...)
SELECT
  customer_id,
  STRING_AGG(product_name, ', ' ORDER BY purchase_date) AS products_bought,
  STRING_AGG(DISTINCT product_name, ', ') AS unique_products  -- avec DISTINCT
FROM purchases
JOIN products ON product_id = products.id
GROUP BY customer_id;

-- MySQL : GROUP_CONCAT (similaire mais syntaxe différente)
SELECT customer_id,
  GROUP_CONCAT(product_name ORDER BY purchase_date SEPARATOR ', ') AS products
FROM purchases JOIN products USING(product_id)
GROUP BY customer_id;

-- SQL Server : STRING_AGG (disponible depuis 2017)
SELECT customer_id,
  STRING_AGG(product_name, ', ') WITHIN GROUP (ORDER BY purchase_date) AS products
FROM purchases JOIN products ON product_id = products.id
GROUP BY customer_id;

ARRAY_AGG et JSON_AGG (PostgreSQL)

Agréger en tableau ou en JSON
-- ARRAY_AGG : groupe de valeurs → tableau PostgreSQL
SELECT
  customer_id,
  ARRAY_AGG(product_id ORDER BY purchase_date) AS product_sequence,
  ARRAY_AGG(DISTINCT product_id)               AS unique_products,
  ARRAY_LENGTH(ARRAY_AGG(DISTINCT product_id), 1) AS nb_products_distincts
FROM purchases
GROUP BY customer_id;

-- JSON_AGG : groupe de lignes → tableau JSON
SELECT
  customer_id,
  JSON_AGG(
    JSON_BUILD_OBJECT(
      'product', product_name,
      'amount', amount,
      'date', purchase_date

Fonctions de percentile et médiane

La médiane et les percentiles sont impossibles à calculer avec AVG. SQL propose deux fonctions d'ordre d'ensemble pour cela.

Médiane et percentiles
-- PERCENTILE_CONT : interpolation continue (peut retourner une valeur non présente)
-- PERCENTILE_DISC : valeur discrète (retourne une valeur réellement présente)
SELECT
  AVG(salary)                                          AS moyenne,
  PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY salary)  AS mediane,
  PERCENTILE_DISC(0.5) WITHIN GROUP (ORDER BY salary)  AS mediane_discrete,
  PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY salary) AS Q1,
  PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY salary) AS Q3,
  PERCENTILE_CONT(0.9)  WITHIN GROUP (ORDER BY salary) AS P90
FROM employees;

-- Différence CONT vs DISC :
-- Salaires [10, 20, 30, 40, 50] : médiane = 30 pour les deux
-- Salaires [10, 20, 30, 40] :
--   CONT(0.5) → 25 (interpolation entre 20 et 30)
--   DISC(0.5) → 20 (valeur la plus basse qui couvre 50% des données)

-- Par groupe (avec GROUP BY)
SELECT dept, PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY salary) AS median_salary
FROM employees GROUP BY dept;

BOOL_AND et BOOL_OR (PostgreSQL)

Agrégations booléennes
-- BOOL_AND : vrai si TOUTES les lignes du groupe satisfont la condition
-- BOOL_OR  : vrai si AU MOINS UNE ligne satisfait la condition
SELECT

  BOOL_OR(item_shipped)  AS partially_shipped -- au moins une expédiée ?
FROM order_items
GROUP BY order_id;

SELECT
  order_id,

  CASE WHEN MAX(CASE WHEN item_shipped THEN 1 ELSE 0 END) = 1 THEN TRUE ELSE FALSE END AS partially_shipped
FROM order_items GROUP BY order_id;

Module 6 — Window Functions › Leçon 1

Concept de fenêtre — OVER(), PARTITION BY, ORDER BY

⏱ 30 min Fondation essentielle

Les window functions (fonctions de fenêtrage) sont la fonctionnalité la plus puissante de SQL analytique. Contrairement à GROUP BY qui compresse les lignes en groupes, les window functions calculent une valeur pour chaque ligne en tenant compte de ses lignes voisines, sans réduire le nombre de lignes du résultat.

La différence fondamentale avec GROUP BY

GROUP BY vs Window Function — résultats différents
-- Table employees(id, name, dept, salary)
-- Alice | Engineering | 90000
-- Bob   | Engineering | 75000
-- Carol | Marketing   | 80000

-- GROUP BY : compresse en 2 lignes (une par dept)
SELECT dept, AVG(salary) AS avg_salary
FROM employees GROUP BY dept;
-- Engineering | 82500
-- Marketing   | 80000

-- Window function : garde toutes les lignes, ajoute le contexte
SELECT name, dept, salary,
  AVG(salary) OVER (PARTITION BY dept) AS dept_avg_salary
FROM employees;
-- Alice | Engineering | 90000 | 82500
-- Bob   | Engineering | 75000 | 82500
-- Carol | Marketing   | 80000 | 80000
-- On peut voir le salaire individuel ET la moyenne du département !

Anatomie de la clause OVER()

Syntaxe complète d'une window function
fonction_window() OVER (
  [PARTITION BY col1, col2, ...]  -- découpe en partitions (comme GROUP BY)
  [ORDER BY col3 ASC/DESC, ...]   -- ordre au sein de la partition
  [frame_clause]                  -- fenêtre de lignes (voir L6.5)
)

-- Exemples :
AVG(salary) OVER ()                         -- fenêtre = toutes les lignes
AVG(salary) OVER (PARTITION BY dept)        -- fenêtre = toutes les lignes du dept
ROW_NUMBER() OVER (ORDER BY salary DESC)    -- rang global par salaire
ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) -- rang par dept

PARTITION BY — découper en sous-fenêtres

PARTITION BY — fenêtre indépendante par groupe
SELECT
  name, dept, salary,
  -- Fenêtre globale : max sur TOUS les employés
  MAX(salary) OVER () AS company_max,
  -- Fenêtre par département : max dans chaque dept séparément
  MAX(salary) OVER (PARTITION BY dept) AS dept_max,
  -- Ratio : salaire / max du département
  ROUND(salary * 100.0 / MAX(salary) OVER (PARTITION BY dept), 1) AS pct_of_dept_max
FROM employees;

-- PARTITION BY peut accepter plusieurs colonnes et des expressions
SELECT *,
  COUNT(*) OVER (PARTITION BY dept, EXTRACT(YEAR FROM hire_date)) AS hired_same_year_dept
FROM employees;

ORDER BY dans OVER — cumul et séquencement

ORDER BY dans OVER — attention au frame par défaut
-- ORDER BY dans OVER crée un ordre au sein de la partition
-- Le frame par défaut devient RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
-- (cumulatif) — voir L6.5 pour les détails

SELECT
  order_date,
  amount,
  -- Running total (cumulatif) — frame par défaut avec ORDER BY
  SUM(amount) OVER (ORDER BY order_date) AS running_total,
  -- Rang dans tout le dataset
  RANK() OVER (ORDER BY amount DESC)     AS global_rank
FROM orders;

-- Sans ORDER BY : la frame est la partition entière (comportement non-cumulatif)
SELECT
  dept,
  salary,
  SUM(salary) OVER (PARTITION BY dept) AS dept_total  -- total du dept, pas cumulatif
FROM employees;

Les window functions ne sont pas dans WHERE

Limitation — pas de window function dans WHERE ou HAVING
-- ✗ ERREUR : window function interdite dans WHERE
SELECT *, ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) AS rn
FROM employees
WHERE rn = 1;  -- ERROR

-- ✓ CORRECT : CTE ou sous-requête
WITH ranked AS (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) AS rn
  FROM employees
)
SELECT * FROM ranked WHERE rn = 1;

-- ✓ AUSSI CORRECT via sous-requête (moins lisible)
SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) AS rn
  FROM employees
) t WHERE rn = 1;

WINDOW clause — nommer une fenêtre réutilisable

WINDOW — éviter la répétition
-- Sans WINDOW : répétition de la définition
SELECT name, dept, salary,
  RANK()    OVER (PARTITION BY dept ORDER BY salary DESC) AS rank_sal,
  DENSE_RANK() OVER (PARTITION BY dept ORDER BY salary DESC) AS dense_rank_sal,
  ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) AS row_num
FROM employees;

-- Avec WINDOW : définition unique, réutilisée (PostgreSQL, standard SQL)
SELECT name, dept, salary,
  RANK()       OVER dept_salary_window AS rank_sal,
  DENSE_RANK() OVER dept_salary_window AS dense_rank_sal,
  ROW_NUMBER() OVER dept_salary_window AS row_num
FROM employees
WINDOW dept_salary_window AS (PARTITION BY dept ORDER BY salary DESC);

Module 6 — Window Functions › Leçon 2

ROW_NUMBER, RANK, DENSE_RANK, NTILE

⏱ 25 min Rang et classement

Les fonctions de rang attribuent une position à chaque ligne au sein d'une partition. Leurs différences — subtiles mais importantes — déterminent le comportement en cas d'égalité.

Les quatre fonctions de rang comparées

Comparaison côte à côte
SELECT
  name, score,
  ROW_NUMBER()  OVER (ORDER BY score DESC) AS row_num,   -- toujours unique
  RANK()        OVER (ORDER BY score DESC) AS rnk,        -- gaps après égalité
  DENSE_RANK()  OVER (ORDER BY score DESC) AS dense_rnk,  -- pas de gaps
  NTILE(4)      OVER (ORDER BY score DESC) AS quartile     -- quartile (1=top)
FROM contestants;

-- score | row_num | rnk | dense_rnk | quartile
-- 95    |    1    |  1  |     1     |    1
-- 90    |    2    |  2  |     2     |    1
-- 90    |    3    |  2  |     2     |    2   ← même score, même rang, bucket différent
-- 85    |    4    |  4  |     3     |    2   ← RANK saute à 4 (2+2 ex-aequo), DENSE à 3
-- 80    |    5    |  5  |     4     |    3
-- 75    |    6    |  6  |     5     |    3
-- 70    |    7    |  7  |     6     |    4
-- 65    |    8    |  8  |     7     |    4

ROW_NUMBER — déduplication et top-1

ROW_NUMBER — usage courant en déduplication
-- Pattern déduplication : garder la ligne la plus récente par ID
WITH deduped AS (
  SELECT *,
    ROW_NUMBER() OVER (
      PARTITION BY user_id          -- un groupe par utilisateur
      ORDER BY updated_at DESC      -- plus récent en premier → rn=1
    ) AS rn
  FROM user_profiles
)
SELECT * FROM deduped WHERE rn = 1;

-- ROW_NUMBER attribue toujours des numéros uniques
-- même si deux lignes ont le même updated_at (arbitraire mais déterministe si ORDER BY unique)

-- Pattern top-N : 3 meilleures ventes par région
WITH ranked_sales AS (
  SELECT *,
    ROW_NUMBER() OVER (PARTITION BY region ORDER BY amount DESC) AS rn
  FROM sales
)
SELECT * FROM ranked_sales WHERE rn <= 3;

RANK — classements sportifs

RANK — permet les ex-aequo avec gaps
-- Les ex-aequo reçoivent le même rang, le rang suivant est sauté
SELECT
  student_id,
  grade,
  RANK() OVER (ORDER BY grade DESC) AS class_rank
FROM exam_results;

-- Usage : classements où les ex-aequo méritent le même rang
-- Médaille d'or pour deux coureurs au même temps : rang 1 pour les deux
-- Pas de médaille d'argent dans ce cas (gap : le suivant est rang 3)

-- Trouver les étudiants dans le top 10% de leur classe
WITH ranked AS (
  SELECT *, RANK() OVER (PARTITION BY class_id ORDER BY grade DESC) AS rnk,
            COUNT(*) OVER (PARTITION BY class_id) AS class_size
  FROM students
)
SELECT * FROM ranked WHERE rnk <= CEIL(class_size * 0.1);

DENSE_RANK — classements continus

DENSE_RANK — pas de sauts
-- DENSE_RANK : pas de gaps, bon pour les tranches de performance
SELECT
  dept, name, salary,
  DENSE_RANK() OVER (PARTITION BY dept ORDER BY salary DESC) AS sal_tier
FROM employees;

-- Avec DENSE_RANK, on peut facilement trouver les N premiers "niveaux" de salaire
-- Exemple : les 3 premiers niveaux de salaire (peut inclure plusieurs employés par niveau)
WITH tiers AS (
  SELECT *, DENSE_RANK() OVER (PARTITION BY dept ORDER BY salary DESC) AS tier
  FROM employees
)
SELECT * FROM tiers WHERE tier <= 3;
-- Vs ROW_NUMBER : ROW_NUMBER ≤ 3 retourne exactement 3 lignes par dept
-- DENSE_RANK ≤ 3 peut retourner plus de 3 lignes si des ex-aequo existent

NTILE — segmentation en buckets

NTILE — déciles, quartiles, percentiles
-- NTILE(n) divise les lignes en n buckets aussi égaux que possible
-- Les buckets plus larges viennent en premier (si non divisible)

-- Quartiles de revenus clients
SELECT
  customer_id,
  total_revenue,
  NTILE(4) OVER (ORDER BY total_revenue) AS quartile,
  -- Labelliser
  CASE NTILE(4) OVER (ORDER BY total_revenue)
    WHEN 1 THEN 'Bronze'
    WHEN 2 THEN 'Silver'
    WHEN 3 THEN 'Gold'
    WHEN 4 THEN 'Platinum'
  END AS tier
FROM customer_revenue;

-- Déciles (10 buckets)
SELECT *, NTILE(10) OVER (ORDER BY score DESC) AS decile
FROM test_scores;
-- decile=1 → top 10%, decile=10 → bottom 10%

-- Attention : NTILE(n) sur M lignes avec M non multiple de n
-- crée des buckets de tailles légèrement inégales (certains ont M%n lignes en plus)

Module 6 — Window Functions › Leçon 3

LAG, LEAD, FIRST_VALUE, LAST_VALUE

⏱ 25 min Piège LAST_VALUE

Les fonctions de décalage permettent d'accéder à des lignes voisines (précédentes ou suivantes) dans la fenêtre sans effectuer de SELF JOIN. Ce sont les outils de base pour les comparaisons temporelles et les analyses de séquences.

LAG et LEAD — accès aux lignes adjacentes

LAG et LEAD — syntaxe et paramètres
-- LAG(expr, n, default)  → valeur n lignes AVANT la ligne courante
-- LEAD(expr, n, default) → valeur n lignes APRÈS la ligne courante
-- n      : décalage (défaut = 1)
-- default: valeur si la ligne n'existe pas (défaut = NULL)

SELECT
  sale_date,
  revenue,
  -- Revenu du mois précédent (1 rang en arrière)
  LAG(revenue, 1, 0) OVER (ORDER BY sale_date) AS prev_month,
  -- Variation par rapport au mois précédent
  revenue - LAG(revenue, 1) OVER (ORDER BY sale_date) AS mom_delta,
  -- Taux de croissance mois/mois
  ROUND(
    (revenue - LAG(revenue) OVER (ORDER BY sale_date))
    / NULLIF(LAG(revenue) OVER (ORDER BY sale_date), 0) * 100,
    1
  ) AS mom_growth_pct,
  -- Revenu du mois suivant (pour comparer avec le futur)
  LEAD(revenue, 1) OVER (ORDER BY sale_date) AS next_month
FROM monthly_sales;

LAG avec PARTITION BY — comparaison par groupe

LAG par partition — variation year-over-year
-- Comparaison année par année par département
SELECT
  dept,
  year,
  headcount,
  LAG(headcount) OVER (PARTITION BY dept ORDER BY year) AS prev_year_headcount,
  headcount - LAG(headcount) OVER (PARTITION BY dept ORDER BY year) AS yoy_change
FROM dept_headcount;

-- Note : la partition RESET à chaque nouveau département
-- LAG de la première ligne d'une partition → NULL (ou la valeur par défaut)

FIRST_VALUE et LAST_VALUE

FIRST_VALUE et le piège de LAST_VALUE
SELECT
  name, dept, salary,
  -- Premier salaire du département (trié par hire_date)
  FIRST_VALUE(salary) OVER (
    PARTITION BY dept ORDER BY hire_date
  ) AS first_hire_salary,

  -- ⚠ PIÈGE : LAST_VALUE avec frame par défaut ne donne pas la dernière !
  -- Frame par défaut avec ORDER BY : RANGE UNBOUNDED PRECEDING AND CURRENT ROW
  -- → LAST_VALUE retourne la ligne COURANTE, pas la dernière du groupe !
  LAST_VALUE(salary) OVER (
    PARTITION BY dept ORDER BY hire_date
    -- Frame par défaut → valeur actuelle, PAS la dernière du groupe
  ) AS last_value_wrong,   -- en fait = salary (ligne courante) !

  -- ✓ CORRECT : étendre la frame jusqu'à la fin de la partition
  LAST_VALUE(salary) OVER (
    PARTITION BY dept ORDER BY hire_date
    ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
  ) AS last_hire_salary_correct
FROM employees;
LAST_VALUE — le piège le plus classique LAST_VALUE(col) OVER (ORDER BY date) retourne la valeur de la ligne courante, pas de la dernière ligne de la partition. Il faut toujours spécifier la frame : ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING. Ce bug ne produit pas d'erreur — il donne silencieusement un résultat incorrect.

NTH_VALUE — n-ième valeur dans la fenêtre

NTH_VALUE
-- Récupérer la 2ème plus haute valeur par département
SELECT
  name, dept, salary,
  NTH_VALUE(salary, 2) OVER (
    PARTITION BY dept
    ORDER BY salary DESC
    ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
  ) AS second_highest_salary
FROM employees;
-- Note : NTH_VALUE a aussi le problème de frame comme LAST_VALUE
-- Toujours spécifier la frame pour des résultats corrects

-- Alternatively : plus simple avec OFFSET dans une CTE
WITH ranked AS (
  SELECT *, DENSE_RANK() OVER (PARTITION BY dept ORDER BY salary DESC) AS dr
  FROM employees
)
SELECT dept, MAX(salary) AS second_highest FROM ranked WHERE dr = 2 GROUP BY dept;

Cas d'usage — détection de changements d'état

LAG pour détecter les transitions
-- Détecter quand le statut d'un abonnement change
SELECT
  user_id,
  subscription_date,
  status,
  LAG(status) OVER (PARTITION BY user_id ORDER BY subscription_date) AS prev_status,
  CASE
    WHEN LAG(status) OVER (PARTITION BY user_id ORDER BY subscription_date) IS NULL
      THEN 'new_user'
    WHEN status = 'active' AND LAG(status) OVER (...) = 'churned'
      THEN 'reactivation'
    WHEN status = 'churned' AND LAG(status) OVER (...) = 'active'
      THEN 'churn_event'
    ELSE 'no_change'
  END AS transition_type
FROM subscription_history;

-- Astuce : nommer la window pour éviter la répétition
SELECT
  user_id, subscription_date, status,
  LAG(status) OVER w AS prev_status,
  LEAD(status) OVER w AS next_status
FROM subscription_history
WINDOW w AS (PARTITION BY user_id ORDER BY subscription_date);

Agrégations glissantes — running totals et moving averages

⏱ 25 min Analytics courant

Les agrégations avec fenêtre temporelle sont au cœur de l'analyse de séries temporelles : cumuls (running totals), moyennes glissantes (moving averages), pourcentages de totaux. Ces patterns apparaissent dans presque tous les tableaux de bord analytiques.

Running Total — cumul progressif

Cumul progressif par date
SELECT
  sale_date,
  daily_revenue,
  -- Running total global (depuis le début de la série)
  SUM(daily_revenue) OVER (
    ORDER BY sale_date
    -- Frame par défaut avec ORDER BY : RANGE UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS cumulative_revenue,

  -- Running total par mois (reset chaque mois)
  SUM(daily_revenue) OVER (
    PARTITION BY DATE_TRUNC('month', sale_date)
    ORDER BY sale_date
  ) AS monthly_cumulative,

  -- Cumul en % du total global
  ROUND(
    SUM(daily_revenue) OVER (ORDER BY sale_date) * 100.0
    / SUM(daily_revenue) OVER (),   -- SUM sur toute la table
    1
  ) AS cumulative_pct
FROM daily_sales
ORDER BY sale_date;

Moving Average — moyenne glissante

Moyenne glissante sur N jours
SELECT
  sale_date,
  revenue,
  -- Moyenne glissante sur 7 jours (incluant le jour courant)
  AVG(revenue) OVER (
    ORDER BY sale_date
    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
  ) AS ma_7d,

  -- Moyenne glissante sur 30 jours
  AVG(revenue) OVER (
    ORDER BY sale_date
    ROWS BETWEEN 29 PRECEDING AND CURRENT ROW
  ) AS ma_30d,

  -- Median glissante 7 jours (plus complexe)
  PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY revenue) -- pas de version fenêtrée standard
FROM daily_sales;

-- Note : ROWS est approprié ici (voir L6.5 pour ROWS vs RANGE)
-- ROWS compte les lignes physiques, parfait pour les séries temporelles quotidiennes
-- Si plusieurs lignes par date → ROWS donne plus de contrôle que RANGE

Ratio et pourcentage de total

Part de chaque ligne dans son groupe
SELECT
  dept,
  name,
  salary,
  -- Part du salaire dans le total du département
  ROUND(salary * 100.0 / SUM(salary) OVER (PARTITION BY dept), 1) AS dept_share_pct,
  -- Part dans le total global
  ROUND(salary * 100.0 / SUM(salary) OVER (), 1) AS global_share_pct,

  -- Rang en % (CUME_DIST : fraction des lignes <= à la valeur courante)
  ROUND(CUME_DIST() OVER (ORDER BY salary) * 100, 1) AS percentile_rank

FROM employees;

Running COUNT et DISTINCT approximatif

Comptage cumulatif
-- Running count : nombre de lignes vues jusqu'ici
SELECT
  event_date,
  user_id,
  COUNT(*) OVER (ORDER BY event_date) AS cumulative_events,
  -- COUNT(*) cumulatif par utilisateur
  COUNT(*) OVER (PARTITION BY user_id ORDER BY event_date) AS user_event_count

FROM events;

-- Running COUNT DISTINCT : difficile en SQL standard
-- Solution : approche CTE + window function
WITH user_firsts AS (
  SELECT user_id, MIN(event_date) AS first_seen
  FROM events GROUP BY user_id
)
SELECT
  first_seen,
  COUNT(*) AS new_users_that_day,
  SUM(COUNT(*)) OVER (ORDER BY first_seen) AS cumulative_users
FROM user_firsts
GROUP BY first_seen;
-- Cette approche compte les nouveaux utilisateurs par jour puis cumule

Max glissant et Min glissant

Running max / min — prix record, plus bas historique
SELECT
  price_date,
  close_price,
  -- Maximum cumulatif (all-time high à date)
  MAX(close_price) OVER (ORDER BY price_date) AS all_time_high,
  -- Minimum cumulatif (all-time low à date)
  MIN(close_price) OVER (ORDER BY price_date) AS all_time_low,
  -- Drawdown : % de chute par rapport au max précédent
  ROUND(
    (close_price - MAX(close_price) OVER (ORDER BY price_date)) * 100.0
    / NULLIF(MAX(close_price) OVER (ORDER BY price_date), 0),
    2
  ) AS drawdown_pct
FROM stock_prices
WHERE ticker = 'AAPL'
ORDER BY price_date;

Frame clauses — ROWS vs RANGE et cadrage précis

⏱ 30 min Subtil et critique

La frame clause définit précisément quelles lignes entrent dans le calcul d'une window function pour chaque ligne courante. C'est la partie la moins comprise des window functions, et elle est responsable de nombreux bugs subtils.

Anatomie d'une frame clause

Structure complète
fonction() OVER (
  [PARTITION BY ...]
  [ORDER BY ...]
  {ROWS | RANGE | GROUPS}
  BETWEEN debut AND fin
)

-- debut et fin peuvent être :
--   UNBOUNDED PRECEDING  → début de la partition
--   N PRECEDING          → N lignes/valeurs avant la ligne courante
--   CURRENT ROW          → la ligne courante
--   N FOLLOWING          → N lignes/valeurs après la ligne courante
--   UNBOUNDED FOLLOWING  → fin de la partition

-- Frame par défaut quand ORDER BY est présent :
--   RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW

-- Frame par défaut sans ORDER BY :
--   ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING

ROWS vs RANGE — la différence cruciale

ROWS vs RANGE — comportement avec les ex-aequo
-- Table : orders(order_date, amount)
-- 2024-01-01 | 100
-- 2024-01-01 | 200   ← même date que ci-dessus
-- 2024-01-02 | 150

-- RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW (défaut)
-- "CURRENT ROW" en RANGE = toutes les lignes avec la MÊME valeur d'ORDER BY
-- Donc pour les deux lignes du 2024-01-01, la frame inclut LES DEUX
SELECT order_date, amount,
  SUM(amount) OVER (ORDER BY order_date RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
FROM orders;
-- 2024-01-01 | 100 → SUM = 300 (les deux lignes du 01/01 sont dans la frame)
-- 2024-01-01 | 200 → SUM = 300 (idem)
-- 2024-01-02 | 150 → SUM = 450

-- ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
-- "CURRENT ROW" en ROWS = exactement cette ligne physique
SELECT order_date, amount,
  SUM(amount) OVER (ORDER BY order_date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
FROM orders;
-- 2024-01-01 | 100 → SUM = 100 (uniquement la 1ère ligne)
-- 2024-01-01 | 200 → SUM = 300 (2 premières lignes physiques)
-- 2024-01-02 | 150 → SUM = 450
RANGE vs ROWS — quelle différence en pratique ? Pour les séries temporelles avec des données déjà agrégées par jour (une seule ligne par date), les deux sont équivalents. La différence apparaît quand plusieurs lignes partagent la même valeur ORDER BY. Utilisez ROWS pour les fenêtres glissantes sur des données à granularité fine (événements) — c'est plus prévisible et souvent plus rapide.

Fenêtres glissantes précises

Frames concrètes pour l'analytique
SELECT
  sale_date,
  revenue,

  -- Toute la partition (total global)
  SUM(revenue) OVER () AS total,

  -- Cumulatif depuis le début
  SUM(revenue) OVER (ORDER BY sale_date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS cumul,

  -- Derniers 6 jours + aujourd'hui = fenêtre glissante de 7 jours
  AVG(revenue) OVER (ORDER BY sale_date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS ma_7d,

  -- Centré sur le jour courant : 3 avant + courant + 3 après = 7 jours
  AVG(revenue) OVER (ORDER BY sale_date ROWS BETWEEN 3 PRECEDING AND 3 FOLLOWING) AS centered_ma_7d,

  -- Toute la partition (équivalent à pas de ORDER BY)
  AVG(revenue) OVER (PARTITION BY DATE_TRUNC('month', sale_date)
                     ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING) AS month_avg
FROM daily_sales;

GROUPS — le troisième mode (SQL:2011, PostgreSQL 11+)

GROUPS — compter par groupes de pairs
-- GROUPS : N PRECEDING = N groupes de valeurs égales précédents
-- Moins utilisé mais utile pour les données avec des ties

SELECT score,
  SUM(score) OVER (
    ORDER BY score
    GROUPS BETWEEN 1 PRECEDING AND 1 FOLLOWING
    -- Inclut le groupe précédent, courant et suivant (groupes de valeurs égales)
  ) AS windowed_sum
FROM scores;

EXCLUDE — exclure des lignes de la frame (PostgreSQL 11+)

EXCLUDE — exclure la ligne courante ou ses pairs
-- Calculer la moyenne des voisins SANS la ligne courante
SELECT name, score,
  AVG(score) OVER (
    ORDER BY score
    ROWS BETWEEN 2 PRECEDING AND 2 FOLLOWING
    EXCLUDE CURRENT ROW   -- exclure la ligne courante de la moyenne
  ) AS neighbor_avg
FROM scores;

-- Autres options EXCLUDE :
-- EXCLUDE NO OTHERS   → défaut, n'exclut rien
-- EXCLUDE CURRENT ROW → exclut la ligne courante
-- EXCLUDE GROUP       → exclut tous les pairs (même valeur ORDER BY)
-- EXCLUDE TIES        → exclut les autres pairs mais garde la ligne courante

Lire un plan d'exécution — EXPLAIN ANALYZE

⏱ 30 min Diagnostic essentiel

EXPLAIN est l'outil de diagnostic le plus important en SQL. Il révèle comment le moteur prévoit d'exécuter votre requête : quelles tables sont scannées, quels index sont utilisés, comment les jointures sont réalisées, et où le temps est passé. Sans EXPLAIN, l'optimisation SQL est aveugle.

EXPLAIN vs EXPLAIN ANALYZE

Modes d'EXPLAIN
-- EXPLAIN : affiche le plan PRÉVU (coûts estimés, lignes estimées)
-- N'exécute PAS la requête
EXPLAIN SELECT * FROM orders WHERE customer_id = 42;

-- EXPLAIN ANALYZE : exécute la requête + compare estimations vs réalité
-- Attention : la requête est réellement exécutée !
EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_id = 42;

-- EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON) : sortie complète et détaillée
EXPLAIN (ANALYZE, BUFFERS, VERBOSE, FORMAT JSON)
SELECT * FROM orders o JOIN customers c ON o.customer_id = c.id WHERE c.country = 'FR';

-- Pour un SELECT dans une transaction (rollback pour éviter les side effects)
BEGIN;
EXPLAIN ANALYZE UPDATE orders SET status='processed' WHERE id=1;
ROLLBACK;

Lire la sortie EXPLAIN

Anatomie d'un plan d'exécution PostgreSQL
EXPLAIN ANALYZE
SELECT c.name, COUNT(o.id) FROM customers c JOIN orders o ON c.id = o.customer_id
WHERE c.country = 'FR' GROUP BY c.name;

-- Sortie :
-- HashAggregate  (cost=1250.00..1275.00 rows=500 width=40)
--                (actual time=45.2..47.3 rows=423 loops=1)
--   Group Key: c.name
--   ->  Hash Join  (cost=150.00..1200.00 rows=2000 width=32)
--                  (actual time=12.5..38.1 rows=1947 loops=1)
--         Hash Cond: (o.customer_id = c.id)
--         ->  Seq Scan on orders  (cost=0..800 rows=50000 width=8)
--                                 (actual time=0.1..18.3 rows=50000 loops=1)
--         ->  Hash  (cost=120..120 rows=2400 width=24)
--                   (actual time=11.2..11.2 rows=2385 loops=1)
--               ->  Seq Scan on customers  (cost=0..80 rows=2400 width=24)
--                   Filter: (country = 'FR')
--                   Rows Removed by Filter: 7615
-- Planning Time: 1.2 ms
-- Execution Time: 47.8 ms

-- Décryptage :
-- cost=X..Y : coût estimé (X=démarrage, Y=total) — unité arbitraire
-- rows=N    : nombre de lignes estimées
-- actual time=X..Y ms : temps réel (démarrage..fin)
-- actual rows=N : lignes réellement retournées
-- loops=N   : combien de fois ce nœud a été exécuté (Nested Loop)

Les nœuds clés à reconnaître

Types de scans et de jointures
-- TYPES DE SCAN (du meilleur au pire selon la sélectivité)
-- Index Only Scan : toutes les colonnes viennent de l'index (idéal)
-- Index Scan      : utilise l'index puis va chercher la ligne (bon)
-- Bitmap Heap Scan : lit plusieurs lignes en vrac via bitmap (moyen, bon pour 1-10%)
-- Seq Scan        : scan séquentiel complet (nécessaire pour gros filtres ou petites tables)

-- TYPES DE JOINTURES
-- Nested Loop  : pour chaque ligne de A, cherche dans B via index (O(N log M))
--               → bon pour petite table de gauche ou index sur la droite
-- Hash Join    : construit une table de hash de B, scan de A (O(N+M))
--               → bon pour jointures sans index, tailles comparables
-- Merge Join   : les deux tables déjà triées, scannées en parallèle (O(N+M))
--               → bon quand les deux côtés sont déjà ordonnés

-- ⚠ Signaux d'alarme :
-- Seq Scan sur grosse table avec peu de lignes retournées → index manquant
-- Nested Loop avec loops=100000 → peut-être un Hash Join serait mieux
-- Rows estimated=100 vs actual=50000 → mauvaises statistiques (ANALYZE)

Diagnostiquer avec les statistiques

Quand les estimations sont mauvaises
-- Si "rows estimated" diverge fortement de "actual rows"
-- → les statistiques sont périmées → ANALYZE la table
ANALYZE customers;       -- mettre à jour les statistiques d'une table
ANALYZE;                 -- toutes les tables

-- Vérifier les statistiques actuelles
SELECT tablename, n_live_tup, n_dead_tup, last_analyze, last_autoanalyze
FROM pg_stat_user_tables WHERE tablename = 'orders';

-- Augmenter la précision des statistiques pour les colonnes très sélectives
ALTER TABLE orders ALTER COLUMN status SET STATISTICS 500;
-- (défaut = 100, max = 10000)
ANALYZE orders;

-- Voir les statistiques d'une colonne
SELECT * FROM pg_stats WHERE tablename = 'orders' AND attname = 'status';
Méthodologie EXPLAIN Lisez le plan de bas en haut et de droite à gauche — les nœuds les plus profonds et indentés sont exécutés en premier. Cherchez le nœud qui prend le plus de temps (actual time élevé) et dont les estimations divergent le plus de la réalité — c'est là qu'est le problème.

Stratégies d'indexation

⏱ 30 min Performance

Un index est une structure de données auxiliaire qui permet au moteur de retrouver des lignes sans scanner toute la table. Bien indexer une base de données peut réduire des requêtes de plusieurs minutes à quelques millisecondes — mal indexer peut ralentir une base par la surcharge de maintenance.

B-tree — l'index universel

B-tree — types de requêtes supportées
-- Un index B-tree sur orders(customer_id) accélère :
CREATE INDEX idx_orders_customer ON orders(customer_id);

-- ✓ Égalité
WHERE customer_id = 42
-- ✓ Comparaisons
WHERE customer_id > 100
WHERE order_date BETWEEN '2024-01-01' AND '2024-12-31'
-- ✓ IS NULL (PostgreSQL)
WHERE customer_id IS NULL
-- ✓ Préfixe (LIKE)
WHERE name LIKE 'Ali%'   -- utilise l'index
WHERE name LIKE '%Ali%'  -- n'utilise PAS l'index (joker au début)

-- L'index n'est PAS utilisé si :
-- Fonction sur la colonne indexée :
WHERE LOWER(email) = 'alice@example.com'  -- ignore l'index sur email
WHERE EXTRACT(YEAR FROM order_date) = 2024  -- ignore l'index sur order_date

-- Solution : index d'expression
CREATE INDEX idx_email_lower ON users(LOWER(email));
CREATE INDEX idx_order_year ON orders(EXTRACT(YEAR FROM order_date));

Index composites — l'ordre des colonnes est critique

Index composite — règle du préfixe gauche
-- Index sur (status, customer_id, order_date)
CREATE INDEX idx_orders_compound ON orders(status, customer_id, order_date);

-- ✓ Utilise l'index (préfixe gauche respecté)
WHERE status = 'completed'
WHERE status = 'completed' AND customer_id = 42
WHERE status = 'completed' AND customer_id = 42 AND order_date > '2024-01-01'

-- ✗ N'utilise PAS l'index (manque le préfixe gauche)
WHERE customer_id = 42               -- pas le préfixe status
WHERE order_date > '2024-01-01'      -- pas le préfixe status+customer_id

-- Règle de design des index composites :
-- 1. Colonnes d'égalité en premier (= filtre le plus sélectif)
-- 2. Colonnes de range à la fin (>, <, BETWEEN)
-- 3. Colonnes ORDER BY après (pour éviter un tri séparé)

-- Exemple optimal pour : WHERE status='active' ORDER BY created_at DESC
CREATE INDEX idx_status_date ON users(status, created_at DESC);

Index partiels — indexer seulement ce qui est utile

Index partiels — plus petits et plus rapides
-- Si 95% des requêtes portent sur status='active', indexer uniquement ça
CREATE INDEX idx_active_users ON users(email)
WHERE status = 'active';

-- L'index est utilisé UNIQUEMENT si la requête a WHERE status='active'
SELECT * FROM users WHERE status = 'active' AND email = 'alice@example.com';
-- → Index partiel utilisé (très petit, très rapide)

SELECT * FROM users WHERE email = 'alice@example.com';
-- → Index complet nécessaire (non couvert par l'index partiel)

-- Cas d'usage courants pour les index partiels :
-- ① Données non-NULL (ignorer les NULLs)
CREATE INDEX idx_orders_discount ON orders(discount) WHERE discount IS NOT NULL;
-- ② Statuts actifs seulement
CREATE INDEX idx_sessions_active ON sessions(user_id) WHERE expires_at > NOW();
-- ③ Données récentes
CREATE INDEX idx_recent ON events(user_id, created_at) WHERE created_at > '2024-01-01';

Index couvrants — Index Only Scan

Covering index — toutes les colonnes dans l'index
-- Si toutes les colonnes SELECT sont dans l'index → Index Only Scan
-- Le moteur n'a pas besoin de lire la table du tout !

-- Requête : SELECT name FROM customers WHERE country = 'FR'
CREATE INDEX idx_customers_covering ON customers(country, name);
-- → Index Only Scan : country (filtre) + name (SELECT) dans l'index

-- PostgreSQL : clause INCLUDE pour ajouter des colonnes non-filtrantes
CREATE INDEX idx_orders_include ON orders(customer_id) INCLUDE (amount, order_date);
-- customer_id → filtre, amount + order_date → dans l'index mais pas de tri dessus
-- SELECT amount, order_date FROM orders WHERE customer_id = 42 → Index Only Scan

Quand NE PAS indexer

Anti-patterns d'indexation
-- ① Petites tables (< quelques milliers de lignes) : Seq Scan est plus rapide
-- ② Très faible cardinalité : boolean, status (2-3 valeurs)
--    SELECT * FROM users WHERE active = true → 80% des lignes → Seq Scan préféré
--    Exception : index partiel sur active=true si 1% des lignes seulement

-- ③ Tables très write-heavy : chaque INSERT/UPDATE/DELETE met à jour tous les index
--    100 indexes sur une table = 100× plus lent pour l'écriture

-- ④ Colonnes rarement dans les requêtes : aucun bénéfice, surcoût en maintenance

-- Surveiller les index inutilisés
SELECT schemaname, tablename, indexname, idx_scan, idx_tup_read
FROM pg_stat_user_indexes
WHERE idx_scan = 0    -- jamais utilisé
ORDER BY schemaname, tablename;

-- Surveiller les index bloated (table fragmentée après beaucoup de DELETEs)
SELECT tablename, indexname, pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
ORDER BY pg_relation_size(indexrelid) DESC;

Optimiser jointures et window functions

⏱ 25 min Performance avancée

Les jointures et les window functions sont les opérations les plus coûteuses dans les requêtes analytiques. Quelques techniques bien ciblées peuvent réduire drastiquement leur impact.

Optimiser les jointures

Prédicats pushdown — filtrer tôt
-- ✗ Mauvais : jointure sur toutes les lignes, puis filtre
SELECT c.name, o.amount
FROM customers c
JOIN orders o ON c.id = o.customer_id
WHERE o.order_date >= '2024-01-01'  -- filtre APRÈS le join complet
  AND c.country = 'FR';

-- ✓ Mieux : pré-filtrer dans des CTEs avant le join (aide l'optimiseur)
WITH fr_customers AS (
  SELECT id, name FROM customers WHERE country = 'FR'
),
recent_orders AS (
  SELECT customer_id, amount FROM orders WHERE order_date >= '2024-01-01'
)
SELECT fc.name, ro.amount
FROM fr_customers fc
JOIN recent_orders ro ON fc.id = ro.customer_id;

-- Note : les moteurs modernes font souvent le pushdown automatiquement
-- Vérifiez avec EXPLAIN avant d'optimiser manuellement
Joindre sur des colonnes indexées
-- La condition ON doit porter sur des colonnes indexées pour un bon plan
-- ✓ Bon : jointure sur des clés primaires/étrangères indexées
SELECT c.name, o.amount
FROM customers c    -- c.id = PRIMARY KEY (indexé automatiquement)
JOIN orders o ON c.id = o.customer_id;  -- doit y avoir un index sur o.customer_id

-- Si l'index sur o.customer_id manque → Seq Scan sur orders → Hash Join lent
-- Solution :
CREATE INDEX ON orders(customer_id);

-- ✗ Mauvais : jointure sur une expression non indexée
JOIN orders o ON CAST(c.id AS TEXT) = o.customer_id_text;
-- L'index sur customer_id_text ne sera pas utilisé avec le CAST côté gauche
-- Solution : normaliser les types à la conception ou créer un index d'expression

Éviter le fan-out avant l'agrégation

Agréger avant de joindre
-- ✗ Fan-out : 50M lignes joints, puis agrégées
SELECT c.country, SUM(o.amount) AS total
FROM customers c JOIN orders o ON c.id = o.customer_id
GROUP BY c.country;

-- ✓ Agréger d'abord (orders : 50M → 1M par customer), puis joindre
SELECT c.country, agg.total
FROM customers c
JOIN (
  SELECT customer_id, SUM(amount) AS total
  FROM orders GROUP BY customer_id
) agg ON c.id = agg.customer_id
GROUP BY c.country;

Optimiser les window functions

Réutiliser les fenêtres identiques
-- Chaque OVER() distinct peut nécessiter un tri ou un hash séparé
-- Réutiliser la même définition de fenêtre via la clause WINDOW

-- ✗ Mauvais : deux tris identiques
SELECT *,
  LAG(amount) OVER (PARTITION BY dept ORDER BY date) AS prev,
  SUM(amount) OVER (PARTITION BY dept ORDER BY date) AS running_sum
FROM sales;
-- Si l'optimiseur reconnaît les OVER() identiques → un seul tri
-- Mais c'est fragile et non garanti

-- ✓ Explicite avec WINDOW (garantit un seul tri)
SELECT *,
  LAG(amount) OVER w   AS prev,
  SUM(amount) OVER w   AS running_sum
FROM sales
WINDOW w AS (PARTITION BY dept ORDER BY date);
Limiter le nombre de partitions scannées
-- Filtrer AVANT d'appliquer les window functions (dans une CTE)
-- pour réduire le volume sur lequel la fenêtre opère

-- ✗ Window function sur toutes les données, puis filtre
SELECT * FROM (
  SELECT *,
    ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) AS rn
  FROM employees  -- 1 million d'employés
) t WHERE rn = 1 AND dept = 'Engineering';

-- ✓ Filtrer d'abord, window function sur le sous-ensemble
SELECT * FROM (
  SELECT *,
    ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) AS rn
  FROM employees
  WHERE dept = 'Engineering'  -- filtrer dans la sous-requête
) t WHERE rn = 1;
Window functions et parallélisme Les window functions avec PARTITION BY peuvent bénéficier du parallélisme (chaque partition traitée par un worker différent). Sans PARTITION BY, la fenêtre est un goulot d'étranglement séquentiel. Sur des moteurs comme Snowflake ou BigQuery, les window functions sont distribuées sur le cluster — la PARTITION BY est clé pour l'efficacité.

Bonnes pratiques data analytiques

⏱ 25 min Anti-patterns

L'optimisation de performance est importante, mais la qualité et la maintenabilité du code SQL le sont tout autant. Cette leçon couvre les anti-patterns les plus fréquents, les bonnes pratiques spécifiques aux entrepôts de données, et les pièges qui font planter les requêtes en production.

Anti-patterns SQL classiques

Anti-patterns à éviter
-- ① SELECT * en production : charge inutile, colonne BYTEA/TEXT incluses
SELECT * FROM users;  -- peut inclure une colonne avatar_data BYTEA de 1MB

-- ✓ Sélectionner uniquement ce dont on a besoin
SELECT id, name, email FROM users;

-- ② DISTINCT comme "cache-misère" de doublons
SELECT DISTINCT customer_id FROM orders JOIN shipments ON ...;
-- DISTINCT signale souvent une jointure qui multiplie les lignes
-- Interrogez-vous : pourquoi y a-t-il des doublons ?

-- ③ Correlated subquery dans SELECT sur gros volume
SELECT name, (SELECT COUNT(*) FROM orders WHERE customer_id = u.id) AS nb_orders
FROM users;  -- N requêtes pour N utilisateurs

-- ✓ Utiliser une window function ou une jointure avec agrégation
SELECT u.name, COALESCE(o.nb_orders, 0) AS nb_orders
FROM users u LEFT JOIN (
  SELECT customer_id, COUNT(*) AS nb_orders FROM orders GROUP BY 1
) o ON u.id = o.customer_id;
Plus d'anti-patterns
-- ④ ORDER BY dans une sous-requête (sans LIMIT) : inutile et coûteux
SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) AS rn
  FROM (SELECT * FROM employees ORDER BY name) sorted  -- ORDER BY inutile ici !
) t WHERE rn = 1;

-- ⑤ Window function dans WHERE (doit aller dans CTE)
-- (voir L6.1)

-- ⑥ NOT IN avec sous-requête pouvant retourner NULL (voir L1.2, L2.3)

-- ⑦ Comparaison de timestamps sans gestion du timezone
WHERE created_at > '2024-01-01'  -- ambigu selon le TZ de la session
-- ✓ Explicit
WHERE created_at > '2024-01-01 00:00:00+00'::TIMESTAMPTZ

Spécificités des entrepôts de données (Data Warehouses)

Snowflake, BigQuery, Redshift — bonnes pratiques
-- Column-store : ces bases stockent par colonne, pas par ligne
-- → SELECT * est encore plus pénalisant (force la lecture de toutes les colonnes)
-- → Sélectionner uniquement les colonnes nécessaires est crucial

-- ① Partition pruning : filtrer sur la colonne de partitionnement
-- BigQuery : tables partitionnées par date
SELECT * FROM `project.dataset.events`
WHERE DATE(_PARTITIONTIME) = '2024-01-15';  -- prune les autres partitions

-- ② Clustering keys (Snowflake) : l'ordre physique des données
-- Si la table est clustered by (date, region), les requêtes sur (date) ou
-- (date, region) bénéficieront du micro-partitionnement

-- ③ APPROX fonctions pour les grands volumes
SELECT
  APPROX_COUNT_DISTINCT(user_id) AS approx_users,  -- BigQuery
  -- vs COUNT(DISTINCT user_id) qui nécessite un tri complet
  APPROX_QUANTILES(amount, 100)[OFFSET(50)] AS approx_median  -- BigQuery
FROM events;

-- ④ Éviter les JOINs croisés entre grandes tables
-- Préférer : broadcaster la petite table, partitionner la grande

EXISTS vs COUNT > 0

Tester l'existence — la bonne approche
-- ✗ Mauvais : compte toutes les lignes pour vérifier si > 0
IF (SELECT COUNT(*) FROM orders WHERE customer_id = 42) > 0 THEN ...

-- ✓ Bon : EXISTS court-circuite au premier match
IF EXISTS (SELECT 1 FROM orders WHERE customer_id = 42) THEN ...

-- En SQL pur (pas de IF) :
-- ✗ Lent sur gros volumes
SELECT CASE WHEN COUNT(*) > 0 THEN 'has orders' ELSE 'no orders' END
FROM orders WHERE customer_id = 42;

-- ✓ Rapide
SELECT CASE WHEN EXISTS (
  SELECT 1 FROM orders WHERE customer_id = 42
) THEN 'has orders' ELSE 'no orders' END;

Bonnes pratiques de style et maintenabilité

SQL lisible et maintenable
-- ① Nommer toutes les sous-requêtes et CTE de façon descriptive
-- ✗ Bad
WITH t1 AS (SELECT ...), t2 AS (SELECT ...)
-- ✓ Good
WITH filtered_orders AS (SELECT ...), customer_totals AS (SELECT ...)

-- ② Aligner les clauses SQL pour la lisibilité
SELECT
  customer_id,
  COUNT(*)       AS order_count,
  SUM(amount)    AS total_spent
FROM orders
WHERE status = 'completed'
  AND order_date >= '2024-01-01'
GROUP BY customer_id
HAVING COUNT(*) > 5
ORDER BY total_spent DESC
LIMIT 100;

-- ③ Éviter les magic numbers — utiliser des variables ou des CTEs
WITH constants AS (SELECT 0.15 AS tax_rate, 10000 AS free_shipping_threshold)
SELECT amount, amount * (1 + c.tax_rate) AS with_tax,
       amount >= c.free_shipping_threshold AS qualifies_free_shipping
FROM orders CROSS JOIN constants c;

-- ④ Commenter les choix non évidents (pas les évidences)
-- ✗ Commentaire inutile : -- filtrer par date
WHERE order_date >= '2024-01-01'

-- ✓ Commentaire utile : expliquer le WHY

WHERE order_date >= '2024-01-04'

Analyse de cohortes

⏱ 35 min Analytics avancé

L'analyse de cohortes groupe les utilisateurs selon leur date d'acquisition et suit l'évolution de leur comportement dans le temps. C'est l'une des analyses les plus révélatrices pour comprendre la qualité des utilisateurs au fil des générations de campagnes marketing.

Définir une cohorte

Étape 1 — identifier la cohorte de chaque utilisateur
-- Cohorte = mois du premier achat de chaque utilisateur
-- Table : purchases(user_id, purchase_date, amount)

-- Étape 1a : trouver la date de première activité par utilisateur
WITH user_cohorts AS (
  SELECT
    user_id,
    DATE_TRUNC('month', MIN(purchase_date)) AS cohort_month
  FROM purchases
  GROUP BY user_id
),

-- Étape 1b : associer chaque achat à la cohorte de son utilisateur
purchases_with_cohort AS (
  SELECT
    p.user_id,
    p.purchase_date,
    p.amount,
    uc.cohort_month,
    -- Nombre de mois depuis l'entrée dans la cohorte (period_number)
    DATE_PART('month', AGE(
      DATE_TRUNC('month', p.purchase_date),
      uc.cohort_month
    )) AS period_number
  FROM purchases p
  JOIN user_cohorts uc ON p.user_id = uc.user_id
),

-- Étape 2 : compter les utilisateurs actifs par (cohorte, période)
cohort_activity AS (
  SELECT
    cohort_month,
    period_number,
    COUNT(DISTINCT user_id) AS active_users,
    SUM(amount) AS revenue
  FROM purchases_with_cohort
  GROUP BY cohort_month, period_number
),

-- Étape 3 : calculer la taille initiale de chaque cohorte
cohort_sizes AS (
  SELECT cohort_month, COUNT(DISTINCT user_id) AS cohort_size
  FROM user_cohorts
  GROUP BY cohort_month
)

-- Rapport final : taux de rétention par cohorte et période
SELECT
  ca.cohort_month,
  ca.period_number,
  ca.active_users,
  cs.cohort_size,
  ROUND(ca.active_users * 100.0 / cs.cohort_size, 1) AS retention_rate
FROM cohort_activity ca
JOIN cohort_sizes cs ON ca.cohort_month = cs.cohort_month
ORDER BY ca.cohort_month, ca.period_number;

Résultat en tableau croisé (pivot)

Pivot de la table de cohortes
-- Transformer les lignes en colonnes (M0, M1, M2, M3...)
WITH retention AS (
  -- ... (requête ci-dessus)
  SELECT cohort_month, period_number, retention_rate
  FROM cohort_activity ca JOIN cohort_sizes cs USING (cohort_month)
)
SELECT
  cohort_month,
  MAX(retention_rate) FILTER (WHERE period_number = 0) AS "M0",
  MAX(retention_rate) FILTER (WHERE period_number = 1) AS "M1",
  MAX(retention_rate) FILTER (WHERE period_number = 2) AS "M2",
  MAX(retention_rate) FILTER (WHERE period_number = 3) AS "M3",
  MAX(retention_rate) FILTER (WHERE period_number = 6) AS "M6",
  MAX(retention_rate) FILTER (WHERE period_number = 12) AS "M12"
FROM retention
GROUP BY cohort_month
ORDER BY cohort_month;

-- (NULL = données non encore disponibles)

Analyse de la valeur de cohorte (LTV)

Lifetime Value cumulée par cohorte
WITH cohort_revenue AS (
  SELECT
    cohort_month,
    period_number,
    SUM(revenue) / MAX(cohort_size) AS avg_revenue_per_user  -- revenu moyen par utilisateur initial
  FROM cohort_activity ca JOIN cohort_sizes cs USING (cohort_month)
  GROUP BY cohort_month, period_number
)
SELECT
  cohort_month,
  period_number,
  avg_revenue_per_user,
  -- LTV cumulée : revenu total accumulé depuis le début de la cohorte
  SUM(avg_revenue_per_user) OVER (
    PARTITION BY cohort_month
    ORDER BY period_number
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS cumulative_ltv
FROM cohort_revenue
ORDER BY cohort_month, period_number;

Calcul de rétention et churn

⏱ 30 min Métrique business

La rétention et le churn sont les indicateurs de santé fondamentaux d'un produit SaaS ou e-commerce. Savoir les calculer avec précision en SQL est une compétence différenciante pour un data analyst.

Rétention périodique (rolling retention)

Taux de rétention D7 — utilisateurs actifs 7 jours après inscription
-- Definition : rétention D7 = % d'utilisateurs qui ont eu une activité
-- entre J+6 et J+8 (fenêtre de ±1 jour autour de J+7)

WITH user_first_activity AS (
  SELECT user_id, MIN(activity_date) AS first_date
  FROM user_activities GROUP BY user_id
)
SELECT
  DATE_TRUNC('week', f.first_date) AS signup_week,
  COUNT(DISTINCT f.user_id) AS cohort_size,
  COUNT(DISTINCT a.user_id) AS retained_d7,
  ROUND(COUNT(DISTINCT a.user_id) * 100.0 / COUNT(DISTINCT f.user_id), 1) AS d7_retention
FROM user_first_activity f
LEFT JOIN user_activities a
  ON f.user_id = a.user_id
  AND a.activity_date BETWEEN f.first_date + 6 AND f.first_date + 8  -- inéquijoin !
GROUP BY DATE_TRUNC('week', f.first_date)
ORDER BY signup_week;

Rétention mensuelle avec LAG

Rétention mois par mois — approche LAG
-- Utilisateurs actifs chaque mois, comparés au mois précédent
WITH monthly_active AS (
  SELECT
    DATE_TRUNC('month', activity_date) AS activity_month,
    user_id
  FROM user_activities
  GROUP BY 1, 2
),
monthly_counts AS (
  SELECT
    activity_month,
    COUNT(DISTINCT user_id) AS mau,  -- Monthly Active Users
    -- Retenu = actif ce mois ET le mois précédent
    COUNT(DISTINCT CASE WHEN EXISTS (
      SELECT 1 FROM monthly_active prev
      WHERE prev.user_id = ma.user_id
        AND prev.activity_month = ma.activity_month - INTERVAL '1 month'
    ) THEN user_id END) AS retained_users
  FROM monthly_active ma
  GROUP BY activity_month
)
SELECT
  activity_month,
  mau,
  retained_users,
  ROUND(retained_users * 100.0 / LAG(mau) OVER (ORDER BY activity_month), 1) AS retention_rate,
  mau - retained_users AS new_or_reactivated,
  LAG(mau) OVER (ORDER BY activity_month) - retained_users AS churned
FROM monthly_counts
ORDER BY activity_month;

Calcul du churn explicite

Identifier les utilisateurs churned — ANTI JOIN
-- Churned = actif le mois précédent mais PAS ce mois-ci
WITH monthly_active AS (
  SELECT DISTINCT DATE_TRUNC('month', activity_date) AS month, user_id
  FROM user_activities
)
SELECT
  curr_month.month AS churn_month,
  COUNT(prev.user_id) AS churned_users
FROM (SELECT DISTINCT month FROM monthly_active) curr_month
CROSS JOIN (SELECT DISTINCT month FROM monthly_active) prev_month
JOIN monthly_active prev ON prev.month = curr_month.month - INTERVAL '1 month'
WHERE NOT EXISTS (
  SELECT 1 FROM monthly_active curr
  WHERE curr.user_id = prev.user_id AND curr.month = curr_month.month
)
GROUP BY curr_month.month
ORDER BY churn_month;

-- Churn rate = churned / mau_prev_month
WITH churn_data AS (
  -- (requête ci-dessus)
  SELECT churn_month, churned_users, mau_prev
  FROM ...
)
SELECT
  churn_month,
  churned_users,
  ROUND(churned_users * 100.0 / mau_prev, 1) AS churn_rate_pct,
  100 - ROUND(churned_users * 100.0 / mau_prev, 1) AS retention_rate_pct
FROM churn_data;

Funnel analysis

⏱ 30 min Conversion

Un funnel (entonnoir) est une séquence d'étapes que les utilisateurs traversent vers un objectif (achat, inscription, activation). L'analyse de funnel mesure les taux de conversion à chaque étape et identifie où les utilisateurs abandonnent.

Funnel simple — comptage par étape

Étape 1 — comptage brut par événement
-- Table events(user_id, event_type, occurred_at)
-- Funnel : view_product → add_to_cart → checkout → purchase

WITH funnel_steps AS (
  SELECT
    COUNT(DISTINCT CASE WHEN event_type = 'view_product' THEN user_id END)  AS step1_view,
    COUNT(DISTINCT CASE WHEN event_type = 'add_to_cart'  THEN user_id END)  AS step2_cart,
    COUNT(DISTINCT CASE WHEN event_type = 'checkout'     THEN user_id END)  AS step3_checkout,
    COUNT(DISTINCT CASE WHEN event_type = 'purchase'     THEN user_id END)  AS step4_purchase
  FROM events
  WHERE occurred_at BETWEEN '2024-01-01' AND '2024-01-31'
)
SELECT
  step1_view     AS "1. Vue produit",
  step2_cart     AS "2. Ajout panier",
  step3_checkout AS "3. Checkout",
  step4_purchase AS "4. Achat",
  ROUND(step2_cart     * 100.0 / NULLIF(step1_view, 0), 1) AS "Conv 1→2 %",
  ROUND(step3_checkout * 100.0 / NULLIF(step2_cart, 0), 1) AS "Conv 2→3 %",
  ROUND(step4_purchase * 100.0 / NULLIF(step3_checkout, 0), 1) AS "Conv 3→4 %",
  ROUND(step4_purchase * 100.0 / NULLIF(step1_view, 0), 1) AS "Conv totale %"
FROM funnel_steps;

Funnel ordonné — séquence temporelle obligatoire

Un funnel simple compte les utilisateurs qui ont eu chaque événement, peu importe l'ordre. Un funnel ordonné exige que les étapes se déroulent dans l'ordre, et souvent dans une fenêtre de temps limitée.

Funnel ordonné avec fenêtre temporelle
WITH user_events AS (
  SELECT user_id, event_type, occurred_at
  FROM events
  WHERE event_type IN ('view_product','add_to_cart','checkout','purchase')
    AND occurred_at >= '2024-01-01'
),
step1_users AS (
  SELECT user_id, MIN(occurred_at) AS step1_time
  FROM user_events WHERE event_type = 'view_product'
  GROUP BY user_id
),
step2_users AS (
  SELECT s1.user_id, MIN(e.occurred_at) AS step2_time
  FROM step1_users s1
  JOIN user_events e ON s1.user_id = e.user_id
    AND e.event_type = 'add_to_cart'
    AND e.occurred_at > s1.step1_time                     -- après l'étape 1
    AND e.occurred_at < s1.step1_time + INTERVAL '24h'   -- dans les 24h
  GROUP BY s1.user_id
),
step3_users AS (
  SELECT s2.user_id, MIN(e.occurred_at) AS step3_time
  FROM step2_users s2
  JOIN user_events e ON s2.user_id = e.user_id
    AND e.event_type = 'checkout'
    AND e.occurred_at > s2.step2_time
    AND e.occurred_at < s2.step2_time + INTERVAL '2h'
  GROUP BY s2.user_id
),
step4_users AS (
  SELECT s3.user_id
  FROM step3_users s3
  JOIN user_events e ON s3.user_id = e.user_id
    AND e.event_type = 'purchase'
    AND e.occurred_at > s3.step3_time
    AND e.occurred_at < s3.step3_time + INTERVAL '30m'
  GROUP BY s3.user_id
)
SELECT
  (SELECT COUNT(*) FROM step1_users) AS step1,
  (SELECT COUNT(*) FROM step2_users) AS step2,
  (SELECT COUNT(*) FROM step3_users) AS step3,
  (SELECT COUNT(*) FROM step4_users) AS step4;

Funnel avec LEAD — approche par séquence d'événements

Détecter l'étape suivante avec LEAD
-- Pour chaque événement, regarder si l'étape suivante est complétée
WITH event_sequences AS (
  SELECT
    user_id,
    event_type,
    occurred_at,
    LEAD(event_type) OVER (PARTITION BY user_id ORDER BY occurred_at) AS next_event,
    LEAD(occurred_at) OVER (PARTITION BY user_id ORDER BY occurred_at) AS next_event_time
  FROM events
  WHERE event_type IN ('view_product','add_to_cart','checkout','purchase')
)
SELECT
  event_type AS current_step,
  next_event AS next_step,
  COUNT(*) AS transitions,
  ROUND(COUNT(*) * 100.0 / SUM(COUNT(*)) OVER (PARTITION BY event_type), 1) AS pct_who_continue
FROM event_sequences
WHERE event_type IS NOT NULL
GROUP BY event_type, next_event
ORDER BY event_type, transitions DESC;
-- Montre les chemins réels : après add_to_cart, 40% font checkout, 35% rebrowsent, 25% abandonnent

Déduplication, Top-N et patterns courants

⏱ 30 min Patterns SQL

Cette leçon synthétise les patterns SQL les plus demandés en entretien data et en production : déduplication, top-N par groupe, détection de séquences (gap-and-island), et pivot/unpivot.

Déduplication avec ROW_NUMBER

Garder la ligne la plus récente par entité
-- Pattern universel de déduplication
WITH deduped AS (
  SELECT *,
    ROW_NUMBER() OVER (
      PARTITION BY entity_id         -- une clé par entité à dédupliquer
      ORDER BY updated_at DESC       -- critère de sélection (le plus récent)
    ) AS rn
  FROM records_with_duplicates
)
SELECT * FROM deduped WHERE rn = 1;

-- PostgreSQL uniquement — DISTINCT ON (plus concis)
SELECT DISTINCT ON (entity_id) *
FROM records_with_duplicates
ORDER BY entity_id, updated_at DESC;

-- Déduplication dans un INSERT (éviter les doublons à l'insertion)
INSERT INTO users (id, email, name)
SELECT DISTINCT ON (email) id, email, name
FROM staging_users
ORDER BY email, created_at DESC
ON CONFLICT (email) DO NOTHING;

Top-N par groupe

Top-3 produits par catégorie
-- Méthode 1 : ROW_NUMBER() (le plus universel)
WITH ranked AS (
  SELECT
    category,
    product_name,
    total_sales,
    ROW_NUMBER() OVER (PARTITION BY category ORDER BY total_sales DESC) AS rn
  FROM product_sales
)
SELECT * FROM ranked WHERE rn <= 3;

-- Méthode 2 : LATERAL JOIN (lisible, performant avec index)
SELECT cat.name AS category, top3.*
FROM categories cat
CROSS JOIN LATERAL (
  SELECT product_name, total_sales
  FROM product_sales
  WHERE category = cat.name
  ORDER BY total_sales DESC
  LIMIT 3
) top3;

-- Méthode 3 : sous-requête corrélée (lent, à éviter sur gros volumes)
SELECT * FROM product_sales ps
WHERE (
  SELECT COUNT(*) FROM product_sales ps2
  WHERE ps2.category = ps.category AND ps2.total_sales > ps.total_sales
) < 3;

Gap-and-Island — détecter des séquences consécutives

Le problème "gap-and-island" consiste à identifier des îlots (islands) de lignes consécutives satisfaisant une condition, séparés par des trous (gaps).

Trouver des séquences de jours consécutifs actifs
-- user_activity(user_id, activity_date)
-- Trouver les périodes de jours consécutifs actifs par utilisateur

WITH numbered AS (
  SELECT
    user_id,
    activity_date,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY activity_date) AS rn
  FROM user_activity
),
islands AS (
  SELECT
    user_id,
    activity_date,
    -- activity_date - rn jours = constante pour toutes les lignes d'un même îlot !
    activity_date - rn * INTERVAL '1 day' AS island_id
  FROM numbered
)
SELECT
  user_id,
  island_id,
  MIN(activity_date) AS streak_start,
  MAX(activity_date) AS streak_end,
  COUNT(*) AS streak_length
FROM islands
GROUP BY user_id, island_id
ORDER BY user_id, streak_start;

-- Interpréter : si l'utilisateur est actif les jours 1, 2, 3, 5, 6
-- Jours 1,2,3 : island_id = même valeur → un îlot de longueur 3
-- Jours 5,6 : island_id = valeur différente → un îlot de longueur 2

Trouver les gaps — séquences d'IDs manquants

Détecter les IDs manquants dans une séquence
-- Trouver les numéros de commandes manquants (séquence incomplète)
SELECT
  id + 1 AS gap_start,
  next_id - 1 AS gap_end,
  next_id - id - 1 AS gap_size
FROM (
  SELECT
    id,
    LEAD(id) OVER (ORDER BY id) AS next_id
  FROM orders
) t
WHERE next_id > id + 1;  -- il y a un saut dans la séquence

-- Vérifier les entiers manquants entre 1 et MAX
WITH expected AS (SELECT generate_series(1, MAX(id)) AS id FROM orders)
SELECT e.id AS missing_id
FROM expected e
LEFT JOIN orders o ON e.id = o.id
WHERE o.id IS NULL;

Pivot et unpivot

Unpivot — colonnes vers lignes
-- Source : table pivotée (une colonne par mois)
-- jan_rev | feb_rev | mar_rev
-- 1000    | 1200    | 950

-- UNPIVOT : transformer en format long (une ligne par mois)
-- PostgreSQL / standard SQL : CROSS JOIN LATERAL + VALUES
SELECT
  year,
  month_name,
  revenue
FROM yearly_summary
CROSS JOIN LATERAL (VALUES
  ('January',  jan_rev),
  ('February', feb_rev),
  ('March',    mar_rev),
  ('April',    apr_rev)
) AS unpivoted(month_name, revenue)
WHERE revenue IS NOT NULL;

-- SQL Server : UNPIVOT syntaxe native
SELECT year, month_name, revenue
FROM yearly_summary
UNPIVOT (revenue FOR month_name IN (jan_rev, feb_rev, mar_rev)) AS up;

-- Pivot : lignes vers colonnes (voir L5.3 — agrégations conditionnelles)
Récapitulatif des patterns clés
  • Dédup : ROW_NUMBER() OVER (PARTITION BY key ORDER BY date DESC) + WHERE rn=1
  • Top-N : idem mais WHERE rn <= N (ou DENSE_RANK pour les ex-aequo)
  • Running total : SUM() OVER (ORDER BY date ROWS UNBOUNDED PRECEDING AND CURRENT ROW)
  • MoM change : LAG(value) OVER (ORDER BY month)
  • Gap-island : date - ROW_NUMBER() OVER (ORDER BY date) = island_constant
  • Anti-join : NOT EXISTS (SELECT 1 FROM B WHERE B.key = A.key)
