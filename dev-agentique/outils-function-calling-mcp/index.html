Le concept fondamental du function calling

Pourquoi les LLMs ont besoin d'outils

Un grand modèle de langage est, par nature, stateless et coupé du monde réel. Il a été entraîné sur des données figées à une date de coupure, ne peut pas exécuter de code, ne connaît pas l'heure qu'il est, et n'a aucun accès à vos bases de données ou APIs privées. Pour dépasser ces limites, le function calling (ou tool use) a été introduit comme mécanisme standard.

L'idée est élégante : au lieu de demander au modèle de répondre à une question qu'il ne peut pas résoudre seul, on lui donne un catalogue d'outils qu'il peut demander à exécuter. Le modèle reste le cerveau décisionnel — il décide quand appeler quel outil et avec quels arguments — mais c'est votre code qui exécute réellement l'action.

Ce que le modèle fait vraiment

Il faut dissiper un malentendu fréquent : le modèle n'exécute pas les fonctions. Il génère une intention d'appel structurée en JSON. C'est votre application qui lit ce JSON, appelle la vraie fonction, puis renvoie le résultat au modèle pour qu'il continue son raisonnement.

Function calling vs RAG vs fine-tuning

Ces trois techniques sont complémentaires mais répondent à des besoins différents :

TechniqueUsage idéalLimite
Function callingActions en temps réel, données dynamiques, exécution de codeLatence aller-retour par outil
RAGAccès à une grande base de connaissance statiquePas d'actions, lecture seule
Fine-tuningStyle, format, domaine de niche bien définiCoûteux, ne donne pas accès aux données réelles

Cas d'usage réels

  • Assistant météo : le modèle appelle get_weather(city) pour avoir la météo actuelle
  • Agent financier : appelle search_stock(ticker) puis get_balance(account_id)
  • Automatisation : create_ticket(title, priority), send_email(to, body)
  • Code exécuté : run_python(code) — le modèle écrit du code, l'outil l'exécute dans un sandbox
⚠ Piège courant

Beaucoup pensent que le modèle "sait" appeler les fonctions parce qu'elles existent dans votre code. En réalité, il ne voit que les descriptions JSON que vous lui fournissez dans le prompt. Si la description est vague ou absente, le modèle ne saura pas quand ni comment appeler l'outil correctement.

Anatomie d'un cycle request/response

Le flow en 5 étapes

Un appel d'outil n'est pas une requête unique — c'est un dialogue en plusieurs tours. Comprendre chaque étape est essentiel pour déboguer et pour concevoir des systèmes robustes.

Étape 1 — La requête initiale

Vous envoyez au modèle : le message utilisateur + la liste des outils disponibles (sous forme de JSON Schema). Le modèle analyse le message et décide si un outil est nécessaire.

Python
import anthropic

client = anthropic.Anthropic()

# Étape 1 : envoi avec la définition d'outil
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=[{
        "name": "get_weather",
        "description": "Retourne la météo actuelle pour une ville donnée.",
        "input_schema": {
            "type": "object",
            "properties": {
                "city": {
                    "type": "string",
                    "description": "Nom de la ville (ex: 'Paris', 'Tokyo')"
                },
                "unit": {
                    "type": "string",
                    "enum": ["celsius", "fahrenheit"],
                    "description": "Unité de température"
                }
            },
            "required": ["city"]
        }
    }],
    messages=[{
        "role": "user",
        "content": "Quel temps fait-il à Lyon en ce moment ?"
    }]
)

Étape 2 — La réponse tool_use

Si le modèle décide d'appeler un outil, sa réponse a stop_reason = "tool_use" et le contenu contient un bloc tool_use avec le nom et les arguments :

JSON — Réponse du modèle
{
  "stop_reason": "tool_use",
  "content": [
    {
      "type": "text",
      "text": "Je vais vérifier la météo à Lyon."
    },
    {
      "type": "tool_use",
      "id": "toolu_01XnFJ2YBKX7gHzQvMPm3Rqs",
      "name": "get_weather",
      "input": {
        "city": "Lyon",
        "unit": "celsius"
      }
    }
  ]
}

Étape 3 — Exécution côté client

Votre code intercepte ce bloc, extrait les arguments, et appelle la vraie fonction :

Python
def get_weather(city: str, unit: str = "celsius") -> dict:
    # Appel réel à une API météo (ex: OpenWeatherMap)
    # Ici on simule
    return {
        "city": city,
        "temperature": 18,
        "unit": unit,
        "condition": "Nuageux avec éclaircies",
        "humidity": 72
    }

# Trouver le bloc tool_use
tool_use_block = next(
    block for block in response.content
    if block.type == "tool_use"
)

# Exécuter la fonction avec les arguments du modèle
result = get_weather(**tool_use_block.input)
print(result)  # {'city': 'Lyon', 'temperature': 18, ...}

Étapes 4 & 5 — Renvoi du résultat et réponse finale

Vous renvoyez le résultat dans un nouveau message avec le rôle user et le type tool_result, en incluant l'historique complet :

Python
import json

# Construire l'historique complet
messages = [
    {"role": "user", "content": "Quel temps fait-il à Lyon en ce moment ?"},
    {"role": "assistant", "content": response.content},  # réponse avec tool_use
    {
        "role": "user",
        "content": [{
            "type": "tool_result",
            "tool_use_id": tool_use_block.id,  # doit correspondre à l'ID du bloc
            "content": json.dumps(result)       # résultat en JSON string
        }]
    }
]

# Étape 5 : deuxième appel pour obtenir la réponse finale
final_response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=[...],  # même liste d'outils
    messages=messages
)

print(final_response.content[0].text)
# "À Lyon en ce moment, il fait 18°C avec un temps nuageux et quelques éclaircies.
#  L'humidité est de 72%."
⚠ Piège — L'ID du tool_use

Le champ tool_use_id dans votre tool_result doit correspondre exactement à l'id du bloc tool_use reçu. Si vous utilisez un ID inventé ou oublié, le modèle retournera une erreur ou ignorera le résultat. Toujours lire l'ID depuis block.id, jamais l'écrire en dur.

💡 Astuce — Résultat enrichi

Le contenu du tool_result n'est pas limité à du texte. Vous pouvez y envoyer un tableau, un objet JSON structuré, voire une image (base64) si votre outil retourne une capture d'écran ou un graphique. Le modèle interprétera le tout.

JSON Schema et définition d'outils

Pourquoi la définition est critique

La qualité de votre définition d'outil est le facteur n°1 de la fiabilité du function calling. Un outil mal défini produit des arguments incorrects, des appels au mauvais moment, ou des paramètres manquants. Le modèle n'a que votre définition pour comprendre ce que fait l'outil et comment l'utiliser.

Rédiger des descriptions efficaces

La description de l'outil lui-même doit contenir trois informations : ce que l'outil fait, quand l'utiliser (et quand ne pas l'utiliser), et ses limites. Celle de chaque paramètre doit expliquer le format attendu et les valeurs acceptables.

📌 Bonne description vs mauvaise

❌ Mauvais : "description": "La ville"
✅ Bon : "description": "Nom de la ville en français ou anglais (ex: 'Paris', 'New York'). Ne pas inclure le code pays."

⚠ Piège — required vs optional

Tout paramètre absent de required peut être omis par le modèle. Si votre code plante quand un paramètre optionnel est absent, vous avez un bug côté serveur, pas côté LLM. Définissez toujours des valeurs par défaut dans votre code pour les paramètres optionnels, et ne comptez pas sur le modèle pour les fournir systématiquement.

La boucle complète en une fonction réutilisable

Plutôt que de gérer manuellement chaque étape, voici un pattern production-ready qui gère la boucle tool_use automatiquement :

Python
import anthropic
import json
from typing import Callable

client = anthropic.Anthropic()

def run_agent(
    user_message: str,
    tools: list[dict],
    tool_handlers: dict[str, Callable],
    model: str = "claude-sonnet-4-6",
    max_iterations: int = 10
) -> str:
    """
    Exécute une boucle agent complète.
    tool_handlers: dict {"nom_outil": fonction_python}
    """
    messages = [{"role": "user", "content": user_message}]

    for _ in range(max_iterations):
        response = client.messages.create(
            model=model,
            max_tokens=4096,
            tools=tools,
            messages=messages
        )

        # Réponse finale : plus d'outil à appeler
        if response.stop_reason == "end_turn":
            return response.content[0].text

        # stop_reason == "tool_use"
        # Ajouter la réponse assistant à l'historique
        messages.append({"role": "assistant", "content": response.content})

        # Exécuter tous les outils demandés et collecter les résultats
        tool_results = []
        for block in response.content:
            if block.type != "tool_use":
                continue
            handler = tool_handlers.get(block.name)
            if not handler:
                raise ValueError(f"Aucun handler pour l'outil: {block.name}")

            try:
                result = handler(**block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": json.dumps(result, ensure_ascii=False)
                })
            except Exception as e:
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "is_error": True,
                    "content": f"Erreur: {str(e)}"
                })

        messages.append({"role": "user", "content": tool_results})

    raise RuntimeError("Nombre max d'itérations atteint")

⚠ Piège — max_tokens insuffisant

Si max_tokens est trop petit, le modèle peut être interrompu en plein raisonnement ou en pleine génération d'un bloc tool_use, produisant un JSON invalide. Pour les agents avec outils complexes, prévoir au minimum 2048 tokens. La valeur du bloc tool_use lui-même n'est pas déduite du budget des tokens de sortie de votre application — mais elle consomme les tokens de la réponse.

Parallel tool use — plusieurs outils simultanément

Le problème du séquentiel naïf

Sans parallel tool use, un agent qui doit récupérer la météo à Paris, Lyon et Marseille effectue 3 appels API séquentiels : 3 × (latence réseau + inférence LLM). Avec le parallel tool use, Claude peut demander ces 3 appels en une seule réponse, réduisant la latence totale au temps du plus lent des trois.

Comment Claude signale plusieurs appels

Quand Claude veut appeler plusieurs outils en parallèle, il inclut plusieurs blocs tool_use dans sa réponse content :

JSON — Réponse parallèle
{
  "stop_reason": "tool_use",
  "content": [
    {
      "type": "tool_use",
      "id": "toolu_01A",
      "name": "get_weather",
      "input": {"city": "Paris"}
    },
    {
      "type": "tool_use",
      "id": "toolu_01B",
      "name": "get_weather",
      "input": {"city": "Lyon"}
    },
    {
      "type": "tool_use",
      "id": "toolu_01C",
      "name": "get_weather",
      "input": {"city": "Marseille"}
    }
  ]
}

Exécution parallèle avec asyncio

Python — asyncio
import asyncio
import anthropic
import json

async def get_weather_async(city: str) -> dict:
    await asyncio.sleep(0.1)  # Simuler latence réseau
    data = {"Paris": 22, "Lyon": 18, "Marseille": 26}
    return {"city": city, "temp_c": data.get(city, 15)}

async def run_parallel_tools(response_content, tool_handlers_async):
    """Exécute tous les tool_use en parallèle."""
    tasks = []
    tool_use_blocks = [b for b in response_content if b.type == "tool_use"]

    for block in tool_use_blocks:
        handler = tool_handlers_async[block.name]
        tasks.append(handler(**block.input))

    results = await asyncio.gather(*tasks, return_exceptions=True)

    tool_results = []
    for block, result in zip(tool_use_blocks, results):
        if isinstance(result, Exception):
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "is_error": True,
                "content": str(result)
            })
        else:
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": json.dumps(result)
            })
    return tool_results

async def main():
    client = anthropic.AsyncAnthropic()

    response = await client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        tools=[{
            "name": "get_weather",
            "description": "Météo actuelle pour une ville.",
            "input_schema": {
                "type": "object",
                "properties": {"city": {"type": "string"}},
                "required": ["city"]
            }
        }],
        messages=[{
            "role": "user",
            "content": "Compare la météo à Paris, Lyon et Marseille."
        }]
    )

    tool_results = await run_parallel_tools(
        response.content,
        {"get_weather": get_weather_async}
    )

    # Deuxième appel pour la réponse finale
    final = await client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        tools=[...],
        messages=[
            {"role": "user", "content": "Compare la météo à Paris, Lyon et Marseille."},
            {"role": "assistant", "content": response.content},
            {"role": "user", "content": tool_results}
        ]
    )
    print(final.content[0].text)

asyncio.run(main())
📌 Vous n'avez rien à activer

Le parallel tool use est actif par défaut sur tous les modèles Claude récents. Vous n'avez qu'à être prêt à traiter un tableau de blocs tool_use plutôt qu'un seul — et à tous renvoyer leurs résultats dans le même message tool_result.

⚠ Piège — Résultats partiels

Si Claude envoie 3 appels parallèles et que vous n'en renvoyez que 2 résultats, le modèle retournera une erreur ou se comportera de manière imprévisible. Vous devez toujours renvoyer un tool_result pour chaque bloc tool_use reçu, même si c'est un résultat d'erreur.

Tool choice — contrôler le comportement

Les 4 modes de tool_choice

Par défaut, Claude décide seul s'il doit appeler un outil. Le paramètre tool_choice vous permet de modifier ce comportement :

ModeValeurComportement
Auto{"type": "auto"}Défaut. Claude décide librement.
Any{"type": "any"}Claude doit appeler au moins un outil.
Tool{"type": "tool", "name": "..."}Claude doit appeler exactement cet outil.
None{"type": "none"}Claude ne peut appeler aucun outil (texte seulement).
Python
# Forcer l'appel d'un outil spécifique (extraction structurée)
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=[{
        "name": "extract_contact_info",
        "description": "Extrait les informations de contact d'un texte.",
        "input_schema": {
            "type": "object",
            "properties": {
                "name":  {"type": "string"},
                "email": {"type": "string", "format": "email"},
                "phone": {"type": "string"},
                "company": {"type": "string"}
            },
            "required": ["name"]
        }
    }],
    tool_choice={"type": "tool", "name": "extract_contact_info"},
    messages=[{
        "role": "user",
        "content": "Jean Dupont, CEO de TechCorp. Email: j.dupont@techcorp.fr, Tel: 06 12 34 56 78"
    }]
)

# La réponse contient TOUJOURS le bloc tool_use
tool_block = response.content[0]
print(tool_block.input)
# {"name": "Jean Dupont", "email": "j.dupont@techcorp.fr",
#  "phone": "06 12 34 56 78", "company": "TechCorp"}

Cas d'usage de chaque mode

  • auto : agents conversationnels généraux où le modèle doit juger quand agir
  • any : vous savez qu'une action est nécessaire mais laissez Claude choisir laquelle (workflow d'extraction multi-outils)
  • tool (nom fixé) : extraction structurée de données, classification forcée, parsing de documents — vous voulez du JSON garanti, pas de prose
  • none : mode "lecture seule" où vous fournissez les outils pour contexte mais ne voulez pas d'actions (revue de code, explication)
💡 Extraction structurée sans hallucination

Forcer {"type": "tool", "name": "extract_..."} est le moyen le plus fiable d'obtenir du JSON structuré depuis Claude. Contrairement au prompting "réponds en JSON", un bloc tool_use est toujours du JSON valide — le modèle ne peut pas sortir du schéma.

Streaming avec les outils

Pourquoi streamer avec des outils

Le streaming permet d'afficher le texte de raisonnement de Claude à l'utilisateur pendant qu'il réfléchit, avant même qu'un outil soit appelé. C'est crucial pour l'expérience utilisateur dans les agents lents (appels API externes, requêtes BDD longues).

Les événements de streaming à surveiller

Python — Streaming
import anthropic
import json

client = anthropic.Anthropic()

def stream_with_tools(user_message: str, tools: list, handlers: dict):
    messages = [{"role": "user", "content": user_message}]

    while True:
        current_tool_calls = {}  # id -> {name, input_str}

        with client.messages.stream(
            model="claude-sonnet-4-6",
            max_tokens=2048,
            tools=tools,
            messages=messages
        ) as stream:

            for event in stream:
                event_type = type(event).__name__

                # Texte de raisonnement — afficher immédiatement
                if event_type == "RawContentBlockDeltaEvent":
                    if hasattr(event.delta, "text"):
                        print(event.delta.text, end="", flush=True)

                    # Accumulation de l'input JSON d'un tool_use
                    elif hasattr(event.delta, "partial_json"):
                        block_idx = event.index
                        if block_idx in current_tool_calls:
                            current_tool_calls[block_idx]["input_str"] += event.delta.partial_json

                # Début d'un bloc tool_use
                elif event_type == "RawContentBlockStartEvent":
                    block = event.content_block
                    if block.type == "tool_use":
                        current_tool_calls[event.index] = {
                            "id": block.id,
                            "name": block.name,
                            "input_str": ""
                        }
                        print(f"\n[→ Appel outil: {block.name}]", flush=True)

            # Récupérer la réponse finale complète
            final_response = stream.get_final_message()

        if final_response.stop_reason == "end_turn":
            print()  # Saut de ligne final
            break

        # Construire les tool_results
        messages.append({"role": "assistant", "content": final_response.content})
        tool_results = []

        for idx, call in current_tool_calls.items():
            try:
                input_data = json.loads(call["input_str"] or "{}")
                result = handlers[call["name"]](**input_data)
                content = json.dumps(result)
                print(f"[← Résultat: {content[:80]}...]")
            except Exception as e:
                content = f"Erreur: {e}"

            tool_results.append({
                "type": "tool_result",
                "tool_use_id": call["id"],
                "content": content
            })

        messages.append({"role": "user", "content": tool_results})
⚠ Piège — JSON partiel du tool_use

En streaming, l'input d'un bloc tool_use arrive en fragments (partial_json). Vous devez les accumuler dans une chaîne et parser le JSON complet après la fin du bloc (RawContentBlockStopEvent), pas au fur et à mesure. Tenter de parser un JSON partiel produira toujours une exception.

Le principe de responsabilité unique

Un outil doit faire une seule chose, clairement définie. Un outil search_and_summarize(query) est une mauvaise idée : si la recherche réussit mais que le modèle ne veut pas de résumé, vous perdez de la flexibilité. Séparez search(query) et laissez Claude décider quand résumer.

Gestion des erreurs et retours d'erreur

Retourner vs lever une exception

Quand un outil échoue, vous avez deux options : lever une exception (qui plante votre agent) ou retourner une erreur structurée (que le modèle peut interpréter et dont il peut se récupérer). La deuxième option est presque toujours préférable pour les agents en production.

Python
# Option 1 : is_error flag dans tool_result (API Anthropic)
# Le modèle sait que c'est une erreur et adapte son comportement

def call_tool_safe(block, handlers):
    try:
        result = handlers[block.name](**block.input)
        return {
            "type": "tool_result",
            "tool_use_id": block.id,
            "content": json.dumps(result)
        }
    except ValueError as e:
        # Erreur de validation — le modèle peut corriger et réessayer
        return {
            "type": "tool_result",
            "tool_use_id": block.id,
            "is_error": True,
            "content": f"Erreur de validation: {e}"
        }
    except TimeoutError:
        return {
            "type": "tool_result",
            "tool_use_id": block.id,
            "is_error": True,
            "content": "L'outil a mis trop de temps à répondre (timeout 30s). Réessayer dans quelques instants."
        }
    except Exception as e:
        return {
            "type": "tool_result",
            "tool_use_id": block.id,
            "is_error": True,
            "content": f"Erreur inattendue: {type(e).__name__}: {e}"

Retry avec backoff exponentiel

Python
import time
from functools import wraps

def with_retry(max_retries=3, backoff_base=1.0, retryable_exceptions=(TimeoutError, ConnectionError)):
    """Décorateur pour retry avec backoff exponentiel."""
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            for attempt in range(max_retries):
                try:
                    return func(*args, **kwargs)
                except retryable_exceptions as e:
                    if attempt == max_retries - 1:
                        raise
                    wait = backoff_base * (2 ** attempt)
                    time.sleep(wait)
            return func(*args, **kwargs)
        return wrapper
    return decorator

@with_retry(max_retries=3, backoff_base=0.5)
def call_external_api(endpoint: str, params: dict) -> dict:
    response = requests.get(endpoint, params=params, timeout=10)
    response.raise_for_status()
    return response.json()

La boucle agent — agentic loop

Qu'est-ce qu'un agent ?

Un agent est un système où un LLM contrôle son propre cycle d'exécution : il décide quand appeler des outils, dans quel ordre, et quand il a suffisamment d'informations pour répondre. Contrairement à un appel unique, un agent peut enchaîner un nombre arbitraire d'étapes pour accomplir une tâche complexe.

┌────────────────────────────────────────────────────────┐ │ Boucle Agent │ │ │ │ Tâche initiale │ │ │ │ │ ▼ │ │ ┌────────┐ Besoin d'un outil ? │ │ │ LLM │──────────── Oui ──────▶ Exécuter outil │ │ │ │◀──────────────────────── Résultat │ │ └────────┘ │ │ │ │ │ │ Non (stop_reason = "end_turn") │ │ ▼ │ │ Réponse finale à l'utilisateur │ └────────────────────────────────────────────────────────┘

Gardes-fous pour les agents en production

Sans contraintes, un agent peut tourner indéfiniment, consommer des tokens en boucle, ou effectuer des actions destructrices enchaînées. Ces gardes-fous sont non-négociables :

Python — Agent avec gardes-fous
import time
from dataclasses import dataclass, field

@dataclass
class AgentConfig:
    max_iterations: int = 15          # Limite d'étapes
    max_tokens_total: int = 100_000   # Budget tokens total
    timeout_seconds: float = 300      # Timeout global
    require_confirmation: list = field(default_factory=list)  # Outils dangereux

class SafeAgent:
    def __init__(self, tools, handlers, config: AgentConfig = None):
        self.tools = tools
        self.handlers = handlers
        self.config = config or AgentConfig()
        self.client = anthropic.Anthropic()

    def run(self, user_message: str) -> str:
        messages = [{"role": "user", "content": user_message}]
        total_tokens = 0
        start_time = time.time()

        for iteration in range(self.config.max_iterations):
            # Vérification du timeout
            if time.time() - start_time > self.config.timeout_seconds:
                return f"[TIMEOUT] L'agent a dépassé {self.config.timeout_seconds}s."

            response = self.client.messages.create(
                model="claude-sonnet-4-6",
                max_tokens=4096,
                tools=self.tools,
                messages=messages
            )

            total_tokens += response.usage.input_tokens + response.usage.output_tokens
            if total_tokens > self.config.max_tokens_total:
                return "[BUDGET] Budget tokens épuisé."

            if response.stop_reason == "end_turn":
                return response.content[0].text

            messages.append({"role": "assistant", "content": response.content})
            tool_results = []

            for block in response.content:
                if block.type != "tool_use":
                    continue

                # Confirmation pour les outils dangereux
                if block.name in self.config.require_confirmation:
                    confirm = input(f"Confirmer l'appel à {block.name}({block.input}) ? [o/N] ")
                    if confirm.lower() != "o":
                        tool_results.append({
                            "type": "tool_result",
                            "tool_use_id": block.id,
                            "is_error": True,
                            "content": "Action refusée par l'utilisateur."
                        })
                        continue

                result = self.handlers[block.name](**block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": json.dumps(result)
                })

            messages.append({"role": "user", "content": tool_results})

        return "[MAX_ITER] L'agent a atteint la limite d'itérations."
⚠ La boucle infinie cachée

Un agent peut boucler silencieusement si un outil retourne toujours une erreur que le modèle essaie de corriger sans succès, ou si le modèle continue de chercher une information qu'aucun outil ne peut fournir. Le max_iterations est votre filet de sécurité — ne jamais le dépasser de 20 dans un contexte de production.

Chaînage d'outils et dépendances

Dépendances implicites vs explicites

Quand l'output d'un outil devient l'input d'un autre, on a un chaînage. Claude gère naturellement ce chaînage — il utilise les résultats des appels précédents dans son contexte. Mais comprendre les patterns de dépendance vous aide à concevoir des outils qui s'enchaînent bien.

Chaînage séquentiel (dépendance forte) search_user(email) │ ▼ (retourne user_id) get_orders(user_id) │ ▼ (retourne order_ids) get_order_details(order_id) ◀── appelé N fois (parallel) │ ▼ Résumé final

Passer des IDs entre outils

Le pattern le plus courant : un outil de recherche retourne un identifiant, qu'un outil d'action utilise ensuite. Vos outils doivent toujours retourner les IDs nécessaires aux étapes suivantes :

# ❌ Retourne des données sans ID — chaînage impossible

# ✅ Retourne toujours les identifiants
def find_user_good(email: str) -> dict:

    if not user:
        return {"found": False, "message": f"Aucun utilisateur avec l'email {email}"}
    return {
        "found": True,
        "user_id": user.id,        # Clé pour les outils suivants
        "name": user.name,
        "email": user.email,
        "account_type": user.account_type
    }

    orders = db.get_orders(user_id)
    return {
        "user_id": user_id,
        "order_count": len(orders),
        "orders": [
            {
                "order_id": o.id,  # ID pour get_order_details

                "total": o.total,
                "date": o.date.isoformat()
            }
            for o in orders
        ]
    }

Si tous vos outils utilisent user_id, order_id, product_id (jamais id seul), Claude comprend instantanément les dépendances sans ambiguïté. Un champ nommé juste id est ambigu quand plusieurs entités sont en jeu.

Le pattern ReAct

Reason + Act. Le modèle alterne explicitement entre une phase de raisonnement ("Je dois d'abord trouver l'ID utilisateur, ensuite...") et une phase d'action (appel d'outil). Claude applique naturellement ce pattern — le texte avant un tool_use est souvent ce raisonnement.

Pour encourager le raisonnement explicite et améliorer la fiabilité sur les tâches complexes, vous pouvez le demander dans votre system prompt :

Le pattern Plan-and-Execute

Pour les tâches très complexes (10+ étapes, décisions conditionnelles), l'approche Plan-and-Execute divise le travail en deux phases :

  1. Phase de planification : Claude génère un plan structuré sans appeler d'outils
  2. Phase d'exécution : Claude exécute le plan étape par étape, avec la possibilité de le réviser
Python — Plan-and-Execute
def plan_and_execute(task: str, tools: list, handlers: dict) -> str:

    plan_response = client.messages.create(
        model="claude-sonnet-4-6",

            "Format: 1. [ACTION] Description\n2. [ACTION] Description...\n"
            "Actions possibles: SEARCH, FETCH, CALCULATE, CREATE, UPDATE, NOTIFY"
        ),
        messages=[{"role": "user", "content": f"Tâche: {task}"}]

    print(f"Plan généré:\n{plan}\n")

            "Après chaque étape, confirme qu'elle est accomplie avant de passer à la suivante."
        ),
        tools=tools,
        messages=[{"role": "user", "content": task}]

Pourquoi MCP ? Le problème M×N

Le problème sans MCP

Imaginez 10 applications IA (Claude, GPT, Gemini, agents custom…) et 10 sources de données (GitHub, Slack, PostgreSQL, Notion, filesystem…). Sans protocole standardisé, chaque combinaison nécessite une intégration sur mesure : 10 × 10 = 100 connecteurs à maintenir. C'est le problème M×N.

Sans MCP — M×N intégrations Avec MCP — M+N Claude ────▶ GitHub connector Claude ┐ Claude ────▶ Slack connector GPT ├──▶ MCP Protocol ──▶ GitHub MCP Server Claude ────▶ Notion connector Gemini ┘ ──▶ Slack MCP Server GPT ────▶ GitHub connector (différent!) ──▶ Notion MCP Server GPT ────▶ Slack connector (différent!) ...

MCP est un protocole standardisé (créé par Anthropic en novembre 2024, donné à la Linux Foundation en décembre 2025) qui résout ce problème : vous écrivez un serveur MCP une seule fois pour GitHub, et il fonctionne avec tous les clients MCP — Claude, GPT, Gemini, et votre agent custom.

La spécification actuelle

La version courante de la spec MCP est 2025-11-25. Le transport recommandé pour les serveurs distants est Streamable HTTP (qui a remplacé SSE depuis la spec 2025-03-26). Architecture MCP — hôte, client, serveur

Les trois acteurs

┌─────────────────────────────────────────────────────────────┐ │ HÔTE MCP │ │ (Claude Desktop, Claude Code, votre application) │ │ │ │ ┌─────────────┐ ┌─────────────┐ │ │ │ Client MCP │ │ Client MCP │ ◀── Un client │ │ │ (1:1) │ │ (1:1) │ par serveur │ │ └──────┬──────┘ └──────┬──────┘ │ └──────────│────────────────────── │──────────────────────────┘ │ MCP Protocol │ MCP Protocol │ (stdio/HTTP) │ (stdio/HTTP) ▼ ▼ ┌───────────────┐ ┌───────────────┐ │ Serveur MCP │ │ Serveur MCP │ │ (GitHub) │ │ (PostgreSQL) │ │ Tools │ │ Tools │ │ Resources │ │ Resources │ └───────────────┘ └───────────────┘
  • Hôte : l'application qui orchestre tout (Claude Desktop, VS Code + extension, votre app custom). Il gère le cycle de vie des connexions, les permissions, et présente les capacités des serveurs au LLM.
  • Client MCP : composant dans l'hôte qui maintient une connexion 1:1 avec un serveur. Gère la session, le handshake, et les requêtes/réponses.
  • Serveur MCP : processus indépendant qui expose des outils, des ressources, et des prompts. C'est ce que vous développez quand vous créez une intégration MCP.

Cycle de vie d'une connexion MCP

Séquence de connexion
Client                              Serveur
  │                                    │
  │──── initialize (protocolVersion) ──▶│
  │◀─── initialized (capabilities) ────│
  │                                    │
  │──── notifications/initialized ─────▶│  (Handshake terminé)
  │                                    │
  │──── tools/list ────────────────────▶│
  │◀─── [liste des outils] ────────────│
  │                                    │
  │──── resources/list ─────────────────▶│
  │◀─── [liste des resources] ──────────│
  │                                    │
  │──── tools/call {name, arguments} ──▶│
  │◀─── {content} ─────────────────────│

Les capacités négociées

Lors du handshake, client et serveur négocient ce qu'ils supportent. Un serveur peut déclarer qu'il supporte tools, resources, prompts, logging, sampling. Le client lui indique ce qu'il peut offrir (ex: support du sampling pour que le serveur puisse demander des inférences LLM).

⚠ Un client par serveur, pas un client pour tous

Une erreur d'architecture courante : créer un unique client MCP "générique" qui se connecte à plusieurs serveurs. Ce n'est pas le modèle MCP. Chaque serveur doit avoir son propre client (connexion dédiée). L'hôte gère l'ensemble des clients et agrège leurs capacités.

Un serveur MCP réel — accès PostgreSQL

Voici un exemple production-grade d'un serveur MCP donnant accès à une base PostgreSQL, avec connection pooling, paramètres sécurisés, et gestion d'erreurs robuste :

Python — postgres_server.py
import os
import asyncpg
from mcp.server.fastmcp import FastMCP
from contextlib import asynccontextmanager
from typing import Any

# Configuration depuis les variables d'environnement
DB_URL = os.environ["DATABASE_URL"]
MAX_ROWS = int(os.environ.get("MAX_ROWS", "100"))

# Tables accessibles (whitelist de sécurité)
ALLOWED_TABLES = frozenset(os.environ.get(
    "ALLOWED_TABLES", "products,orders,customers"
).split(","))

mcp = FastMCP("postgres-reader")

# Pool de connexions (partagé entre les requêtes)
_pool: asyncpg.Pool | None = None

async def get_pool() -> asyncpg.Pool:
    global _pool
    if _pool is None:
        _pool = await asyncpg.create_pool(DB_URL, min_size=2, max_size=10)
    return _pool

@mcp.tool()
async def query_table(
    table: str,
    filters: dict = None,
    columns: list[str] = None,
    limit: int = 20
) -> dict:
    """
    Interroge une table de la base de données.
    Uniquement des opérations de lecture (SELECT).

    Args:
        table: Nom de la table (doit être dans la liste autorisée)
        filters: Conditions WHERE sous forme {colonne: valeur}
        columns: Colonnes à retourner (toutes si omis)
        limit: Nombre max de lignes (max: 100)
    """
    # Validation de sécurité
    if table not in ALLOWED_TABLES:
        return {
            "error": f"Table '{table}' non autorisée. Tables disponibles: {', '.join(ALLOWED_TABLES)}"
        }

    limit = min(limit, MAX_ROWS)
    cols = ", ".join(columns) if columns else "*"

    # Construction sécurisée de la requête (pas d'interpolation de chaîne!)
    query = f"SELECT {cols} FROM {table}"
    params = []

    if filters:
        conditions = []
        for i, (col, val) in enumerate(filters.items(), 1):
            conditions.append(f"{col} = ${i}")
            params.append(val)
        query += " WHERE " + " AND ".join(conditions)

    query += f" LIMIT ${len(params) + 1}"
    params.append(limit)

    pool = await get_pool()
    async with pool.acquire() as conn:
        rows = await conn.fetch(query, *params)

    return {
        "table": table,
        "count": len(rows),
        "rows": [dict(row) for row in rows]
    }

@mcp.tool()
async def get_table_schema(table: str) -> dict:
    """
    Retourne le schéma (colonnes et types) d'une table.

    Args:
        table: Nom de la table
    """
    if table not in ALLOWED_TABLES:
        return {"error": f"Table '{table}' non autorisée"}

    pool = await get_pool()
    async with pool.acquire() as conn:
        columns = await conn.fetch("""
            SELECT column_name, data_type, is_nullable
            FROM information_schema.columns
            WHERE table_name = $1
            ORDER BY ordinal_position
        """, table)

    return {
        "table": table,
        "columns": [dict(col) for col in columns]
    }

if __name__ == "__main__":
    mcp.run()

VALID_TOKENS = set(os.environ["API_TOKENS"].split(","))

class BearerAuthMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):

Resources et Resource Templates

Qu'est-ce qu'une Resource MCP ?

Une Resource est une donnée exposée par le serveur que le client peut lire — à l'opposé d'un Tool, elle ne déclenche pas d'action. Pensez-y comme à un système de fichiers virtuel : chaque resource a une URI unique, un type MIME, et un contenu (texte ou binaire).

Tools vs Resources Tool → Déclenche une action, peut avoir des effets secondaires Comparable à : un endpoint POST d'API REST Exemple : send_email(), create_ticket(), execute_sql() Resource → Expose des données en lecture, aucun effet secondaire Comparable à : un endpoint GET d'API REST Exemple : file://logs/app.log, db://schema/users, config://settings

Exposer des Resources avec FastMCP

Python

import json
import os

mcp = FastMCP("resource-server")

@mcp.resource("config://app/settings")
def get_app_settings() -> str:

    settings = {
        "version": "2.1.0",
        "environment": os.environ.get("ENV", "production"),
        "features": {"dark_mode": True, "beta_api": False}
    }

@mcp.resource("logs://app/{date}")
def get_logs(date: str) -> str:
    """
    Logs de l'application pour une date donnée.
    URI: logs://app/2024-01-15
    """
    log_file = f"/var/log/app/{date}.log"
    if not os.path.exists(log_file):
        return f"Aucun log pour la date {date}"

        return f.read()

    return json.dumps(dict(row), default=str)

    """Retourne toutes les resources disponibles."""

        {
            "uri": f"db://tables/{table}/schema",
            "name": f"Schéma de {table}",
            "mimeType": "application/json"
        }
        for table in tables

⚠ Resources ≠ persistance de données

Les Resources sont des lectures — elles ne stockent pas de données. Sampling — le serveur qui interroge le LLM

Le concept de Sampling

Le Sampling est la primitive MCP la plus avancée — et la plus souvent mal comprise. Elle inverse le flux habituel : le serveur MCP peut demander au client d'effectuer une inférence LLM. Cela permet de construire des serveurs MCP qui utilisent l'IA pour traiter leurs données, sans avoir besoin d'une clé API ou d'un LLM intégré.

Flux normal (Tool) Flux Sampling Hôte/Client Hôte/Client │ │ │ tools/call │◀── sampling/createMessage ▼ │ Serveur MCP ▼ │ Serveur MCP │ exécute la logique │ a besoin d'IA pour traiter │ (pas d'IA) │ ses données internes ▼ │ → demande une inférence au client Résultat ▼ Résultat LLM → traitement → réponse

Implémentation du Sampling côté serveur

}}"""
                )

        ],
        max_tokens=200,

Contrôle humain dans la boucle (Human-in-the-loop)

Le Sampling est conçu avec un principe de sécurité fort : l'hôte contrôle toutes les inférences. Il peut inspecter les messages, modifier les paramètres, limiter le budget tokens, ou refuser complètement une demande de sampling. Cela empêche un serveur MCP malveillant de faire des inférences LLM non autorisées :

📌 Principe de moindre privilège LLM

Un serveur MCP qui implémente le Sampling doit demander explicitement la capacité sampling lors du handshake. Si l'hôte ne la supporte pas ou la refuse, le serveur doit se dégrader gracieusement. Ne jamais supposer que le Sampling sera disponible — vérifiez les capacités négociées.

Sécurité dans MCP — OAuth et contrôle d'accès

Le modèle de confiance MCP

MCP définit explicitement qui fait confiance à qui. Les utilisateurs font confiance à l'hôte (ils l'ont installé). L'hôte fait confiance aux serveurs MCP (après vérification). Les serveurs MCP ne font pas automatiquement confiance aux utilisateurs — ils ne connaissent que ce que l'hôte leur envoie.

Authentification OAuth 2.1

La spec MCP recommande OAuth 2.1 avec PKCE pour les serveurs distants. Voici l'implémentation avec le SDK Python :

Python — OAuth avec FastMCP
from mcp.server.auth.provider import OAuthAuthorizationServerProvider
from mcp.server.auth.settings import AuthSettings, ClientRegistrationOptions
from mcp.server.fastmcp import FastMCP
import secrets
import time

# Implémentation d'un provider OAuth simple (en production: utiliser un vrai IdP)
class SimpleOAuthProvider(OAuthAuthorizationServerProvider):
    def __init__(self):
        self.clients = {}    # client_id -> client_secret
        self.tokens = {}     # access_token -> {client_id, scopes, expires}
        self.codes = {}      # auth_code -> {client_id, scopes, code_verifier}

    async def get_client(self, client_id: str):
        return self.clients.get(client_id)

    async def create_token(self, client_id: str, scopes: list[str]) -> str:
        token = secrets.token_urlsafe(32)
        self.tokens[token] = {
            "client_id": client_id,
            "scopes": scopes,
            "expires": time.time() + 3600  # 1 heure
        }
        return token

    async def validate_token(self, token: str) -> dict | None:
        info = self.tokens.get(token)
        if not info or info["expires"] < time.time():
            return None
        return info

provider = SimpleOAuthProvider()

mcp = FastMCP(
    "secure-mcp",
    auth=AuthSettings(
        issuer_url="https://mon-serveur.com",
        client_registration_options=ClientRegistrationOptions(
            enabled=True,
            valid_scopes=["read:data", "write:data", "admin"],
        )
    )
)

# Contrôle d'accès par scope dans les outils
@mcp.tool()
async def read_data(query: str) -> dict:
    """Lecture de données — nécessite le scope read:data"""
    # FastMCP vérifie automatiquement l'authentification
    # Les scopes sont disponibles dans le contexte
    ctx = mcp.get_context()
    token_info = await provider.validate_token(ctx.request_context.auth_token)

    if "read:data" not in token_info["scopes"]:
        return {"error": "Permission refusée — scope read:data requis"}

    return await db.query(query)

@mcp.tool()
async def delete_record(record_id: str) -> dict:
    """Suppression — nécessite le scope admin"""
    ctx = mcp.get_context()
    token_info = await provider.validate_token(ctx.request_context.auth_token)

    if "admin" not in token_info["scopes"]:
        return {"error": "Permission refusée — scope admin requis"}

    await db.delete(record_id)
    return {"deleted": record_id}

Checklist de sécurité pour un serveur MCP en production

📋 Checklist sécurité

✅ Transport : TLS obligatoire pour Streamable HTTP (HTTPS uniquement)
✅ Auth : OAuth 2.1 + PKCE pour les clients distants
✅ Validation : Valider tous les arguments des outils côté serveur
✅ Whitelist : Tables/fichiers/endpoints accessibles définis statiquement
✅ Rate limiting : Limiter les appels par client/IP/token
✅ Logging : Auditer tous les appels d'outils avec l'identité du client
✅ Injection : Utiliser des requêtes paramétrées (jamais de string formatting pour SQL)
✅ Secrets : Jamais de credentials dans le code — variables d'environnement uniquement
✅ SSRF : Valider et filtrer les URLs si le serveur fait des requêtes HTTP
✅ DoS : Limiter la taille des réponses et le temps d'exécution des outils

⚠ Serveurs MCP tiers non vérifiés

N'installez jamais un serveur MCP tiers sans auditer son code. Un serveur MCP malveillant a accès à tout ce que l'hôte lui envoie (y compris des données confidentielles dans le contexte de conversation). Préférez les serveurs open-source avec une communauté active, et isolez les serveurs à risque dans des containers ou sandboxes.

def log_tool_call(func):
    """Décorateur qui logue tous les appels d'outils."""
    @wraps(func)
    async def wrapper(**kwargs):
        start = time.time()
        tool_name = func.__name__
        logger.info(json.dumps({
            "event": "tool_call_start",
            "tool": tool_name,
            "args": {k: str(v)[:100] for k, v in kwargs.items()}
        }))
        try:
            result = await func(**kwargs)
            elapsed = (time.time() - start) * 1000
            logger.info(json.dumps({
                "event": "tool_call_success",
                "tool": tool_name,
                "elapsed_ms": round(elapsed, 2)
            }))
            return result
        except Exception as e:
            elapsed = (time.time() - start) * 1000
            logger.error(json.dumps({
                "event": "tool_call_error",
                "tool": tool_name,
                "error": str(e),
                "elapsed_ms": round(elapsed, 2)
            }))
            raise
    return wrapper

@mcp.tool()
@log_tool_call
async def my_tool(param: str) -> str:
    return f"Résultat pour {param}"

def instrument_tool(func):
    """Décorateur Prometheus pour les outils MCP."""
    @wraps(func)
    async def wrapper(**kwargs):
        start = time.time()
        status = "success"
        try:
            result = await func(**kwargs)
            if isinstance(result, dict) and result.get("error"):
                status = "error"
            return result
        except Exception:
            status = "error"
            raise
        finally:
            tool_calls_total.labels(func.__name__, status).inc()
            tool_duration.labels(func.__name__).observe(time.time() - start)
    return wrapper
