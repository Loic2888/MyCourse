Module 1 · Leçon 1 · Fondamentaux RAG

Pourquoi le RAG ?

Un LLM, aussi puissant soit-il, souffre de trois limitations structurelles qui le rendent insuffisant pour la plupart des applications réelles :

  • Connaissances figées — le modèle ne sait rien après sa date de coupure (souvent 6 à 12 mois avant son lancement).
  • Hallucinations — face à une question à laquelle il ne connaît pas la réponse, le modèle invente avec une confiance totale.
  • Données privées inaccessibles — vos documents internes, base de clients, wiki d'entreprise n'ont jamais été dans l'entraînement.

La solution : Retrieval-Augmented Generation

Le RAG résout ces trois problèmes avec une idée simple : avant de générer une réponse, cherchez les documents pertinents dans votre base de connaissances, et injectez-les dans le contexte du LLM.

Sans RAG : Question ──────────────────────────────► LLM ──► Réponse (peut halluciner) Avec RAG : Question ──► [Recherche vectorielle] ──► Documents pertinents │ Question ──────────────────────────────────────────► LLM ──► Réponse ancrée

Pourquoi "retrieval" plutôt que "fine-tuning" ?

Une alternative au RAG est le fine-tuning : réentraîner le modèle sur vos données. Mais ce n'est généralement pas la bonne solution :

CritèreRAGFine-tuning
Données qui changent fréquemment✓ Idéal✗ Réentraîner à chaque MAJ
Traçabilité des sources✓ Citations possibles✗ Le modèle "sait" sans dire d'où
Coût d'implémentation✓ Faible✗ Élevé (GPU, temps)
Nouvelles connaissances factuelles✓ Excellent~ Partiel
Adapter le style/tone du modèle✗ Limité✓ Excellent
ℹ Règle pratique Fine-tunez pour changer comment le modèle répond (style, format, ton). Utilisez le RAG pour changer ce que le modèle sait (données factuelles, documents internes, actualités).

Architecture RAG de bout en bout

Un système RAG complet comprend deux pipelines distincts qui s'exécutent à des moments différents.

Pipeline 1 : Indexation (offline)

Documents bruts (PDF, HTML, Markdown, CSV…) │ ▼ [1. Chargement] → loaders : PyPDF, WebBaseLoader, UnstructuredLoader… │ ▼ [2. Chunking] → découpage en morceaux (~500-1000 tokens) │ ▼ [3. Embedding] → modèle d'embedding : chaque chunk → vecteur [768 dims] │ ▼ [4. Stockage] → vector store (ChromaDB, Qdrant…) + métadonnées

Pipeline 2 : Retrieval + Generation (runtime)

Question utilisateur │ ▼ [5. Embedding] → même modèle que l'indexation : question → vecteur │ ▼ [6. Recherche] → similarité cosinus dans le vector store → top-k chunks │ ▼ [7. Prompt] → "Réponds à la question en te basant sur ce contexte : {chunks}" │ ▼ [8. Génération] → LLM génère la réponse ancrée dans les documents

Implémentation minimale avec LangChain + ChromaDB

Pythonrag_minimal.py
from langchain_community.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.vectorstores import Chroma
from langchain_community.embeddings import OllamaEmbeddings  # ou OpenAIEmbeddings
from langchain_anthropic import ChatAnthropic
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

# ── INDEXATION ──────────────────────────────────────────────────
loader = PyPDFLoader("documentation.pdf")
docs = loader.load()

splitter = RecursiveCharacterTextSplitter(chunk_size=800, chunk_overlap=150)
chunks = splitter.split_documents(docs)

embeddings = OllamaEmbeddings(model="nomic-embed-text")  # gratuit, local
vectorstore = Chroma.from_documents(chunks, embeddings, persist_directory="./chroma_db")

# ── RETRIEVAL + GENERATION ───────────────────────────────────────
retriever = vectorstore.as_retriever(search_kwargs={"k": 4})

prompt = ChatPromptTemplate.from_messages([
    ("system", """Réponds à la question en te basant UNIQUEMENT sur le contexte.
Si la réponse n'est pas dans le contexte, dis-le clairement.

Contexte :
{context}"""),
    ("human", "{question}")
])

def format_docs(docs):
    return "\n\n---\n\n".join(
        f"[Source: {d.metadata.get('source','?')}, p.{d.metadata.get('page','?')}]\n{d.page_content}"
        for d in docs
    )

chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | ChatAnthropic(model="claude-opus-4-8")
    | StrOutputParser()
)

answer = chain.invoke("Quelle est la procédure de remboursement ?")
print(answer)
⚠ Piège : deux pipelines, un seul modèle d'embedding Le modèle d'embedding utilisé à l'indexation doit être identique à celui utilisé au retrieval. Changer de modèle d'embedding sans réindexer toute la base produit des résultats absurdes — les vecteurs de la question et des documents ne sont plus dans le même espace.

Embeddings et représentation sémantique

Un embedding est la traduction d'un texte (mot, phrase, document) en un vecteur de nombres — une coordonnée dans un espace à haute dimension. Ce qui rend les embeddings magiques : les textes sémantiquement proches se retrouvent géographiquement proches dans cet espace.

Comment un modèle d'embedding produit un vecteur

Texte : "Comment annuler une commande ?" │ ▼ Tokenisation : ["Comment", "annuler", "une", "commande", "?"] │ ▼ Transformer encoder (BERT-like) → représentation contextuelle │ ▼ Pooling (CLS token ou mean) → vecteur unique │ ▼ [0.12, -0.34, 0.89, 0.02, -0.67, ..., 0.44] ← 768 dimensions (ex: all-MiniLM)

Deux phrases sémantiquement équivalentes produisent des vecteurs très proches même si elles n'ont pas de mot en commun :

"Comment annuler une commande ?" → vecteur A "Je voudrais retourner mon achat" → vecteur B cosine_similarity(A, B) ≈ 0.87 ← très proches sémantiquement ✓

Distance et similarité vectorielle

La recherche dans un vector store consiste à trouver les vecteurs les plus "proches" d'une requête. Mais "proche" a plusieurs définitions mathématiques, et le choix impacte les résultats.

Similarité cosinus

La métrique la plus utilisée en RAG. Elle mesure l'angle entre deux vecteurs, ignorant leur magnitude. Idéale pour les embeddings qui représentent du sens.

Pythonmétriques de distance
import numpy as np

def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))
# Résultat : [-1, 1], où 1 = identique, 0 = orthogonal, -1 = opposé

def euclidean_distance(a, b):
    return np.linalg.norm(a - b)
# Résultat : [0, ∞], où 0 = identique

def dot_product(a, b):
    return np.dot(a, b)
# Similaire à cosinus si les vecteurs sont normalisés (||v|| = 1)

# ChromaDB utilise par défaut : L2 (distance euclidienne)
# mais peut être configuré pour cosinus ou dot product :
import chromadb
client = chromadb.Client()
collection = client.create_collection(
    name="ma_collection",
    metadata={"hnsw:space": "cosine"}  # "l2", "cosine", ou "ip" (inner product)
)

# ⚠ Toujours normaliser vos embeddings si vous utilisez dot product ou IP
# La plupart des modèles retournent des vecteurs déjà normalisés
# Vérifiez : np.linalg.norm(vecteur) ≈ 1.0

Algorithmes d'index : ANN vs exact search

Pour des millions de vecteurs, comparer la requête avec chacun (exact search = O(n)) est trop lent. On utilise des algorithmes Approximate Nearest Neighbor (ANN) qui sacrifient un peu de précision pour gagner énormément en vitesse.

AlgorithmeUtilisé parAvantageLimite
HNSWChromaDB, Qdrant, WeaviateTrès rapide, bonne précisionConsomme beaucoup de RAM
IVF (Flat/PQ)FAISSÉconomique en mémoireMoins précis sur petits datasets
DiskANNAzure AI SearchFonctionne sur disquePlus lent que HNSW en RAM
Exact (brute-force)FAISS, petits datasets100% précisO(n), lent sur de grands datasets
ℹ HNSW dans ChromaDB ChromaDB utilise HNSW par défaut via la bibliothèque hnswlib. Les paramètres clés : hnsw:M (connectivité — défaut 16, plus = meilleure précision mais plus de RAM) et hnsw:ef_construction (qualité de l'index — défaut 100). Pour la plupart des usages, les defaults sont bons.

Qu'est-ce qu'un vector store ?

Un vector store (ou base de données vectorielle) est un système de stockage spécialisé qui permet de stocker des vecteurs d'embedding et d'effectuer des recherches par similarité de façon efficace. C'est la brique centrale de tout système RAG.

Ce qu'il stocke et ce qu'il fait

┌────────────────────────────────────────────────────────────┐ │ VECTOR STORE │ │ │ │ id vecteur [768 dims] métadonnées │ │ ─────── ────────────────── ───────────────── │ │ doc-001 [0.12, -0.34, …, 0.89] {source: "faq.pdf"} │ │ doc-002 [0.55, 0.21, …, -0.12] {page: 3, date: …} │ │ doc-003 [-0.08, 0.67, …, 0.44] {category: "RH"} │ │ … │ │ │ │ Opérations : │ │ • add(vecteurs, textes, métadonnées) │ │ • query(vecteur_requête, k=5) → top-5 plus proches │ │ • query_with_filter(vecteur, where={"category": "RH"}) │ │ • update, delete, get │ └────────────────────────────────────────────────────────────┘

Vector store vs base de données classique

CritèreBase relationnelle (SQL)Vector store
Requête typiqueWHERE name = 'Alice'Les 5 docs les plus proches de cette question
Type de rechercheExacte (égalité, comparaison)Approximative (similarité sémantique)
IndexB-tree, hashHNSW, IVF
ScalabilitéMilliards de lignes (mature)Milliards de vecteurs (en progrès)
TransactionnelOui (ACID)Rarement
ℹ La plupart des vector stores stockent aussi le texte ChromaDB, Qdrant, Weaviate stockent le document original (texte) en plus du vecteur. FAISS ne stocke que les vecteurs — vous devez gérer le texte vous-même en parallèle.

Maîtriser l'API ChromaDB dans ses détails évite les surprises en production. Voici le guide complet des opérations CRUD.

Ajouter des documents

Pythonadd() — tous les cas
import chromadb
from chromadb.utils import embedding_functions

client = chromadb.PersistentClient(path="./db")

# Fonction d'embedding : Chroma calcule les vecteurs pour vous
ef = embedding_functions.SentenceTransformerEmbeddingFunction(
    model_name="intfloat/multilingual-e5-small"
)
collection = client.get_or_create_collection("docs", embedding_function=ef)

# Ajout simple
collection.add(
    documents=[
        "La politique de remboursement est de 30 jours.",
        "Les livraisons sont effectuées sous 48h ouvrées.",
        "Notre support est disponible 24h/24 par email.",
    ],
    metadatas=[
        {"category": "remboursement", "source": "faq.pdf", "page": 12},
        {"category": "livraison",     "source": "faq.pdf", "page": 8},
        {"category": "support",       "source": "faq.pdf", "page": 3},
    ],
    ids=["doc-001", "doc-002", "doc-003"]   # IDs uniques obligatoires
)

# Ajout avec embeddings pré-calculés (si vous gérez les embeddings vous-même)
import numpy as np
custom_embeddings = [np.random.rand(384).tolist() for _ in range(3)]
collection.add(
    embeddings=custom_embeddings,
    documents=["Texte 1", "Texte 2", "Texte 3"],
    ids=["e-001", "e-002", "e-003"]
)

# ⚠ Les IDs doivent être uniques dans la collection
# Un add() avec un ID existant lève une erreur → utilisez upsert()
collection.upsert(
    documents=["Version mise à jour du doc 1"],
    metadatas=[{"category": "remboursement", "source": "faq_v2.pdf", "page": 15}],
    ids=["doc-001"]
)

Chunk size, overlap et qualité

Le chunk size est le paramètre qui impacte le plus la qualité du RAG. Ni trop petit, ni trop grand — mais le bon équilibre dépend de votre use case.

Impact du chunk size

Chunk TROP PETIT (< 200 tokens) ✓ Précision de la récupération (moins de bruit) ✗ Perd le contexte → le LLM n'a pas assez de contexte pour répondre ✗ Plus de chunks = index plus grand = plus lent Chunk TROP GRAND (> 2000 tokens) ✓ Contexte riche pour le LLM ✗ Le vecteur "dilue" le sens → mauvaise précision de récupération ✗ Dépasse le budget de contexte si k=5 → 5 × 2000 = 10k tokens injectés SWEET SPOT : 400-1200 tokens selon le cas d'usage • FAQ courtes, code → 300-500 tokens • Documentation tech → 600-900 tokens • Documents légaux → 800-1200 tokens

Métadonnées et enrichissement

Les métadonnées transforment un vecteur muet en document intelligent. Elles permettent le filtrage, la traçabilité des sources, et l'enrichissement du contexte donné au LLM.

Métadonnées essentielles à toujours capturer

Pythonenrichissement des métadonnées
from langchain_community.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from datetime import datetime
import hashlib, os

def load_and_enrich(filepath: str, splitter) -> list:
    """Charge un document et enrichit chaque chunk avec des métadonnées."""
    loader = PyPDFLoader(filepath)
    pages = loader.load()

    chunks = splitter.split_documents(pages)

    file_stat = os.stat(filepath)

    for i, chunk in enumerate(chunks):
        # Métadonnées de base
        chunk.metadata.update({
            "source_file": os.path.basename(filepath),
            "source_path": os.path.abspath(filepath),
            "chunk_index": i,
            "total_chunks": len(chunks),
            "char_count": len(chunk.page_content),
            # Date d'indexation (pour filtrer les docs récents)
            "indexed_at": datetime.now().isoformat(),
            # Date de modification du fichier
            "file_modified": datetime.fromtimestamp(file_stat.st_mtime).isoformat(),
            # Hash pour la déduplication
            "content_hash": hashlib.sha256(chunk.page_content.encode()).hexdigest()[:16],
        })

        # Métadonnées dérivées du contenu (ex: détection de langue)
        if any(word in chunk.page_content.lower() for word in ["legal", "contrat", "article"]):
            chunk.metadata["content_type"] = "legal"
        elif "def " in chunk.page_content or "class " in chunk.page_content:
            chunk.metadata["content_type"] = "code"
        else:
            chunk.metadata["content_type"] = "general"

    return chunks

# Utilisation dans le contexte du LLM
def format_with_sources(docs) -> str:
    parts = []
    for doc in docs:
        m = doc.metadata
        header = f"[{m['source_file']}, p.{m.get('page', '?')}, chunk {m['chunk_index']}]"
        parts.append(f"{header}\n{doc.page_content}")
    return "\n\n---\n\n".join(parts)

Hybrid Search : dense + BM25

La recherche vectorielle seule échoue sur les requêtes qui contiennent des termes très spécifiques (noms propres, codes produits, termes techniques rares). La recherche par mots-clés (BM25) excelle là où les embeddings peinent. En 2026, le hybrid search est la pratique standard en production.

Pourquoi combiner les deux

Dense (vecteurs) Sparse (BM25) ───────────────── ──────────────────── Sémantique ✓ Mots-clés exacts ✓ Synonymes ✓ Noms propres ✓ Langue naturelle ✓ Codes (REF-4521) ✓ Bruit ambiant ✗ Contexte sémantique ✗ Termes rares ✗ Paraphrase ✗ → Combiner les deux via Reciprocal Rank Fusion (RRF)

Reciprocal Rank Fusion (RRF)

PythonRRF — comment ça marche
def reciprocal_rank_fusion(results_lists: list[list], k: int = 60) -> list:
    """
    results_lists : liste de listes, chaque liste = résultats d'un retriever
    k : constante de fusion (60 est le standard)
    """
    scores = {}
    for results in results_lists:
        for rank, doc in enumerate(results):
            doc_id = doc.page_content  # ou un hash
            if doc_id not in scores:
                scores[doc_id] = {"doc": doc, "score": 0}
            # Score RRF : 1 / (k + rang)
            # Rang 1 → 1/61 ≈ 0.016
            # Rang 5 → 1/65 ≈ 0.015
            # Rang 10 → 1/70 ≈ 0.014
            scores[doc_id]["score"] += 1 / (k + rank + 1)

    # Trier par score décroissant
    return sorted(scores.values(), key=lambda x: x["score"], reverse=True)

# Pourquoi k=60 ? Il atténue l'impact des positions élevées (rank 1 vs rank 2)
# et favorise les documents qui apparaissent dans les DEUX listes
⚠ Piège : BM25Retriever et la RAM BM25Retriever.from_documents() charge tous vos documents en RAM pour construire l'index BM25. Sur 1M de documents, c'est plusieurs GB. Pour la production, utilisez rank_bm25 avec un stockage sur disque ou Elasticsearch (qui a un BM25 natif et scalable).

Reranking avec cross-encoders

MMR et diversité des résultats

Sans contrôle, une recherche vectorielle peut retourner 5 chunks qui disent tous la même chose — différentes parties du même paragraphe. Le MMR (Maximal Marginal Relevance) résout ce problème en favorisant la diversité.

L'algorithme MMR

MMR sélectionne les documents un par un : Pour chaque document candidat non encore sélectionné, calcule : MMR_score = λ × similarity(doc, query) - (1-λ) × max_similarity(doc, selected_docs) ↑ pertinence par rapport à la requête ↑ différence par rapport à ce qu'on a déjà λ = 1 → tri pur par pertinence (= recherche classique) λ = 0 → maximise la diversité (ignore la pertinence) λ = 0.5 → équilibre pertinence et diversité (défaut)
PythonMMR dans ChromaDB et LangChain

)
# ChromaDB n'a pas MMR natif → utiliser LangChain

# Via LangChain (recommandé)

    search_type="mmr",                  # activer MMR
    search_kwargs={
        "k": 5,                         # retourner 5 docs
        "fetch_k": 20,                  # évaluer 20 candidats
        "lambda_mult": 0.7,             # 0.7 = penche vers la pertinence
    }
)

⚠ Piège : MMR = plus lent MMR évalue fetch_k candidats (ici 20) contre tous les documents déjà sélectionnés. La complexité est O(k × fetch_k). Sur de grandes collections avec un grand fetch_k, la latence augmente. Gardez fetch_k <= 50 en production pour rester sous 200ms.

Mémoire long-terme vectorielle

La mémoire long-terme persiste au-delà d'une session dans un vector store dédié aux souvenirs utilisateur — distinct de la base RAG de connaissances.

Architecture

SESSION 1 (lundi) User: "Je travaille sur un projet ML en Python" Agent memorise: vectorise et stocke dans user_memory_store SESSION 2 (vendredi) — nouveau thread, contexte vide User: "J'ai un problème avec mon code" Agent: [recherche user_memory] -> "projet ML Python" -> repond en tenant compte du contexte
Pythonmémoire long-terme ChromaDB
import chromadb
from langchain_community.embeddings import HuggingFaceEmbeddings
from datetime import datetime
import json

embed = HuggingFaceEmbeddings(model_name="all-MiniLM-L6-v2")
chroma = chromadb.PersistentClient(path="./user_memory_db")

def get_col(user_id):
    return chroma.get_or_create_collection(f"memory_{user_id}", metadata={"hnsw:space": "cosine"})

def save_memory(user_id, text, mem_type="general"):
    col = get_col(user_id)
    col.add(
        embeddings=[embed.embed_query(text)],
        documents=[text],
        metadatas=[{"type": mem_type, "created_at": datetime.now().isoformat()}],
        ids=[f"{user_id}_{datetime.now().timestamp()}"]
    )

def recall(user_id, query, k=3):
    col = get_col(user_id)
    if col.count() == 0:
        return []
    res = col.query(query_embeddings=[embed.embed_query(query)],
                    n_results=min(k, col.count()), include=["documents"])
    return res["documents"][0]

async def agent_with_ltm(user_id, user_message, history, model):
    memories = recall(user_id, user_message)
    mem_ctx = "\n".join(f"- {m}" for m in memories) or "Aucun souvenir"

    response = await model.ainvoke([
        {"role": "system", "content": f"Souvenirs:\n{mem_ctx}"},
        *history,
        {"role": "user", "content": user_message}
    ])

    # Extraire et sauvegarder les faits importants
    extraction = await model.ainvoke([{"role": "user", "content":
        f'Extrais les faits importants a memoriser. JSON list: ["fait1",...] ou []\nUser:{user_message}\nAssistant:{response.content}'}])
    try:
        for fact in json.loads(extraction.content):
            save_memory(user_id, fact)
    except Exception:
        pass
    return response.content
⚠ Piège : mémoriser trop de choses Si vous mémorisez chaque phrase, la collection grossit vite et le retrieval se noie dans le bruit. Filtrez agressivement : mémorisez uniquement les faits persistants (préférences, contraintes, historique projet) — jamais les salutations ou confirmations éphémères.

LangGraph Store et mémoire cross-thread

LangGraph 1.0 introduit le Store — une mémoire partagée accessible depuis tous les threads d'un même agent. C'est la solution intégrée pour la mémoire long-terme sans gérer ChromaDB manuellement.

PythonLangGraph Store complet
from langgraph.store.memory import InMemoryStore
from langgraph.store.base import BaseStore
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from typing import TypedDict, Annotated
from langgraph.graph.message import add_messages
from datetime import datetime

# Dev : InMemoryStore | Prod : PostgresStore
store = InMemoryStore(index={"embed": embed.embed_query, "dims": 384})

class State(TypedDict):
    messages: Annotated[list, add_messages]
    user_id: str

def agent_node(state: State, store: BaseStore) -> dict:
    user_id = state["user_id"]
    last_msg = state["messages"][-1].content

    # Recherche sémantique dans les souvenirs
    memories = store.search(("memories", user_id), query=last_msg, limit=3)
    context = "\n".join(m.value.get("text", "") for m in memories)

    response = model.invoke([
        {"role": "system", "content": f"Contexte memorise:\n{context}"},
        *state["messages"]
    ])

    # Sauvegarder si pertinent
    store.put(
        ("memories", user_id),
        key=f"mem_{datetime.now().timestamp()}",
        value={"text": last_msg, "type": "episodic"}
    )
    return {"messages": [response]}

builder = StateGraph(State)
builder.add_node("agent", agent_node)
builder.add_edge(START, "agent")
builder.add_edge("agent", END)

agent = builder.compile(checkpointer=MemorySaver(), store=store)

# Thread 1 (lundi)
agent.invoke(
    {"messages": [("user", "Je prefere les reponses courtes")], "user_id": "alice"},
    config={"configurable": {"thread_id": "alice-lundi"}}
)
# Thread 2 (vendredi) - nouveau thread, meme user_id, agent se souvient
agent.invoke(
    {"messages": [("user", "Explique les transformers")], "user_id": "alice"},
    config={"configurable": {"thread_id": "alice-vendredi"}}
)

Du RAG linéaire à l'agent de retrieval

Le RAG classique est un pipeline rigide : chercher → injecter → générer. L'Agentic RAG remplace ce pipeline par un agent qui décide dynamiquement quand chercher, quoi chercher, si les résultats sont suffisants, et quand s'arrêter.

Limites du RAG linéaire

RAG LINEAIRE : Question -> [1 recherche fixe] -> [contexte] -> [LLM genere] Echecs : • Question multi-aspect -> 1 seule recherche insuffisante • Question vague -> mauvaise requete -> mauvais resultats • Resultats insuffisants -> hallucination quand meme • Plusieurs docs a croiser -> non supporte

from langchain_core.tools import tool
from langgraph.prebuilt import create_react_agent

@tool
def search_knowledge_base(query: str, category: str = None) -> str:
    """Recherche dans la base de connaissances interne.
    Args:
        query: La question ou sujet a rechercher
        category: Filtre optionnel par categorie (legal, technique, rh)
    """
    kwargs = {"k": 5}
    if category:
        kwargs["filter"] = {"category": category}
    docs = vectorstore.similarity_search(query, **kwargs)
    if not docs:
        return "Aucun document trouve pour cette requete."
    return "\n\n---\n\n".join(
        f"[{d.metadata.get('source','?')}]\n{d.page_content}" for d in docs
    )

@tool
def search_recent_documents(query: str, days_ago: int = 7) -> str:
    """Recherche uniquement dans les documents recents.
    Args:
        query: Le sujet a rechercher
        days_ago: Periode en jours (defaut: 7)
    """
    from datetime import datetime, timedelta
    cutoff = (datetime.now() - timedelta(days=days_ago)).isoformat()
    docs = vectorstore.similarity_search(query, k=5,
               filter={"indexed_at": {"$gte": cutoff}})
    return "\n\n".join(d.page_content for d in docs) if docs else "Aucun document recent."

@tool
def evaluate_sufficiency(question: str, draft_answer: str) -> str:
    """Evalue si la reponse est suffisante. Retourne SUFFICIENT ou INSUFFICIENT."""
    if len(draft_answer) < 50:
        return "INSUFFICIENT: reponse trop courte, cherche plus d'informations."
    return "SUFFICIENT: la reponse couvre bien la question."

agentic_rag = create_react_agent(
    model=ChatAnthropic(model="claude-opus-4-8"),
    tools=[search_knowledge_base, search_recent_documents, evaluate_sufficiency],
    state_modifier="""Tu es un assistant de recherche rigoureux.
Pour chaque question : cherche les infos pertinentes, si multi-aspects fais
plusieurs recherches, evalue ta reponse avant de la donner, cite tes sources."""
)

Self-RAG et Corrective RAG

Ces deux patterns sont devenus des références pour améliorer la fiabilité du RAG en production.

Self-RAG : l'agent grade ses propres sources

Self-RAG (ICLR 2024) : Question -> [Retrieve top-k] -> [Relevance grade] -- NON PERTINENT -> ignorer ce chunk -> [Generate avec chunks pertinents] -> [Faithfulness check] -- HALLUCINATION -> retry -> [Utility check] -- PAS UTILE -> retry -> Reponse finale validee
Pythongrade de documents avec LLM
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

grader_prompt = ChatPromptTemplate.from_messages([
    ("system", "Reponds OUI ou NON uniquement."),
    ("human", "Question: {question}\nDocument: {doc}\nCe document est-il pertinent ?")
])
grader = grader_prompt | model | StrOutputParser()

def filter_relevant_docs(question: str, docs: list) -> list:
    relevant = []
    for doc in docs:
        score = grader.invoke({"question": question, "doc": doc.page_content[:500]})
        if "OUI" in score.upper():
            relevant.append(doc)
    return relevant

# Corrective RAG (CRAG) : si qualite insuffisante, fallback web
def crag_retrieve(question: str, retriever, web_search_tool) -> list:
    docs = retriever.invoke(question)
    relevant = filter_relevant_docs(question, docs)
    ratio = len(relevant) / max(len(docs), 1)

    if ratio >= 0.7:
        return relevant                           # docs locaux suffisants
    elif ratio >= 0.3:
        web_docs = web_search_tool(question)      # completer avec web
        return relevant + web_docs
    else:
        return web_search_tool(question)          # ignorer docs locaux

RAGAS et métriques qualité

RAGAS (RAG Assessment) est le framework standard pour mesurer automatiquement la qualité d'un système RAG selon 4 dimensions complémentaires.

Les 4 métriques RAGAS

FAITHFULNESS La reponse est-elle ancree dans les docs ? -> detecte les hallucinations ANSWER RELEVANCY La reponse repond-elle a la question ? -> evite les reponses vraies mais hors sujet CONTEXT PRECISION Les chunks recuperes sont-ils pertinents (peu de bruit) ? -> evalue la qualite du retrieval CONTEXT RECALL Les chunks couvrent-ils toute l'info necessaire ? -> necessite une reference de reponse correcte

def debug_rag(question: str, retriever, chain, vectorstore=None):
    print(f"\n{'='*50}\nQUESTION: {question}\n")

    # 1. Inspecter le retrieval
    docs = retriever.invoke(question)
    print(f"[RETRIEVAL] {len(docs)} documents:")
    for i, doc in enumerate(docs):
        print(f"  {i+1}. [{doc.metadata.get('source','?')}] {doc.page_content[:100]}...")

    if not docs:
        print("PROBLEME: Aucun document recupere !")
        return

    # 2. Inspecter les distances
    if vectorstore:
        raw = vectorstore.similarity_search_with_score(question, k=5)
        print("\n[DISTANCES]:")
        for doc, score in raw:
            flag = "OK" if score < 0.3 else "FAIBLE"
            print(f"  {flag} score={score:.4f} | {doc.page_content[:60]}")

    # 3. Budget contexte
    total_chars = sum(len(d.page_content) for d in docs)
    print(f"\n[CONTEXTE] {total_chars} chars (~{total_chars//4} tokens)")
    if total_chars > 15000:
        print("WARNING: Contexte tres long -> envisagez reranking")

    # 4. Generer et verifier fidelite
    response = chain.invoke(question)
    print(f"\n[REPONSE]: {response}")

    doc_words = set(" ".join(d.page_content for d in docs).lower().split())
    ans_words = set(response.lower().split())
    overlap = len(ans_words & doc_words) / max(len(ans_words), 1)
    if overlap < 0.3:
        print(f"WARNING: Faible chevauchement lexical ({overlap:.0%}) -> possible hallucination")

    return {"docs": docs, "response": response, "overlap": overlap}

import re

INJECTION_PATTERNS = [
    r"ignore\s+previous\s+instructions",
    r"forget\s+your\s+system\s+prompt",
    r"new\s+system:",
    r"<\|im_start\|>system",
]

def validate_before_indexing(content: str) -> bool:
    for pattern in INJECTION_PATTERNS:
        if re.search(pattern, content, re.IGNORECASE):
            print(f"Document rejete : injection detectee")
            return False
    return True

✅ Checklist avant de déployer
  • Même modèle d'embedding pour indexation ET retrieval
  • Hybrid search si votre domaine a des termes techniques/codes
  • Reranking pour améliorer la précision (fetch 20, rerank → top 5)
  • Troncature/résumé du contexte si plus de 5k tokens
  • Cache des embeddings de requête et des appels LLM identiques
  • Dataset RAGAS de 50+ questions, CI/CD sur chaque déploiement
  • Contrôle d'accès par rôle sur les documents indexés
  • Validation anti-injection à chaque document indexé
  • Backup régulier du vector store (pas de cp à chaud)
